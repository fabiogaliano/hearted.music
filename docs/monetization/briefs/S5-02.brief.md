export const getOnboardingData = createServerFn({ method: "GET" })

export function WelcomeStep() {
	const theme = useTheme();
	const { goToStep } = useOnboardingNavigation();

## Constraints (From DECISIONS.md & TERMINOLOGY.md)

### Monetization Isolation (§1, Product Invariants)

From **DECISIONS.md §1**:
> Demo song is completely outside monetization — no unlock row, no credit use, no replacement-credit semantics.

**Implementation consequence**: 
- No call to billing domain unlock functions
- No `account_song_unlock` row creation
- No credit balance changes
- No replacement credit grants on failure

### Entitlement Predicate (§2, Architecture Invariants)

From **DECISIONS.md §2**:
> A song is entitled when: unlock row exists with revoked_at IS NULL **OR** account has active unlimited access.

**Implementation consequence**:
- Demo song does **not** need an unlock row to be displayed
- Display is purely informational, outside the entitlement system
- No billing state lookups needed

### Song Display State (From TERMINOLOGY.md §2)

From **TERMINOLOGY.md**:
> `SongDisplayState` values: `locked | pending | analyzing | analyzed | failed`

Demo song analysis display does **not** interact with `SongDisplayState`. It's:
- Pre-analyzed (analysis exists in DB)
- Outside the user's entitlement flow
- Not subject to locked/pending states

## Open Questions / Risks

### 1. **Demo Song Identification**

The story says "Reads demo song analysis from the database (pre-seeded)" but does **not** specify:
- How is the demo song identified? (Environment variable? Hardcoded UUID? Lookup by name/artist?)
- Does a demo song exist in the database already?
- If the demo song is missing at runtime, should the component skip to the next step, show an error, or fall back to static content?

**Risk**: Without knowing how to identify the demo song, the implementation cannot fetch it. The demo song "seeding" is explicitly marked "out of scope" and "setup task," meaning it's expected to exist but the code to create it does not exist in this repo.

**Assumption**: The implementation agent will either:
- Ask where/how the demo song ID is configured
- Implement a fallback that gracefully handles missing demo data (skip to next step or show placeholder)
- Define a `DEMO_SONG_ID` environment variable that must be set in dev/test environments

### 2. **Graceful Fallback Strategy**

The story requires "Graceful handling if demo song is not found" but does not specify the UX:
- Skip to `match-showcase` automatically?
- Show an error message and allow user to continue?
- Display static/canned demo content?
- Show a loading error with retry?

**Risk**: Implementation agent must decide the user-facing behavior and define when the graceful fallback is triggered (database error, song not found, analysis not found, network error, etc.).

### 3. **Server Function vs Loader**

The story says "May need a loader/function to fetch demo analysis" but does not specify:
- Should demo data be loaded in the route loader (like `getOnboardingData()`) or fetched client-side?
- If route loader: does it block navigation if demo song is missing?
- If client-side: should it be a separate server function or embedded in the component?

**Assumption**: Following the existing pattern, demo data should be fetched via a server function (not route loader) to avoid blocking navigation. Error handling would be in the component (catch + graceful fallback).

### 4. **Analysis Display Customization**

The story says "Displays pre-seeded demo song analysis (song name, analysis content, visual highlights)" but does not specify:
- Which analysis fields are displayed? (headline, interpretation, themes, journey, key_lines, sonic_texture, mood, etc.?)
- Is the layout identical to liked-songs detail view or customized for onboarding context?
- Should it include audio features, genres, or other metadata?
- What are "visual highlights"? (color coding, icons, animations?)

**Assumption**: Implementation agent should:
- Reuse `PanelContent.tsx` display logic for consistency
- Include headline + interpretation toggle, themes, journey, key_lines (core analysis fields)
- Consider onboarding context: fast-moving experience, may benefit from simplified layout vs full detail view
- Use existing `useTheme()` for color/visual consistency

### 5. **Navigation Trigger**

The story says "Navigates to `match-showcase` on user action" but does not specify:
- What user action? (Button click? Auto-advance after delay? Scroll to bottom?)
- Button label? ("Continue," "Next," "See Your Matches," etc.?)
- Should it clear `phaseJobIds` like other steps do?

**Assumption**: 
- Button labeled "Next" or "See Your Matches" (product language TBD)
- onClick calls `goToStep("match-showcase")`
- Follows pattern from `WelcomeStep` and `ReadyStep`

### 6. **Component File Location**

The story specifies: `src/features/onboarding/components/SongShowcaseStep.tsx` (new)

**Status**: File does not exist. Implementation agent must create it with standard React component export.
