```typescript
export function makeInitialProgress(
  batchSize: number,
  batchSequence: number,
  totalSongs: number,
): InitializedEnrichmentChunkProgress {
  const stages = createPendingEnrichmentStages();

  return {
    total: totalSongs * ALL_STAGE_NAMES.length,  // ← Fixed: always songs × 4
    done: 0,
    succeeded: 0,
    failed: 0,
    currentStage: undefined,
    stages,
    batchSize,
    batchSequence,
  };
}
```

**Problem:** For a Phase-A-only song (needs only audio features + genre tagging), this calculates 4 units of work per song. Actual work is 2 stages. For a 10-song batch with 5 Phase-A and 5 Phase-B/C, total would report as 40 but actual planned work is 5×2 + 5×4 = 30.

### `src/lib/domains/library/liked-songs/status-queries.ts`

**Function signature (unchanged):**

```typescript
export function markPipelineProcessed(
  accountId: string,
  itemType: ItemType,
  itemIds: string[],
): Promise<Result<ItemStatus[], DbError>> {
  if (itemIds.length === 0) {
    return Promise.resolve(Result.ok<ItemStatus[], DbError>([]));
  }

  const supabase = createAdminSupabaseClient();
  return fromSupabaseMany(
    supabase
      .from("item_status")
      .upsert(
        itemIds.map((itemId) => ({
          account_id: accountId,
          item_id: itemId,
          item_type: itemType,
          is_new: false,
        })),
        {
          onConflict: "account_id,item_id,item_type",
          ignoreDuplicates: true,
        },
      )
      .select(),
  );
}
```

The function exists and is used only in:
- `orchestrator.ts` production code (line 315)
- `orchestrator.test.ts` mock (line 62)

No other production code paths call it.

## Open Questions / Risks

### 1. Content Activation as a tracked stage in progress

The story says "Include content activation as a stage in the count." However:
- `ENRICHMENT_STAGE_NAMES` in `src/lib/platform/jobs/progress/enrichment.ts` is a frozen type (used by the jobs platform)
- Adding `content_activation` to this list would require updating the shared type and its schema
- Content activation is called once per batch (account-scoped), not once per song like the 4 enrichment stages
- The progress tracking in `orchestrator.ts` does not currently update progress for content activation — it's called but not tracked

The implementation must decide:
- Should `content_activation` be added to `ENRICHMENT_STAGE_NAMES`? (This would affect the shared progress schema.)
- Should content activation progress be tracked separately or included in the stage map?
- Should a single `content_activation` stage be counted as 1 unit regardless of how many songs need it, or proportionally?

### 2. Progress calculation with variable stage count per song

Deriving `total` from work plan flags is straightforward: sum all `true` flags across all `SongStageFlags` plus the content activation count. However:
- Progress tracking currently assumes each stage is independent and trackable via `applyStageResult`
- Content activation is different: it's a single operation for multiple songs, not one operation per song
- The progress object needs to show `total` work units and accumulated `done`/`succeeded`/`failed`
- Mixing per-song stages (audio features, genre tagging, etc.) with a single account-scoped stage (content activation) in the same progress counter may be confusing or misleading

Example calculation:
- 5 Phase-A songs: 5 audio_features + 5 genre_tagging = 10 units
- 5 Phase-B/C songs: 5 analysis + 5 embedding = 10 units
- Content activation: 5 songs need it = 1 unit (or 5 units?)
- Total: 20 or 25?

### 3. Tests may reference hardcoded stage count

The test file mocks all stage runners but does not assert on progress totals. If tests were to verify `progress.total` values, they would need updating to match the new per-work-plan calculation.

### 4. `markPipelineProcessed` still defined in status-queries

The function will remain in the module after removal from orchestrator. The story constraint says "may still be used by devtools — if so, scope appropriately." The implementation should verify whether devtools or tests use it before deciding whether to keep the export.

## Implementation Notes

Based on code inspection:

1. **Removal is clean:** `markPipelineProcessed` is only imported and called once in orchestrator.ts (line 315). No other production code paths call it.

2. **Content activation already writes `item_status`:** S3-03 is complete. The `runContentActivation` function handles all three paths (unlimited subscription, self-hosted, free/pack) and writes `item_status` appropriately.

3. **Work plan is available:** `selectEnrichmentWorkPlan` already populates `needContentActivation` and per-song flags. The orchestrator receives the full work plan before calling `enrichSongs`.

4. **Progress initialization can be updated:** `makeInitialProgress` is called immediately after `selectEnrichmentWorkPlan` and receives `batchSize`, `batchSequence`, and `totalSongs`. It can be extended to also receive the work plan (or just the flags array) to derive accurate totals.

5. **Progress schema scope:** The `EnrichmentChunkProgress` type is defined in the shared jobs platform module. If content activation needs to be a tracked stage, the shared type must be updated. Otherwise, progress can simply report an aggregate `total` without breaking it into individual stage progress units.
