# Implementation Brief: S2-07 Devtools Reset/Reseed for Billing

## Goal

Update `warmReplayReset()`, `reseedAfterReset()`, and `reset-onboarding.ts` to clear and re-seed billing tables so dev workflows remain consistent after billing tables are added.

### `src/lib/workflows/library-processing/devtools/reset.ts`

`warmReplayReset(accountId)` clears `item_status`, `match_snapshot`, `match_result`, and resets `library_processing_state`. It does **not** touch any billing tables.

```typescript
export async function warmReplayReset(accountId: string): Promise<WarmResetResult> {
  const supabase = createAdminSupabaseClient();
  const cancelledJobs = await cancelActiveLibraryProcessingJobs(supabase, accountId);
  const { data: deletedStatuses } = await supabase
    .from("item_status").delete().eq("account_id", accountId).select("item_id");
  const clearedMatchSnapshots = await clearMatchSnapshots(supabase, accountId);
  await resetLibraryProcessingState(accountId);
  return { cancelledJobs, clearedItemStatuses: deletedStatuses?.length ?? 0, clearedMatchSnapshots };
}
```

## Constraints

From DECISIONS.md §3 (Data Model Invariants):
- **"Every account gets an `account_billing` row in every deployment; missing row is an invariant violation"** — reset must UPDATE (not DELETE) the `account_billing` row.
- **"`self_hosted` is orthogonal to `plan`; those accounts may keep `plan='free'` and `credit_balance=0`"** — reset to `plan='free', credit_balance=0` is correct in all cases.
- **"SQL never infers deployment mode from missing rows"** — `account_billing` row must always exist after reset.

From DECISIONS.md §5 (Workflow Invariants):
- **"Every billing mutation that can change the resolved queue band invokes `reprioritize_pending_jobs_for_account`"** — after resetting billing in self-hosted mode (which sets `unlimited_access_source='self_hosted'`, `queueBand='priority'`), call the reprioritization RPC.
- **"App-layer `self_hosted` provisioning calls the reprioritization RPC immediately after the billing write"** — same applies to the reset that restores self-hosted state.

From DECISIONS.md §3:
- **`billing_webhook_event` and `billing_bridge_event`** are keyed on `stripe_event_id` (no `account_id`) — do NOT clear these tables in account-scoped resets.

Tables to clear per account:
- `account_song_unlock` (account_id FK)
- `credit_transaction` (account_id FK)
- `pack_credit_lot` (account_id FK)
- `subscription_credit_conversion` (account_id FK)
- `subscription_credit_conversion_allocation` (FK chain via `subscription_credit_conversion_id` — check if directly has account_id or needs join)
- `billing_activation` (account_id FK per S1-03 schema)

Tables to preserve:
- `account_billing` — reset to defaults, never delete
- `billing_webhook_event` — no account_id, global idempotency table
- `billing_bridge_event` — no account_id, global idempotency table

## Risks

1. **`subscription_credit_conversion_allocation` may not have a direct `account_id` column.** Per S1-02 schema (not provided in full), it may be joined to `subscription_credit_conversion` via `subscription_credit_conversion_id`. If so, the delete must be done via a subquery: `DELETE FROM subscription_credit_conversion_allocation WHERE subscription_credit_conversion_id IN (SELECT id FROM subscription_credit_conversion WHERE account_id = $1)`. Verify the schema before implementing direct `account_id` delete.

2. **`reprioritize_pending_jobs_for_account` after reset in self-hosted mode**: The reset happens before any jobs exist (new dev account), so the RPC will return 0. Safe to ignore the return value, but should not fail if the RPC errors. Match the pattern from S2-03: `await supabase.rpc(...)` and ignore the result.

3. **`deleteCount` helper in `reset-onboarding.ts` uses a typed union for `table`.** Adding new billing tables requires extending that union type. The implementation must add each billing table name to the union (or use a different deletion pattern for billing tables to avoid modifying the shared helper).

4. **Reseed `seedBillingState` upsert conflict behavior**: If `account_billing` already exists with the correct `unlimited_access_source`, the upsert will overwrite all fields. This is intentional — reseed should always produce a clean known state.
