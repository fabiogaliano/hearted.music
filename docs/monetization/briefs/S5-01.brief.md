### `src/routes/_authenticated/onboarding.tsx`

**Route loader (lines 22–78):**
```typescript
const onboardingSearchSchema = z.object({
	step: fallback(ONBOARDING_STEPS, "welcome").default("welcome"),
});
export type OnboardingSearch = z.infer<typeof onboardingSearchSchema>;

export const Route = createFileRoute("/_authenticated/onboarding")({
	validateSearch: zodValidator(onboardingSearchSchema),
	beforeLoad: async ({ search, context }) => {
		const { session } = context;
		const accountId = session.accountId;

		// Load onboarding data
		const data = await getOnboardingData();

		// Completed onboarding? Go to dashboard
		if (data.isComplete) {
			throw redirect({ to: "/dashboard" });
		}

		// Step progression validation
		const savedStep = data.currentStep;
		const stepOrder = ONBOARDING_STEPS.options;
		const urlStepIndex = stepOrder.indexOf(search.step);
		const savedStepIndex = stepOrder.indexOf(savedStep);

		// Auto-skip flag-playlists → ready when no playlists
		const isAutoSkipToReady =
			search.step === "ready" &&
			savedStep === "flag-playlists" &&
			data.playlists.length === 0;

		// Guard: URL step ahead of saved step?
		if (urlStepIndex > savedStepIndex && !isAutoSkipToReady) {
			throw redirect({
				to: "/onboarding",
				search: { step: savedStep },
			});
		}

		// Resume: If URL is 'welcome' but user progressed, resume from saved step
		if (search.step === "welcome" && savedStep !== "welcome") {
			throw redirect({
				to: "/onboarding",
				search: { step: savedStep },
			});
		}

		// Skip flag-playlists if no playlists
		if (search.step === "flag-playlists" && data.playlists.length === 0) {
			throw redirect({
				to: "/onboarding",
				search: { step: "ready" },
			});
		}

		return {
			accountId,
			onboardingData: data,
		};
	},
	component: OnboardingPage,
});
```

**Current behavior:**
- Validates `step` URL search param against ONBOARDING_STEPS enum
- Uses `stepOrder.indexOf()` for sequential progression checking
- Auto-skips flag-playlists → ready if no playlists exist
- No BILLING_ENABLED checks yet — **S5-01 must add plan-selection auto-skip logic here**

```typescript
import {
	// ...
	ONBOARDING_STEPS,
	type OnboardingStep,
	updateOnboardingStep,
} from "@/lib/domains/library/accounts/preferences-queries";

// ...

const stepInputSchema = z.object({
	step: ONBOARDING_STEPS,
});

// ...

/**
 * Saves the current onboarding step for resumability.
 * Clears phaseJobIds when transitioning past syncing step.
 * Updates the DB every time the user navigates to a new step.
 */
export const saveOnboardingStep = createServerFn({ method: "POST" })
	.middleware([authMiddleware])
	.inputValidator(stepInputSchema)
	.handler(async ({ data, context }): Promise<{ success: true }> => {
		const { session } = context;

		const result = await updateOnboardingStep(session.accountId, data.step);

		if (Result.isError(result)) {
			throw new OnboardingError("save_onboarding_step", result.error);
		}

		// Clear phase job IDs when transitioning past syncing step
		if (data.step === "flag-playlists" || data.step === "ready") {
			const clearResult = await clearPhaseJobIds(session.accountId);
			if (Result.isError(clearResult)) {
				console.warn("Failed to clear phase job IDs:", clearResult.error);
			}
		}

		return { success: true };
	});
```

## Open Questions / Risks

### 1. **Placeholder component implementation**

The story says "placeholder components initially" but does not specify what that means:
- Return `null`? (like the `complete` step)
- Return a simple div with the step name? (for manual testing)
- Return a full stub component matching the interface of working steps?

**Risk**: If placeholder components don't render properly, the test acceptance criteria "Placeholder components render without errors" may fail. Implementation agent should decide the placeholder strategy and document it.

### 2. **Auto-skip integration point**

The story says "`plan-selection` auto-skip when `BILLING_ENABLED=false`" but does not specify:
- Should the route loader redirect (like the current flag-playlists auto-skip)?
- Should it be checked at step save time?
- Should the indicator hide the step?

**Assumption**: Route loader redirect is consistent with existing auto-skip patterns. The implementation agent should follow the flag-playlists pattern:
```typescript
// Skip plan-selection step if BILLING_ENABLED=false
if (search.step === "plan-selection" && !env.BILLING_ENABLED) {
	throw redirect({
		to: "/onboarding",
		search: { step: "ready" },
	});
}
```

And also handle the reverse: if user is on ready but BILLING_ENABLED=true and they haven't visited plan-selection, they should be redirected back to plan-selection (step progression validation).

### 3. **Indicator display for hidden steps**

Current code filters INDICATOR_STEPS based on `hideIndicator` flag. The new steps should appear in the indicator unless marked otherwise. If any of the three new steps should be hidden from the indicator, the config must include `hideIndicator: true`.

**Assumption**: None of the new steps should hide from the indicator (unlike `syncing` and `complete`). Implementation agent should verify with design/product.

### 4. **URL navigation guards with new steps**

The route loader's step progression validation uses `ONBOARDING_STEPS.options.indexOf()` to determine order. Adding new steps changes the indices for `ready` and `complete`. The validation logic should still work correctly (it's index-based, not name-based), but the implementation agent should test:
- User at `flag-playlists`, manually navigates to `plan-selection` — should succeed (forward)
- User at `flag-playlists`, manually navigates to `ready` — should fail and redirect to saved step (unless plan-selection is explicitly skipped)
- User at `song-showcase`, manually navigates back to `syncing` — should fail (backward navigation)

### 5. **Type safety: STEP_CONFIG must be exhaustive**

TypeScript will error if STEP_CONFIG doesn't include all enum values. The implementation agent must ensure all 10 steps have entries in STEP_CONFIG, or the build will fail.

**Verification**: After adding steps to enum and creating STEP_CONFIG entries, run `bun run typecheck` to confirm exhaustiveness.

### 6. **Sequence correctness**

The story specifies: `welcome → pick-color → install-extension → syncing → flag-playlists → song-showcase → match-showcase → plan-selection → ready → complete`

This is a **10-step sequence**, with the three new steps inserted **between flag-playlists and ready**. The implementation agent must not reorder or miss steps.

No new exports, no breaking changes to existing interfaces, no architectural changes.
