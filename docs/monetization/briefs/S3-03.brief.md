## Constraints

From **DECISIONS.md §4 — Read-Model Invariants**:
> "`item_status` is written only by the content-activation step, never by generic pipeline-completion"

From **DECISIONS.md §5 — Workflow Invariants / Content activation**:
> "Writes `item_status` only for entitled songs with `song_analysis`"
> "Persists unlimited unlock rows (`source='unlimited'` or `source='self_hosted'`) for songs that became account-visible but lack durable unlock rows"
> "Does not wait for embedding; analysis is the user-visible value"
> "Runs after shared stages, driven by current DB truth (not 'which songs ran B/C in this chunk')"

From **DECISIONS.md §2 — Effective entitlement predicate**:
> "`past_due` and `unpaid` subscription statuses do NOT grant unlimited access for `subscription`-sourced accounts"

From **DECISIONS.md §5 — Unlimited users**:
> "Unlock rows are written at content activation time for post-cancellation durability"
> "No pre-materialization of unlock rows for the whole library"

From **DECISIONS.md §6 — Terminology Invariants**:
- Unlock sources frozen: `free_auto`, `pack`, `unlimited`, `self_hosted`, `admin`
- Selector flags frozen: `needs_content_activation`

From **DECISIONS.md §7 — Non-Goals**:
> "Embedding required for content visibility — Analysis alone is the visibility threshold; embedding failure is handled separately"

From **S1-05 Brief** — `activate_unlimited_songs` RPC behavior:
- Finds account-visible songs = currently liked AND `song_analysis` exists
- Upserts `item_status` for them (`ON CONFLICT DO NOTHING`)
- Upserts `account_song_unlock` with `source='unlimited'` and subscription provenance
- Returns the set of account-visible song IDs
- Active (non-revoked) unlock rows are NOT updated — only revoked rows get re-activated

## Open Questions / Risks

### 1. `activate_unlimited_songs` requires non-nullable `p_granted_stripe_subscription_id`

The RPC signature has `p_granted_stripe_subscription_id: string` (non-nullable). For `self_hosted` accounts, there is no Stripe subscription ID. The S1-05 brief shows the RPC sets `source='unlimited'` unconditionally based on passed params — it doesn't differentiate `self_hosted`. The story says "Self-hosted users get unlock rows with `source='self_hosted'`", but the `activate_unlimited_songs` RPC hardcodes `source='unlimited'`. Whether content activation calls `activate_unlimited_songs` for self-hosted, uses a different RPC, or calls `insert_song_unlocks_without_charge` with `source='self_hosted'` directly is not defined in the codebase. There is no `activate_self_hosted_songs` RPC.

### 2. `BillingState` does not include `stripe_subscription_id` or `subscription_period_end`

`activate_unlimited_songs` requires both. These fields exist on `account_billing` Row but are not exposed in the `BillingState` type (normalized state omits raw Stripe IDs). The implementation agent must decide how to obtain these values — either by querying `account_billing` directly (separate from `readBillingState`), or by extending `BillingState` (type change), or via another approach.

### 3. The `needContentActivation` flag scope vs RPC scope

`workPlan.needContentActivation` contains song IDs that need activation from this batch. However, `activate_unlimited_songs` (per S1-05) internally queries ALL account-visible songs (all liked songs with `song_analysis`) — not just those in the current batch. This means calling it will also write unlock rows for songs outside the current batch. The story says "driven by current DB truth, not by which songs ran B/C in this chunk" and "Does not pre-materialize unlock rows for the whole library — only for songs that became visible in this batch." There is a design tension between the batch-scoped intent and the full-library scope of the existing RPC.

### 4. `item_status` written by both activation and `markPipelineProcessed` (until S3-04)

Currently `executeWorkerChunk` calls `markPipelineProcessed(accountId, "song", batch.songIds)` for ALL batch songs after `enrichSongs`. Content activation will also write `item_status` for the entitled + analyzed subset. Until S3-04 removes `markPipelineProcessed`, both writes co-exist. The DECISIONS.md invariant says `item_status` should be written ONLY by content activation — but S3-04 is the removal story, not this one.

### 5. `is_new` semantics for `activate_unlimited_songs` vs `markItemsNew`

The `activate_unlimited_songs` RPC sets `is_new=true` via `INSERT INTO item_status` (default value). `markPipelineProcessed` sets `is_new=false`. If content activation uses `activate_unlimited_songs`, the RPC handles `item_status` internally. If the implementation writes `item_status` directly from TS (e.g., via `markItemsNew`), then the `is_new=true` semantic is under TS control. Either approach is possible but leads to different code organization.

### 6. `newCandidatesAvailable` calculation unchanged

The story says "Compute `newCandidatesAvailable` delta from the before/after candidate snapshots." The current implementation already computes this via `loadDataEnrichedSongIds` before and after `enrichSongs`. This check requires audio features, analysis, embeddings, AND genres to all be present. Content activation does not change this logic — the delta is still computed the same way. The story mentions this as a deliverable but no change to the existing delta logic appears necessary.

### 7. Content activation for pack-unlock songs

Songs unlocked via `pack` source already have `account_song_unlock` rows. For these songs, content activation should write `item_status` but NOT call `activate_unlimited_songs` (which would overwrite their `source` if revoked, or be a no-op if active). The implementation must check `unlimitedAccess` kind before deciding which path to take.
