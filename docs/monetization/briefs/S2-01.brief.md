## Changes Required

### 1. Create `src/lib/domains/billing/state.ts` (new file)

```typescript
/**
 * Canonical billing domain types.
 *
 * These types overlay the raw `account_billing` DB row with narrowed,
 * product-meaningful values. All downstream consumers (workflows, loaders,
 * routes, UI) import from this file — never from database.types.ts directly.
 */

/** The account's current subscription plan. */
export type BillingPlan = "free" | "quarterly" | "yearly";

/**
 * How the account has unlimited access, if at all.
 * Discriminated union prevents treating absent access as a boolean.
 */
export type UnlimitedAccess =
  | { kind: "none" }
  | { kind: "subscription" }
  | { kind: "self_hosted" };

/**
 * Normalized subscription status.
 *
 * Raw Stripe statuses from `account_billing.subscription_status` are mapped
 * to this set. Callers should never compare against raw Stripe strings.
 *
 * Mapping:
 *   Stripe 'none'                 → 'none'
 *   Stripe 'active'               → 'active'
 *   Stripe 'canceled'             → 'none'         (terminal, access ended)
 *   Stripe 'unpaid'               → 'past_due'     (delinquent, grouped)
 *   Stripe 'past_due'             → 'past_due'
 *   Stripe 'incomplete'           → 'none'         (setup not completed)
 *   Stripe 'incomplete_expired'   → 'none'         (setup expired)
 *   cancel_at_period_end=true     → 'ending'       (override applied in BillingState construction)
 */
export type NormalizedSubscriptionStatus = "none" | "active" | "ending" | "past_due";

/**
 * Maps raw `account_billing.subscription_status` CHECK-constraint values to
 * the normalized set. `cancelAtPeriodEnd=true` overrides 'active' → 'ending'
 * at the BillingState construction site (not here — the mapping is purely for
 * the Stripe status string).
 */
export const STRIPE_STATUS_TO_NORMALIZED: Record<
  string,
  NormalizedSubscriptionStatus
> = {
  none: "none",
  active: "active",
  canceled: "none",
  unpaid: "past_due",
  past_due: "past_due",
  incomplete: "none",
  incomplete_expired: "none",
} as const;

/**
 * Resolved billing state for an account.
 *
 * Built from `account_billing` + conversion logic; not a direct DB row type.
 * Consumed by UI read models, loaders, and workflow queue-band resolution.
 */
export interface BillingState {
  plan: BillingPlan;
  creditBalance: number;
  /** Normalized subscription lifecycle status (Stripe statuses collapsed). */
  subscriptionStatus: NormalizedSubscriptionStatus;
  cancelAtPeriodEnd: boolean;
  unlimitedAccess: UnlimitedAccess;
  /** Resolved queue band for this account's pending jobs. */
  queueBand: "low" | "standard" | "priority";
}

/**
 * Display state for a single song in an account's library.
 *
 * Replaces `UIAnalysisStatus`. `locked` is new — supersedes both analysis
 * and matching status for non-entitled songs.
 */
export type SongDisplayState =
  | "locked"
  | "pending"
  | "analyzing"
  | "analyzed"
  | "failed";

/**
 * Derived helper: returns true when the account has any active unlimited access
 * (subscription or self-hosted). Does not gate on plan — `self_hosted` accounts
 * may have `plan='free'`.
 */
export function hasUnlimitedAccess(state: BillingState): boolean {
  return state.unlimitedAccess.kind !== "none";
}
```

**Key decisions:**
- `STRIPE_STATUS_TO_NORMALIZED` covers all 7 CHECK constraint values from `account_billing.subscription_status` (`none | active | past_due | canceled | unpaid | incomplete | incomplete_expired`). `cancelAtPeriodEnd` → `"ending"` override is a construction-site concern (S2-02), not a map entry.
- `queueBand` is included in `BillingState` as a derived field populated by the billing state query (S2-02). It keeps read models self-contained.
- No re-export of `QueueBand` from `queue-priority.ts` — the type is inlined as a literal union to avoid cross-module coupling at the type level. If a future refactor wants to unify, that's a separate story.
- `STRIPE_STATUS_TO_NORMALIZED` is exported so the query builder (S2-02) can use it without duplicating the mapping.

### 3. Modify `src/env.ts`

Add four billing vars to the `server` block and mirror them in `runtimeEnv`.

**Schema additions in `server` block:**
```typescript
// Billing integration
BILLING_ENABLED: z.coerce.boolean().default(false),
BILLING_SERVICE_URL: z.string().url().optional(),
BILLING_SHARED_SECRET: z.string().min(1).optional(),
QUARTERLY_PLAN_ENABLED: z.coerce.boolean().default(false),
```

**`runtimeEnv` additions:**
```typescript
BILLING_ENABLED: serverEnv.BILLING_ENABLED,
BILLING_SERVICE_URL: serverEnv.BILLING_SERVICE_URL,
BILLING_SHARED_SECRET: serverEnv.BILLING_SHARED_SECRET,
QUARTERLY_PLAN_ENABLED: serverEnv.QUARTERLY_PLAN_ENABLED,
```

**Why `z.coerce.boolean()`:** Env vars arrive as strings (`"false"`, `"true"`, `"0"`, `"1"`). `z.coerce.boolean()` correctly coerces them. `z.boolean()` would reject strings. The `.default(false)` ensures the vars are never `undefined` in validated env, satisfying "default false" without needing `.optional()`.

**Note on `BILLING_SERVICE_URL`:** Using `z.string().url()` not `z.url()` because `z.url()` is a Zod v4 shorthand — the codebase uses Zod v3 (confirmed by `import { z } from "zod"` with `@t3-oss/env-core` which pins to v3). Both validate URL format; `z.string().url()` is the v3 syntax used in the existing `SERVER_URL: z.url().optional()` — actually `z.url()` is also valid in v3. Use `z.url().optional()` to match the existing `SERVER_URL` pattern.

Revised:
```typescript
BILLING_SERVICE_URL: z.url().optional(),
```

## Risks / Ambiguities

### 1. `z.coerce.boolean()` vs `z.boolean()`

Env vars are always strings. `z.boolean()` rejects `"false"` as not a boolean. `z.coerce.boolean()` coerces `"false"` → `false`, `"true"` → `true`, `""` → `false`, `"0"` → `false`. With `emptyStringAsUndefined: true` already set in `createEnv`, `BILLING_ENABLED=` (empty string) becomes `undefined` then falls through to `.default(false)`. This is the safest combination.

However: check whether `@t3-oss/env-core` version in use supports `z.coerce.boolean()` with `.default()`. If the version predates coerce support in the env integration, use `z.enum(["true", "false"]).transform(v => v === "true").default("false")` instead. Either approach is acceptable; the former is cleaner.

### 2. `QueueBand` already exists in `queue-priority.ts`

`BillingState.queueBand` uses an inline literal union `"low" | "standard" | "priority"` rather than importing `QueueBand` from `queue-priority.ts`. This avoids a circular dependency risk (billing domain → workflow layer). If a future story wants to unify these, it can import the billing type into the workflow file instead.

### 3. `cancelAtPeriodEnd` → `"ending"` override

The `STRIPE_STATUS_TO_NORMALIZED` map does not handle `cancelAtPeriodEnd`. The override (`active` + `cancelAtPeriodEnd=true` → `ending`) is intentionally left to the `getBillingState` query (S2-02), which constructs the `BillingState` object. The mapping table covers only raw Stripe status strings.

### 4. No `index.ts` barrel

Per project convention (`CLAUDE.md`: "Don't use barrel exports"), neither `state.ts` nor `offers.ts` should be accompanied by an `index.ts`. Consumers must import directly from the specific file path.
