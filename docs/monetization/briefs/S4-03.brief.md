**Table: `pack_credit_lot`** (from `20260405110000_billing_pack_conversion_tables.sql`)
```sql
CREATE TABLE pack_credit_lot (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id        UUID NOT NULL REFERENCES account(id) ON DELETE CASCADE,
  stripe_event_id   TEXT NOT NULL UNIQUE,
  offer_id          TEXT NOT NULL,
  original_credits  INTEGER NOT NULL CHECK (original_credits > 0),
  remaining_credits INTEGER NOT NULL CHECK (remaining_credits >= 0 AND remaining_credits <= original_credits),
  price_cents       INTEGER NOT NULL CHECK (price_cents > 0),
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_pack_credit_lot_account_open
  ON pack_credit_lot(account_id, created_at)
  WHERE remaining_credits > 0;
```
- Index `idx_pack_credit_lot_account_open` efficiently finds open (remaining_credits > 0) lots for an account
- Will be queried by `prepare_subscription_upgrade_conversion` RPC

**Table: `subscription_credit_conversion`** (from `20260405110000_billing_pack_conversion_tables.sql`)
```sql
CREATE TABLE subscription_credit_conversion (
  id                      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id              UUID NOT NULL REFERENCES account(id) ON DELETE CASCADE,
  checkout_session_id     TEXT,
  target_plan             TEXT NOT NULL CHECK (target_plan IN ('quarterly', 'yearly')),
  status                  TEXT NOT NULL CHECK (status IN ('pending', 'applied', 'released', 'reversed')),
  converted_credits       INTEGER NOT NULL CHECK (converted_credits >= 0),
  discount_cents          INTEGER NOT NULL CHECK (discount_cents >= 0),
  stripe_subscription_id  TEXT,
  stripe_invoice_id       TEXT,
  applied_stripe_event_id TEXT,
  created_at              TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at              TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_subscription_credit_conversion_checkout_session
  ON subscription_credit_conversion(checkout_session_id)
  WHERE checkout_session_id IS NOT NULL;

CREATE UNIQUE INDEX idx_subscription_credit_conversion_pending_per_account
  ON subscription_credit_conversion(account_id)
  WHERE status = 'pending';
```
- `status` field drives the conversion lifecycle state machine
- `discount_cents` holds the pro-rated pack value computed by `prepare_subscription_upgrade_conversion`
- `converted_credits` holds the total reserved credits
- Unique index prevents duplicate checkout_session_id within active conversions

### Conversion Lifecycle RPCs (from S1-08)

Available through `supabase.rpc()` calls:

**`prepare_subscription_upgrade_conversion(account_id: UUID, target_plan: TEXT)`**
```typescript
// Call via: supabase.rpc("prepare_subscription_upgrade_conversion", {
//   p_account_id: accountId,
//   p_target_plan: "quarterly" | "yearly"
// })

// Returns: { converted_credits: number, discount_cents: number, conversion_id: UUID }

// Behavior:
// 1. Checks for existing pending conversion; reuses if found
// 2. Locks open pack_credit_lot rows (FIFO by created_at, id)
// 3. Creates subscription_credit_conversion_allocation rows for each lot
// 4. Computes discount_cents as: floor(remaining_credits * price_cents / original_credits) per lot
// 5. Inserts subscription_credit_conversion row with status='pending'
// 6. Returns computed values and conversion_id
// 
// Zero-lot case: Returns converted_credits=0, discount_cents=0, but conversion still created
// Reuse case: Returns existing pending conversion's values
```

**`link_subscription_upgrade_checkout(conversion_id: UUID, checkout_session_id: TEXT)`**
```typescript
// Call via: supabase.rpc("link_subscription_upgrade_checkout", {
//   p_conversion_id: conversionId,
//   p_checkout_session_id: sessionId
// })

// Returns: undefined (void)

// Behavior:
// Updates the pending conversion row to attach Stripe checkout session ID
// Raises exception if conversion not found or not in 'pending' status
// Idempotency: Unique index will reject duplicate checkout_session_id
```

**`release_subscription_upgrade_conversion(conversion_id: UUID)`**
```typescript
// Call via: supabase.rpc("release_subscription_upgrade_conversion", {
//   p_conversion_id: conversionId
// })

// Returns: undefined (void)

// Behavior:
// Sets conversion status to 'released'
// Virtual reservation is lifted (unlock_songs_for_account no longer subtracts from spendable)
// Does NOT physically reduce pack_credit_lot remaining_credits or credit_balance
// Raises exception if conversion not found or not in 'pending' status
```

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Offer ID Validation (TERMINOLOGY.md §2, Frozen)
- Canonical internal offer IDs: `unlimited_quarterly`, `unlimited_yearly`
- S4-03 must validate that incoming `offer_id` is one of these exact values
- Reject with error if invalid

### Quarterly Plan Gating (DECISIONS.md §6, Frozen)
- If `QUARTERLY_PLAN_ENABLED=false` and caller requests `unlimited_quarterly`: reject with error
- `unlimited_yearly` has no such flag and should always be available

### Stripe Price ID Resolution (DECISIONS.ms §2, Frozen)
- **Only billing service resolves Stripe price IDs**
- Billing service must know the mapping from `unlimited_quarterly` → Stripe price ID, `unlimited_yearly` → Stripe price ID
- Must store these as environment variables or configuration
- Current pattern: `env.STRIPE_PRICE_ID_SONG_PACK_500`; follow same pattern for unlimited prices

### Conversion Reservation Invariant (DECISIONS.md §3, Frozen)
- "Conversion reservation must be released on any failure after prepare"
- If `prepare_subscription_upgrade_conversion` succeeds but coupon or checkout creation fails, must call `release_subscription_upgrade_conversion` before returning error
- If prepare fails, no release needed (no reservation created)

### `conversion_id` in Metadata (Story Scope, DECISIONS.md §2)
- Set `subscription_data.metadata.conversion_id` when applicable
- Also set `subscription_data.metadata.account_id`
- Metadata persists to webhook (S4-05 uses it to find the conversion)

### Idempotency Key (DECISIONS.md §2, Frozen)
- Forward `checkout_attempt_id` as Stripe `idempotencyKey` on `checkout.sessions.create()`
- Ensures duplicate requests with same ID return same session
- Must be part of signed request body (already validated by HMAC middleware)

### Subscription Mode (Story Scope)
- Use `mode: "subscription"` for unlimited checkout (not `mode: "payment"` like pack)
- Unlimited is a recurring subscription; pack is one-time payment

### "Already unlimited" Check (Story Scope, DECISIONS.md §1)
- Reject checkout if account already has active unlimited access
- Check: `account_billing.plan IN ('quarterly', 'yearly')` OR `account_billing.unlimited_access_source IS NOT NULL`
- Cannot upgrade to unlimited if already unlimited

### Stripe Coupon Creation (Story Scope)
- Create one-time Stripe coupon with amount_off in cents
- Apply coupon to first invoice only (via `coupon` param on `subscription.create()` or checkout line item discount)
- If coupon creation fails, must release conversion before returning error

### Service Role Only (DECISIONS.md §3, Frozen)
- All Supabase operations use service-role key (already configured in `config.ts`)
- RPC calls are SECURITY DEFINER and bypass RLS

### HMAC Auth (DECISIONS.md §2, Frozen)
- Middleware already validates signature
- No additional auth checks needed in endpoint handler
- All `/api/*` requests are pre-authenticated

## Open Questions / Risks

### 1. Environment Variables for Unlimited Price IDs

**Issue:** `env.ts` does not yet define `STRIPE_PRICE_ID_UNLIMITED_QUARTERLY` and `STRIPE_PRICE_ID_UNLIMITED_YEARLY`.

**Current state:** Only `STRIPE_PRICE_ID_SONG_PACK_500` is defined.

**Decision needed:** Where should unlimited price IDs come from?
- Option A: Add two new env vars following the same pattern as pack
- Option B: Fetch from a database configuration table
- Option C: Stripe product lookup by product name

**Recommended:** Follow S4-02 pattern and add env vars `STRIPE_PRICE_ID_UNLIMITED_QUARTERLY` and `STRIPE_PRICE_ID_UNLIMITED_YEARLY` to `env.ts`.

**Risk:** If not configured correctly in production, endpoint will fail at startup with validation error.

### 2. `QUARTERLY_PLAN_ENABLED` in Billing Service

**Issue:** The flag is defined in v1_hearted app's `env.ts`, but S4-03 endpoint needs to validate it.

**Options:**
- Option A: Add `QUARTERLY_PLAN_ENABLED` to billing service `env.ts`
- Option B: Receive as part of request body (but this seems odd for a feature flag)
- Option C: Query database config table (doesn't exist)

**Recommended:** Add to billing service `env.ts` as a boolean with default=false, matching main app pattern.

**Risk:** Inconsistency between app and service if flags diverge.

### 3. Stripe Coupon vs Discount Creation

**Issue:** Stripe offers multiple ways to apply discounts:
- One-time coupon created via `stripe.coupons.create()`
- Discount object created via `stripe.discounts.create()`
- Coupon applied at checkout via `coupon` param

**Current state:** S4-02 doesn't use coupons; story specifies "create one-time Stripe coupon".

**Ambiguity:** Should the coupon be:
- Reusable (can apply to multiple customers/subscriptions)?
- Single-use (one-time, consumed after first use)?
- Time-limited (expires after N days)?

**Story hint:** "one-time Stripe coupon" suggests single-use or highly scoped to this conversion.

**Recommended:** Create coupon with `max_redemptions: 1` to prevent accidental reuse.

**Risk:** If coupon is reusable, it could be applied to wrong subscriptions if retrieved later.

### 4. Coupon Application to First Invoice Only

**Issue:** Stripe Checkout with subscriptions supports coupon application, but unclear whether coupon applies to first invoice only or all invoices.

**Stripe behavior:** By default, coupon applies to all invoices for the subscription lifetime.

**Story requirement:** "applied to first invoice"

**Implementation challenge:** May need custom discount logic or per-invoice discount handling in S4-05 webhook.

**Current state:** Not explicitly addressed; deferred to implementation.

**Risk:** If coupon applies to all invoices, user gets wrong discount amount.

### 5. "Already Unlimited" Check Correctness

**Issue:** Must determine if account already has active unlimited access.

**Current schema checks:**
- `account_billing.plan IN ('quarterly', 'yearly')` — indicates past/current plan
- `account_billing.unlimited_access_source = 'subscription'` — indicates currently unlimited
- `account_billing.subscription_status = 'active'` — indicates subscription is live

**Ambiguity:** What if subscription is `past_due` or `canceled`? User may have had unlimited but not anymore.

**Constraint from DECISIONS.md §2:** "past_due and unpaid subscription statuses do NOT grant unlimited access".

**Recommended check:**
```typescript
// Account has active unlimited if:
const hasActiveUnlimited = (
  (billing.unlimited_access_source === 'subscription' && 
   billing.subscription_status === 'active') ||
  billing.unlimited_access_source === 'self_hosted'
);

if (hasActiveUnlimited) {
  return c.json({ error: "Account already has active unlimited access" }, 400);
}
```

**Risk:** If check is wrong, user can double-subscribe.

### 6. Stripe Customer Reuse

**Issue:** S4-02 pattern is to look up `stripe_customer_id` and reuse if exists. Should S4-03 do the same?

**Current state:** Pack checkout reuses; no explicit guidance for unlimited.

**Assumption:** Yes, unlimited checkout should also reuse existing Stripe customer.

**Implementation requirement:** Query `account_billing.stripe_customer_id` and pass to `stripe.checkout.sessions.create()` if non-null.

**Risk:** If customer lookup fails, endpoint returns 500 without creating fallback customer.

### 7. Failure Cleanup Order

**Issue:** Story says "If coupon or Checkout creation fails: call `release_subscription_upgrade_conversion` before returning error".

**Sequence:**
1. Call `prepare_subscription_upgrade_conversion` → gets conversion_id
2. Call `stripe.coupons.create()` → may fail
3. Call `stripe.checkout.sessions.create()` → may fail
4. Call `link_subscription_upgrade_checkout()` → must succeed or link fails

**Current state:** Not explicit whether to release on coupon failure vs checkout failure.

**Recommended:** Release if either coupon or checkout creation fails, before returning error response.

**Implementation pattern:**
```typescript
try {
  const coupon = await stripe.coupons.create(...);
  const session = await stripe.checkout.sessions.create(...);
  await supabase.rpc("link_subscription_upgrade_checkout", {...});
  return c.json({ checkout_url: session.url });
} catch (error) {
  // Release conversion on any Stripe or link failure
  await supabase.rpc("release_subscription_upgrade_conversion", {
    p_conversion_id: conversionId
  });
  return c.json({ error: ... }, 500);
}
```

**Risk:** If release itself fails, error handling becomes complex.

### 8. Zero Discount Edge Case

**Issue:** If `prepare_subscription_upgrade_conversion` returns `discount_cents=0` (no open pack lots), should endpoint still create coupon?

**Options:**
- Option A: Skip coupon creation if discount_cents=0, create checkout without coupon
- Option B: Always create coupon, even if amount is 0

**Recommended:** Skip coupon creation if `discount_cents=0`; simpler and avoids unnecessary Stripe API calls.

**Risk:** Inconsistent behavior if logic is inverted.

### 9. Metadata Structure on Checkout Session vs Subscription

**Issue:** Story says "Set `subscription_data.metadata`" but Stripe Checkout has metadata at session level AND subscription level.

**Clarification needed:** Should metadata be set on:

- Subscription (readable in subscription.paid webhook)

**Current pattern in S4-02:** Sets metadata on checkout session only.

**Recommended:** Set on subscription_data.metadata so it persists to subscription webhooks (S4-05 needs this to find conversion).

**Risk:** If metadata is on session only, webhook handler can't access it from subscription object.

### 11. Coupon Idempotency

**Issue:** If the same request is retried with the same `checkout_attempt_id`, Stripe will return the same checkout session, but coupon creation may fail if coupon already exists.

**Stripe behavior:** No built-in idempotency key for coupon creation; each call creates a new coupon.

**Current state:** Story doesn't address coupon idempotency explicitly.

**Challenge:** If coupon creation is retried, we may create duplicate coupons with same amount.

**Options:**
- Option A: Reuse previous coupon if request is retried (requires lookup by conversion_id)
- Option B: Accept duplicate coupon creation (minimal impact, coupons don't cost money)
- Option C: Use coupon lookup by ID pattern

**Recommended:** For MVP, accept that retries may create duplicate coupons. Document this as acceptable for now.

**Risk:** Multiple coupons with same amount in Stripe account if retries happen.

### 12. Type Safety for Unlimited Offer IDs

**Issue:** Pack checkout uses `z.enum(["song_pack_500"])` for validation, but S4-03 needs to validate `"unlimited_quarterly" | "unlimited_yearly"`.

**Current state:** S4-03 must extend `VALID_OFFER_IDS` in checkout.ts to include unlimited offers, or create separate schema/handler.

**Options:**
- Option A: One `checkoutRoutes` handler with merged VALID_OFFER_IDS
- Option B: Separate `unlimitedCheckoutRoutes` handler
- Option C: Shared handler with conditional logic

**Recommended:** Keep separate handlers (`/api/checkout/pack` and `/api/checkout/unlimited`) for clarity. Each has its own schema and price map.

**Implementation pattern:**

### 13. Subscription Data vs Line Items

**Issue:** S4-03 is creating a subscription checkout, not a one-time payment. Stripe Checkout with `mode: "subscription"` requires `line_items` with price ID, plus optional `subscription_data` for metadata and other subscription parameters.

**Current implementation needed:**

**Ambiguity:** How to apply coupon?
- Option A: On line_item as discount
- Option B: On session as coupon
- Option C: Via subscription_data

**Stripe documentation:** Coupons apply at subscription level. May need to use `discounts: [{ coupon: couponId }]` in subscription_data.

**Risk:** Coupon may not apply correctly if Stripe API structure differs from assumption.

### 14. Conversion Lifecycle Edge Cases

**Issue:** What if `prepare_subscription_upgrade_conversion` returns a zero-discount conversion (no open lots) but the endpoint is called again with the same account?

**Current RPC behavior:** Reuses existing pending conversion, returns same values.

**Scenario:** 
1. User adds pack lots after first prepare call
2. Calls endpoint again

**Question:** Should endpoint call prepare again (to capture new lots) or reuse existing conversion?

**Story guidance:** "At most one pending conversion per account" (DECISIONS.md §1).

**Recommended:** Accept that prepare reuses existing conversion. If user adds lots after prepare, they must abandon checkout and start new one.

**Risk:** User confusion if pack is added after prepare but before payment.

## Summary

S4-03 is the most complex checkout endpoint: it must handle conversion lifecycle (prepare → link or release), Stripe coupon creation, and graceful failure handling. Key implementation decisions:

1. **Add env vars** for unlimited price IDs and `QUARTERLY_PLAN_ENABLED` flag
2. **Validate offer ID** against frozen list (`unlimited_quarterly`, `unlimited_yearly`)
3. **Check quarterly flag** before accepting quarterly offer
4. **Check "already unlimited"** condition before proceeding
5. **Call prepare RPC** if any open pack lots exist; skip if none
6. **Create coupon** (if discount > 0) with `max_redemptions: 1`
7. **Apply coupon** to subscription via subscription_data
8. **Create checkout session** in subscription mode with account_id and conversion_id in metadata
9. **Link conversion** to checkout session via RPC
