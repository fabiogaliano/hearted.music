### BillingState Type
**File:** `src/lib/domains/billing/state.ts`

**Relevant type definition:**
```typescript
export interface BillingState {
	plan: BillingPlan;  // "free" | "quarterly" | "yearly"
	creditBalance: number;
	subscriptionStatus: NormalizedSubscriptionStatus;  // "none" | "active" | "ending" | "past_due"
	cancelAtPeriodEnd: boolean;
	unlimitedAccess: UnlimitedAccess;  // { kind: "none" | "subscription" | "self_hosted" }
	queueBand: "low" | "standard" | "priority";
}

export function hasUnlimitedAccess(state: BillingState): boolean {
	return state.unlimitedAccess.kind !== "none";
}
```

**Note:** `subscription_period_end` date is stored in the database (`account_billing.subscription_period_end`) but **not exposed in BillingState**. Raw row is available in `readBillingState()` but not passed through.

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Billing Architecture (DECISIONS.md §1, §2, §3)

- **Portal launch**: Only when `BILLING_ENABLED=true` AND account has active subscription
  - Active subscription: `subscriptionStatus === "active"` AND `unlimitedAccess.kind === "subscription"`
  - Not for `past_due` status — access is revoked during delinquency (§2, effective entitlement predicate)

- **Pack purchase entry point**: Only when provider-enabled AND account does not have unlimited access
  - Hidden when: `hasUnlimitedAccess(billingState) === true`
  - Visible when: provider-enabled AND `unlimitedAccess.kind === "none"`

- **Self-hosted deployment**: Shows simple "Unlimited" status with no purchase actions
  - `unlimited_access_source === "self_hosted"` (§3, provider-disabled access)
  - Still visible in billing section, but no portal or pack buttons

- **Provider-disabled**: No billing section or minimal status view
  - When `BILLING_ENABLED=false`, consider hiding billing section entirely or showing minimal "Unlimited (Self-hosted)" status
  - No purchase action buttons

### Terminology (TERMINOLOGY.md §2, §5)

- **Frozen plan values**: `free`, `quarterly`, `yearly`
- **Frozen offer IDs**: `song_pack_500`, `unlimited_quarterly`, `unlimited_yearly`
- **Frozen normalized subscription statuses**: `none | active | ending | past_due`
- **Frozen unlimited access sources**: `null` (none) | `'subscription'` | `'self_hosted'`
- **Frozen user-facing plan language**:
  - Free: "Free Plan"
  - Quarterly: "3-Month Unlimited"
  - Yearly: "Backstage Pass" (brand name)
  - Pack: "Song Pack"
- **User-facing balance**: "X songs to explore" (via `getDisplayBalance()`)

## Open Questions / Risks

### 1. Subscription Period End Date Not in BillingState

**Issue:** Story specifies "Subscription period end date (if applicable)" should be shown, but `BillingState` does not include `subscription_period_end`. The date exists in the database (`account_billing.subscription_period_end`) but is not exposed in the read model returned by `getBillingState()`.

**Unclear:**
- Should the period end date be added to `BillingState` type? (requires extending the type and queries)
- Or should the story accept showing only the status (active/ending/past_due) without the specific date?
- If adding it, formatting required? (ISO 8601, relative like "1 month remaining", etc.)

**Impact:** Implementation agent must decide whether to extend BillingState or interpret "period end" as captured by `subscriptionStatus: "ending"`.

### 2. Portal Button Visibility Logic

**Issue:** Story says "Portal launch button (provider-enabled + has subscription)". Multiple interpretations of "has subscription":
- Option A: `unlimitedAccess.kind === "subscription"` (has or had a subscription)
- Option B: `subscriptionStatus === "active"` (subscription currently active)
- Option C: Both conditions together

**Unclear:**
- Should the button show for `past_due` status? (Likely NO per DECISIONS.md §2 effective entitlement, but not explicitly stated for this button)
- Should the button show for `ending` status? (Likely YES, user can manage before it ends)
- Should both conditions be `active` OR `ending`? Or only `active`?

**Impact:** User can navigate to portal mid-cancellation or after delinquency, affecting error handling in portal flow.

### 3. Pack Purchase Entry Point — Hidden vs Disabled vs Removed

**Issue:** Story says "Pack entry points hidden while unlimited is active" but doesn't specify:
- Option A: Conditionally render `display: none` (hidden, takes no space)
- Option B: Conditionally render to `null` (not rendered at all)
- Option C: Render but disabled with explanatory tooltip

**Impact:** UI layout and user intent signal. If hidden, section may look empty. If disabled, user sees why it's unavailable.

### 4. Provider-Disabled Billing Section Visibility

**Issue:** Story says "Provider-disabled: no billing section, or minimal status view" — unclear threshold.

**Unclear:**
- Should billing section be completely hidden when `BILLING_ENABLED=false`?
- Or always show a minimal status section (e.g., "Unlimited (Self-hosted)" text) for consistency?
- Does "no purchase actions" include hiding the entire section, or just hiding buttons?

**Impact:** UX consistency and code simplicity. Full removal is simpler; minimal status provides context.

### 5. Subscription Status Display Format

**Issue:** Story mentions showing "Subscription status (active, ending, past_due)" but doesn't specify:
- User-facing labels (e.g., "Active", "Canceling", "Overdue")
- Whether to show reason for `past_due` or other diagnostic info
- Casing and punctuation

**Impact:** Copy tone and clarity. Internal values are lowercase; user-facing should be capitalized and readable.

### 6. Billing Section Placement in Settings

**Issue:** Current SettingsPage has sections in order: Profile, Theme, Extension, Sign Out. Story doesn't specify where billing section should go.

**Unclear:**
- Should billing come before or after the extension status?
- Above or below the separator that precedes sign out?
- Does it need its own visual hierarchy (icon, larger heading, etc.) or match extension status row pattern?

**Impact:** Visual hierarchy and reading order. Profile → Billing → Theme → Extension → Sign Out (logical flow from account to preferences) is one option.

### 7. Error Handling for Portal/Checkout Navigation

**Issue:** `createPortalSession` and `createCheckoutSession` return typed error responses, not exceptions. Settings page must handle these.

**Unclear:**
- What should happen if portal button is clicked and the function returns `{ error: "billing_service_error" }`?
- Show error toast? Inline error message? Disable button?
- Should error messages be user-facing or technical?

**Impact:** User feedback and error recovery. Toast + logging recommended; exact copy TBD.

### 8. Loading State During Portal/Checkout Request

**Issue:** Calling `createPortalSession()` or `createCheckoutSession()` is async. Settings page must signal loading state during the call.

**Unclear:**
- Should button show loading spinner, change text to "Loading…", become disabled?
- Should there be a global loading indicator?
- How long is a typical request (for UX feedback)?

**Impact:** User perceives responsiveness. Button state change is minimum viable.

### 9. Redirect Behavior After Checkout URL

**Issue:** `createCheckoutSession()` returns a `checkoutUrl`. Settings page must navigate to it. Story doesn't specify mechanism.

**Unclear:**
- Use `window.location.href = checkoutUrl` (full page unload)?
- Use router navigation if checkoutUrl is internal (unlikely)?
- Open in new tab?

**Impact:** Session state, browser history, and user expectations. Likely a full redirect via `window.location.href`.

### 10. Balance Display Integration with Billing Section

**Issue:** `getDisplayBalance()` returns balance for sidebar. Settings billing section should also show balance for pack users.

**Unclear:**
- Reuse `getDisplayBalance()` or compute independently?
- Display alongside plan label (e.g., "Song Pack - 500 songs") or separately below status?
- Include in status line or own row?

**Impact:** Code reuse and UI consistency. Reusing the helper maintains single source of truth.

### 11. Pack Entry Point Target — Checkout or Dedicated Page

**Issue:** Story says "pack purchase entry point" but doesn't specify destination.

**Unclear:**
- Should it navigate to `/checkout?offer=song_pack_500` (if that route exists)?
- Or directly call `createCheckoutSession()` like portal button?
- Or navigate to a dedicated pack purchase page (`/purchase/pack`)?

**Impact:** Implementation scope and route structure. If no dedicated page exists, likely calls function directly.

### 12. Quarterly Plan Visibility

**Issue:** Yearly unlimited is always shown, but quarterly is gated behind `QUARTERLY_PLAN_ENABLED` flag.

**Unclear:**
- If quarterly is disabled, should "Buy more packs" entry point still show, or should settings show only yearly unlimited option?
- Does billing section need to reflect quarterly flag, or is it orthogonal?

**Impact:** Likely orthogonal — billing section shows current state; flag only affects checkout. But needs verification.
