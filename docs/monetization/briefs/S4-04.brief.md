### Account Billing Database Schema
**File:** `v1_hearted/supabase/migrations/20260405100000_billing_core_tables.sql`

```sql
CREATE TABLE account_billing (
  account_id    UUID PRIMARY KEY REFERENCES account(id) ON DELETE CASCADE,
  plan          TEXT NOT NULL DEFAULT 'free'
                  CHECK (plan IN ('free', 'quarterly', 'yearly')),
  credit_balance INTEGER NOT NULL DEFAULT 0
                  CHECK (credit_balance >= 0),
  unlimited_access_source TEXT
                  CHECK (unlimited_access_source IN ('subscription', 'self_hosted')),
  
  -- Stripe refs (null until first interaction)
  stripe_customer_id       TEXT UNIQUE,
  stripe_subscription_id   TEXT UNIQUE,
  subscription_status      TEXT NOT NULL DEFAULT 'none'
                  CHECK (subscription_status IN (
                    'none', 'active', 'past_due',
                    'canceled', 'unpaid', 'incomplete', 'incomplete_expired'
                  )),
  subscription_period_end  TIMESTAMPTZ,
  cancel_at_period_end     BOOLEAN NOT NULL DEFAULT FALSE,
  
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

The `stripe_customer_id` column is nullable and unique; it is populated when a customer is created (either during checkout or portal session creation).

### HMAC Auth Middleware
**File:** `v1_hearted_brand/src/middleware/hmac-auth.ts`

```typescript
export function hmacAuth(secret: string): MiddlewareHandler {
  // Validates:
  // - X-Timestamp header (Unix epoch seconds)
  // - X-Signature header (hex(HMAC-SHA256(secret, timestamp + "." + hex(SHA256(body)))))
  // - Clock skew: ±5 minutes from now
  // Rejects missing headers, invalid timestamp, expired timestamp, or invalid signature
}
```

Portal `/portal/session` endpoint should be mounted at `/api/portal/session` to require HMAC auth (already applied to `/api/*` prefix).

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Architecture Boundary (DECISIONS.md §2)
- **Only `v1_hearted_brand/` imports Stripe SDK** — app (`v1_hearted/`) never sees Stripe price IDs or portal configuration IDs
- **Service-role Supabase client only** — all DB operations use service-role key, never anon or authenticated
- **Portal is Stripe-hosted** — service only creates the session URL and returns it to the app

### Billing State (DECISIONS.md §2)
- **Entitlement predicate:** A song is entitled when:
  - `account_song_unlock` row exists with `revoked_at IS NULL`, **OR**
  - Account has active unlimited access (`unlimited_access_source IS NOT NULL` with valid lifecycle state)
- Portal is a **management surface only** — it does not drive entitlements; entitlements are controlled by `account_billing` and `account_song_unlock` schema

### Portal Session Rules (from story constraints)
- **Plan switching disabled** — user must re-subscribe after current term ends; no in-portal plan changes
- **Only accounts with `stripe_customer_id` can create portal sessions** — if null, endpoint rejects with appropriate error
- **No business-idempotency contract** — duplicate session creation is acceptable (unlike checkout, which uses `checkout_attempt_id`)

### Frozen Terminology (TERMINOLOGY.md §2, §6)
- **Plans:** `free`, `quarterly`, `yearly`
- **Offer IDs:** `song_pack_500`, `unlimited_quarterly`, `unlimited_yearly`
- **Env flags:** `BILLING_ENABLED`, `BILLING_SERVICE_URL`, `BILLING_SHARED_SECRET`, `QUARTERLY_PLAN_ENABLED`

### Stripe Isolation (DECISIONS.md §2)
- Stripe identifiers (`stripe_customer_id`, `stripe_subscription_id`, `stripe_event_id`) are opaque audit columns
- `stripe_customer_id` is the only Stripe ref needed to create a portal session
- App sends internal offer IDs; billing service translates to/from Stripe price IDs

## Open Questions / Risks

### 1. Stripe Portal Configuration ID
**Issue:** Story specifies portal config requirements (cancel enabled, payment update enabled, plan switching disabled) but does not mention whether to use a specific portal configuration ID or rely on defaults.

**Current state:** Stripe's default configuration has:
- Cancel subscription: **enabled** ✓
- Switch plan: **disabled** (default is OFF) ✓
- Payment method update: **requires inspection** (not guaranteed enabled by default)

**Unclear:** Should the endpoint:
- Use Stripe's default configuration (no `configuration` parameter)?
- Require a specific portal configuration ID in environment variables?
- Assume a pre-created portal configuration exists in Stripe Dashboard?

**Impact:** If payment method update is not enabled in the default configuration, session creation will succeed but users won't see the payment method update option. Test should verify the feature is present in the returned session's configuration.

### 2. Error When `stripe_customer_id` is NULL
**Issue:** Story says "Rejects accounts without `stripe_customer_id`" but does not specify:
- HTTP status code (400, 403, 422?)
- Error message format
- Whether to create a customer (as checkout endpoints do) or strictly require pre-existing customer

**Current pattern in checkout.ts:** If no customer exists, endpoint creates one automatically and persists to DB.

**Story constraint:** "Only accounts with `stripe_customer_id` can create portal sessions" — this suggests portal endpoint should **not** create a customer, unlike checkout.

**Clarification needed:** Should endpoint:
- Return 400/404 with error message if customer doesn't exist?
- Return 403 Forbidden (access denied without customer)?
- Attempt to create customer if missing?

### 3. Return URL Configuration
**Issue:** Stripe portal session requires a `return_url` (URL where user returns after portal). Story does not specify what this URL should be.

**Pattern:** Checkout uses `env.APP_BRIDGE_URL` as success/cancel redirect; portal should have an equivalent.

**Options:**
- Use `env.APP_BRIDGE_URL` + path, e.g., `/settings/billing`?
- Require a new env var like `STRIPE_PORTAL_RETURN_URL`?
- Hard-code a known path on app domain?

**Current env:** `APP_BRIDGE_URL` exists; unclear if it's appropriate for portal return.

### 4. Request Body Schema
**Issue:** Story specifies request body as `{ account_id }` but does not mention whether endpoint should also accept optional parameters like:
- `return_url` override?
- `locale` (language preference)?
- Flow-data parameters (deep-linking to cancel/payment-update flows)?

**Clarification:** Should endpoint accept only `account_id`, or support optional parameters?

### 5. Idempotency & Duplicate Sessions
**Issue:** Story says "No business-idempotency contract needed for portal (duplicate sessions acceptable)". Checkout uses `checkout_attempt_id` forwarded as Stripe's `idempotencyKey`.

**Decision made:** Portal endpoint does **not** require `checkout_attempt_id`; duplicate session creation is allowed. Each request creates a new session, each with a short-lived URL.

**Verify:** Confirm that portal session creation is safe to call multiple times (it is, per Stripe docs).

### 6. Portal Configuration Features Verification
**Issue:** Acceptance criteria state specific features must be enabled/disabled:
- "Cancel subscription enabled"
- "Payment method update enabled"
- "Plan switching disabled"

**Stripe response:** The `billingPortal.sessions.create()` call returns a session object, **not** the configuration object. The configuration is inspected separately via `stripe.billingPortal.configurations.retrieve()`.

**Test implication:** Integration test must:
- Create a portal session
- Optionally fetch the configuration to verify features
- Redirect user to `session.url` and verify UI (out of scope for backend test)

### 7. Test Isolation & Stripe Test Mode
**Issue:** Story mentions "Integration test with Stripe test mode" — indicates test should use live Stripe API (not mocked).

**Current pattern:** `checkout.test.ts` uses mocked Stripe client (via `vi.mock`).

**For S4-04:** A separate integration test file should:
- Use real Stripe test API key
- Create a test customer with known ID
- Call endpoint with HMAC signature
- Verify returned portal URL is valid
- Should NOT be part of unit test suite (`tests/` directory may run unit tests with mocks)

### 8. Portal Session Lifespan
**Issue:** Stripe portal session URLs are short-lived and single-use. App must redirect user immediately.

**No action needed:** This is Stripe behavior; endpoint simply returns URL and app handles redirect. No storage of session ID needed (unlike checkout sessions, which are tracked for webhook matching).

### 9. Customer Provisioning During Portal Creation
**Current decision:** Portal endpoint does NOT create customers (unlike checkout). But what if:
- Account exists (has `account_id`)
- No `stripe_customer_id` yet (customer not created)

**Options:**
- Reject request (as per story: "only accounts with `stripe_customer_id`")
- Create customer on-demand (like checkout does)

**Story language:** "Only accounts with `stripe_customer_id` can create portal sessions" suggests strict rejection, but is inconsistent with user experience (user might reach portal CTA without ever checking out first).

**Likely intent:** First-time users go through checkout (which creates customer); portal is for existing customers. Reject if no customer.

## Summary

S4-04 is a straightforward portal session endpoint that:
1. Validates HMAC auth (inherited from S4-01 middleware)
2. Looks up `stripe_customer_id` from `account_billing`
3. Creates a Stripe billing portal session
4. Returns the short-lived portal URL

**Major unknowns:**
1. Whether to use default Stripe configuration or a specific config ID
2. Error status code/message when customer not found
3. Return URL destination (where user returns after exiting portal)
4. Whether integration test should use mocked or real Stripe API
5. Whether to create customer on-demand or require pre-existing customer

All are standard implementation decisions; the infrastructure (Stripe client, Supabase query pattern, Hono routing, HMAC auth) is ready.
