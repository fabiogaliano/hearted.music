## Codebase Snapshot

### Current `get_liked_songs_stats` SQL (last defined in `20260402235223_rename_match_context_to_match_snapshot.sql`, lines 261–340)

```sql
DROP FUNCTION IF EXISTS get_liked_songs_stats(UUID);

CREATE OR REPLACE FUNCTION get_liked_songs_stats(p_account_id UUID)
RETURNS TABLE (
  total          BIGINT,
  analyzed       BIGINT,
  matched        BIGINT,
  has_suggestions BIGINT,
  new_suggestions BIGINT,
  pending        BIGINT
) AS $$
DECLARE
  v_latest_snapshot_id UUID;
BEGIN
  SELECT ms.id INTO v_latest_snapshot_id
  FROM match_snapshot ms
  WHERE ms.account_id = p_account_id
  ORDER BY ms.created_at DESC
  LIMIT 1;

  RETURN QUERY
  SELECT
    COUNT(*),
    -- analyzed: songs with any song_analysis row (NOT entitlement-gated)
    COUNT(*) FILTER (WHERE EXISTS (
      SELECT 1 FROM song_analysis sa WHERE sa.song_id = ls.song_id
    )),
    -- matched: all match_results covered by decisions
    COUNT(*) FILTER (WHERE EXISTS (...) AND NOT EXISTS (...)),
    -- has_suggestions: at least one undecided match_result
    COUNT(*) FILTER (WHERE EXISTS (...undecided mr...)),
    -- new_suggestions: undecided match_result AND is_new = true
    COUNT(*) FILTER (WHERE EXISTS (...undecided mr...) AND EXISTS (...ist.is_new = true...)),
    -- pending: NO item_status row (includes non-entitled/locked songs)
    COUNT(*) FILTER (WHERE NOT EXISTS (
      SELECT 1 FROM item_status ist
      WHERE ist.item_id = ls.song_id
        AND ist.account_id = ls.account_id
        AND ist.item_type = 'song'
    ))
  FROM liked_song ls
  WHERE ls.account_id = p_account_id
    AND ls.unliked_at IS NULL;
END;
$$ LANGUAGE plpgsql STABLE;
```

**Problems:**
- `analyzed` counts songs with any `song_analysis` row regardless of entitlement — non-entitled songs inflate the count
- `pending` counts songs with no `item_status` row regardless of entitlement — locked songs inflate the count
- No `locked` count column exists

CREATE OR REPLACE FUNCTION count_analyzed_songs_for_account(p_account_id UUID)

## Constraints

From **DECISIONS.md §4 (Read-Model Invariants)**:

> | `locked` | Not entitled, regardless of shared cache state |
> | `pending` | Entitled, queued for processing, not yet started |
> | `analyzed` | Entitled, content visible |
> - Missing `item_status` for a non-entitled song means `locked`, not `pending`
> - Read models must filter by effective entitlement, not by `song_analysis` or `item_status` existence
> - `song_analysis` rows are global (not account-scoped); existence does not mean the account may see that analysis
> - Dashboard stats (`analyzedPercent`, etc.) must be billing-aware

From **DECISIONS.md §2 (Effective entitlement predicate)**:

> A song is entitled when either:
> - an `account_song_unlock` row exists with `revoked_at IS NULL`, **or**
> - `unlimited_access_source IS NOT NULL` with valid lifecycle state
> - `past_due` and `unpaid` subscription statuses do **not** grant unlimited access for `subscription`-sourced accounts

From **DECISIONS.md §3 (Self-hosted)**:

> When `BILLING_ENABLED=false`, account provisioning grants `unlimited_access_source = 'self_hosted'` — these must see all songs as entitled.
> SQL never infers deployment mode from missing rows or absent billing state.

Frozen type name: `SongDisplayState` = `locked | pending | analyzing | analyzed | failed` (TERMINOLOGY.md §5 / DECISIONS.md §4).

## Open Questions / Risks

1. **`matched`, `has_suggestions`, `new_suggestions` entitlement gating.** The story scope mentions adjusting `analyzing` and `failed` counts for entitlement, but the current SQL has `matched`, `has_suggestions`, and `new_suggestions` columns (not `analyzing`/`failed`). It is unclear whether these three match-related counts should also be entitlement-gated (only counting entitled songs with match activity) or left ungated. The story does not explicitly address them.

2. **`analyzing` and `failed` counts not present in current SQL.** The story says "Existing `analyzing` and `failed` counts adjusted for entitlement" but neither `analyzing` nor `failed` columns exist in the current `get_liked_songs_stats` return type. The current schema has `matched`, `has_suggestions`, `new_suggestions`, `analyzed`, `pending`, `total`. The implementation agent must decide whether to add `analyzing`/`failed` columns or interpret this as applying entitlement to the existing columns.

3. **Return type change triggers cascade updates.** Adding `locked` to the SQL RETURNS TABLE requires updating: (a) `database.types.ts` manually or via `bun run gen:types`, (b) `LikedSongsStatsResult` in `liked-songs.functions.ts`, and (c) the handler's return object. Any UI consumer of `LikedSongsStatsResult` that destructures the success shape must be checked; currently `src/features/liked-songs/queries.ts` only passes it through, but route/component consumers may use specific fields.

4. **`LANGUAGE plpgsql STABLE` vs `LANGUAGE sql STABLE`.** The current function uses `LANGUAGE plpgsql` (requires `DECLARE`/`BEGIN`/`END` for the snapshot lookup). S3-09 uses `LANGUAGE sql`. Since the stats function still needs the `v_latest_snapshot_id` variable for match-related counts, it likely must remain `LANGUAGE plpgsql` — unless the snapshot lookup is inlined as a subquery.

5. **`total` count semantics.** The story says "Total counts still sum correctly." Currently `total` is simply `COUNT(*)` of all active liked songs, including locked ones. Whether `total = locked + pending + analyzed + (other)` is the intended invariant — or whether `total` remains the raw liked song count — is not explicitly stated.

6. **Timestamp for new migration.** The latest migration is `20260405220000`. The new migration should use `20260405230000` as the prefix.
