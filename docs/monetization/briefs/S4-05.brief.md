### RPCs (S1-06)

**`fulfill_pack_purchase(p_account_id UUID, p_stripe_event_id TEXT, p_offer_id TEXT, p_credits INTEGER, p_price_cents INTEGER) → JSONB`**
```
Returns: { "new_balance": INTEGER, "bonus_unlocked_song_ids": UUID[] }
```
- **Idempotency:** `ON CONFLICT (stripe_event_id) DO NOTHING` on `pack_credit_lot` insert. Second call with same `stripe_event_id` returns current state without mutations.
- **Bonus unlocks:** Up to 25 most-recent liked songs, `source='pack'`, tagged with `granted_stripe_event_id = p_stripe_event_id`
- **Balance mutation:** Increments `credit_balance` and writes `credit_transaction` with `reason='pack_purchase'`
- **Final step:** Calls `reprioritize_pending_jobs_for_account(p_account_id)`
- **Callable from service-role RPC context only** (SECURITY DEFINER)

### Checkout Routes (S4-01)

**Pack checkout** (`POST /api/checkout/pack`):
- Validates request: `account_id`, `offer_id` (`song_pack_500`), `checkout_attempt_id`
- Creates/fetches Stripe customer (stored in `account_billing.stripe_customer_id`)
- Creates Stripe checkout session with:
  - `mode: 'payment'`
  - `metadata: { account_id }`
  - `idempotencyKey: checkout_attempt_id`
- Returns `{ checkout_url: string }`

**Subscription checkout** (`POST /api/checkout/unlimited`):
- Validates request: `account_id`, `offer_id` (`unlimited_quarterly` or `unlimited_yearly`), `checkout_attempt_id`
- Checks for existing active unlimited (blocks if present)
- Creates Stripe customer if needed
- Calls RPC `prepare_subscription_upgrade_conversion` to reserve pack value for discount
- Creates coupon for discount (if discount > 0)
- Creates Stripe checkout session with:
  - `mode: 'subscription'`
  - `subscription_data: { metadata: { account_id, conversion_id } }`
  - `idempotencyKey: checkout_attempt_id`
- Links conversion to checkout via `link_subscription_upgrade_checkout` RPC

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Stripe Webhook Signature Verification
- **Method:** Stripe's own mechanism, **not** the app's HMAC auth middleware
- **Stripe function:** `Stripe.webhooks.constructEvent(rawBody, sig, webhookSecret)` (or equivalent in Stripe SDK)
- **Rejection:** Invalid signatures must return 401
- **Raw body requirement:** Stripe signature verification requires the raw request body (before JSON parsing)

### billing_webhook_event Idempotency Pattern
From story scope:
```sql
INSERT INTO billing_webhook_event (stripe_event_id, status) 
VALUES (?, 'processing') 
ON CONFLICT DO NOTHING
```
- **Winner detection:** Check `INSERT ... ON CONFLICT DO NOTHING` result; only the first insert succeeds
- **Webhook winner:** Subsequent webhooks with same `stripe_event_id` find the row already in `processing` or `processed`/`failed`
- **Status progression:** `processing` → `processed` (success) or `failed` (error)
- **Processed events:** Skip processing if status is already `processed`
- **Failed events:** Allow retry (status remains `failed` to allow manual recovery or reprocessing)

### Event Dispatch Router
- Route by `event.type` (Stripe's event type: `checkout.session.completed`, `invoice.paid`, etc.)
- Scope for S4-05: Only `checkout.session.completed` (with mode=payment and mode=subscription)
- Future stories (S4-06, S4-07) add more event types

### Pack Fulfillment Handler (`checkout.session.completed`, mode=payment)
- **Extract `account_id`:** From `session.metadata.account_id`
- **RPC call:** `fulfill_pack_purchase(account_id, stripe_event_id, offer_id, credits, price_cents)`
  - `stripe_event_id` from `event.id`
  - `offer_id` must be inferred from Stripe session (e.g., by matching line item price to `STRIPE_PRICE_ID_SONG_PACK_500`)
  - `credits` is the pack value (500 for `song_pack_500`)
  - `price_cents` is the session's payment amount
- **Bridge call:** `POST` to `APP_BRIDGE_URL` with:
  - Body: JSON with `stripe_event_id` and `bonus_unlocked_song_ids` (from RPC response)
  - HMAC signature: `X-Timestamp`, `X-Signature` headers (same format as incoming app→service calls)
  - Timing-safe comparison (use `crypto.subtle` or library utility)
- **Mark event:** Update `billing_webhook_event` to `processed` on success; `failed` on error

### Subscription Checkout Handler (`checkout.session.completed`, mode=subscription)
- **Store refs only:** No entitlement grant (activation happens on `invoice.paid` in S4-06)
- **Update `account_billing`:**
  - `stripe_subscription_id = session.subscription`
  - `subscription_status = 'incomplete'` or fetch and set from subscription object
  - `subscription_period_end = (compute from subscription period)` (optional for S4-05; may be filled in later)
- **Mark event:** Update `billing_webhook_event` to `processed`

### HMAC Signature Format for Bridge Calls
(Same pattern as hmacAuth middleware for consistency)
- **Headers:**
  - `X-Timestamp`: Unix epoch seconds
  - `X-Signature`: hex(HMAC-SHA256(BILLING_SHARED_SECRET, `${timestamp}.${hex(SHA256(body))}`))
- **Clock skew window:** 5 minutes (CLOCK_SKEW_MS = 5 * 60 * 1000)
- **Timing-safe comparison:** Use constant-time comparison to prevent timing attacks

## Open Questions / Risks

### 1. Missing `event_type` Column in `billing_webhook_event`
**Issue:** Story scope mentions `INSERT INTO billing_webhook_event (stripe_event_id, event_type, status)` but the actual S1-03 migration creates the table without an `event_type` column.

**Impact:** The webhook handler must dispatch on `event.type` from the Stripe event object at runtime. Event type is not stored in the database for this story's scope.

**Unclear:** Whether a future story will add event type storage, or whether the current pattern (dispatch on Stripe event type directly) is intentional.

### 2. Bridge Call Failure Handling
**Issue:** When pack fulfillment succeeds but the bridge call fails, the `billing_webhook_event` is marked `processed`. However, the app won't receive the notification, and control-plane changes won't be triggered.

**Risk:** Asymmetric state: billing service believes it fulfilled the pack; app doesn't know. The story doesn't specify retry/reconciliation for failed bridge calls.

**Unclear:** Should failed bridge calls:
- Keep `billing_webhook_event` as `failed` to allow manual retry?
- Be retried automatically (with exponential backoff)?
- Be logged and escalated to monitoring/alerting?

The story says "Bridge call must include HMAC signature" but doesn't specify failure recovery.

### 3. Stripe Signature Verification Method
**Issue:** The story says "Stripe signature verification (not HMAC — Stripe's own verification)" but doesn't specify the exact library call or error handling.

**Clarification:** Stripe SDK provides `stripe.webhooks.constructEvent(body, sig, secret)`, which throws on invalid signature. The handler must catch and return 401.

**Unclear:** How to handle edge cases:
- Stripe sends unsigned webhook (missing signature header)?
- Signature header present but malformed?
- Should these be logged differently than intentional attacks?

### 4. Offer ID Inference from Stripe Line Items
**Issue:** The story doesn't specify how the webhook handler will map a Stripe checkout session's line item price to the internal `offer_id` (e.g., `song_pack_500`).

**Pattern available:** Checkout routes have `STRIPE_PRICE_MAP` with the mapping.

**Unclear:** Should the webhook handler:
- Reverse-lookup the offer_id from `STRIPE_PRICE_ID_SONG_PACK_500` env var?
- Query the session's line item price and match against the map?
- Store `offer_id` in session metadata at checkout creation time?

The cleanest approach would be to store `offer_id` in session metadata at checkout time, but checkout routes don't currently do this.

### 5. Subscription Checkout Storage Scope
**Issue:** The story says "store customer/subscription refs only" for `checkout.session.completed (mode=subscription)` but the RPC and database updates are in scope for S4-06 (`invoice.paid` handler), not S4-05.

**Unclear:** What exactly does S4-05 store for subscription checkouts?
- The `stripe_subscription_id` in `account_billing`?
- Just acknowledge the event without DB writes?
- Fetch the subscription object and store the period end?

The story explicitly says S4-05 only stores refs without granting access, which suggests database writes are needed.

### 6. Raw Request Body Handling in Hono
**Issue:** Stripe signature verification requires the raw request body as a string, before JSON parsing. Hono may have already consumed or buffered the body stream.

**Risk:** `c.req.raw.clone().text()` pattern (used in hmacAuth middleware) works, but requires careful body handling to ensure it's available for both signature verification and JSON parsing.

**Clarification:** The hmacAuth middleware already uses `c.req.raw.clone().text()` successfully, so the pattern is known to work in this codebase.

### 7. Concurrency: Multiple Webhooks for Same Event
**Issue:** Stripe can deliver the same webhook multiple times due to network issues, retries, etc. The `ON CONFLICT DO NOTHING` pattern ensures only one insertion succeeds.

**Unclear:** What happens if:
- Two webhook requests arrive for the same `stripe_event_id` simultaneously?
- One reaches `INSERT` first and gets `processing`, the other does too (but second INSERT is a no-op)?
- Both then try to call the handler?

The story says "Check whether insert succeeded (webhook winner detection)" but doesn't specify how. The Supabase UPSERT pattern would be clearer:
```sql
INSERT INTO billing_webhook_event (...) VALUES (...)
ON CONFLICT (stripe_event_id) DO UPDATE SET ... (returning a flag about insertion vs update)
```

The current S1-03 schema doesn't support this directly; it relies on checking the return value of the INSERT statement.

### 8. Bridge Call Signature Timing
**Issue:** The bridge call signature includes a `X-Timestamp` header. If the webhook handler is delayed (queue backlog, slow database), the timestamp at signature time may differ from execution time.

**Risk:** The app's HMAC validation may reject the request if the delay exceeds the clock skew window (5 minutes).

**Unclear:** Should timestamps be generated immediately before signing, or earlier (at webhook receipt time)? The pattern suggests immediate generation is correct, but high-latency scenarios could cause issues.

### 9. Error Message Storage in `billing_webhook_event`
**Issue:** The `error_message` column can store details for debugging. The story doesn't specify:
- What level of detail (full stack trace vs. user-facing message)?
- Sensitive data handling (e.g., customer PII in error context)?
- Truncation (column has no length limit but TEXT can be large)?

**Unclear:** Whether to sanitize error messages before storage.

### 10. Pack Offer Detection at Webhook Time
**Issue:** The webhook handler receives a Stripe session and must determine which offer (e.g., `song_pack_500`) was purchased.

**Current pattern in checkout routes:** Build `STRIPE_PRICE_ID_SONG_PACK_500` → lookup from env. Reverse lookup is needed at webhook time.

**Simplest solution:** Store `offer_id` in session metadata at checkout creation time. But checkout routes don't currently do this.

**Workaround:** Maintain a reverse map at webhook handler time:
```ts
const PRICE_TO_OFFER: Record<string, string> = {
  [env.STRIPE_PRICE_ID_SONG_PACK_500]: 'song_pack_500',
  // ... etc
};
```

**Unclear:** Which approach is expected.
