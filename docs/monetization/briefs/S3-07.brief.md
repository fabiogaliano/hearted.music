# Implementation Brief: S3-07 Liked Songs Page — Locked/Pending Split

## Goal

Update `get_liked_songs_page` SQL RPC and `getLikedSongsPage` server function to distinguish `locked` from `pending` and suppress `song_analysis` content for non-entitled songs.

```sql

  p_account_id UUID,
  p_cursor TIMESTAMPTZ DEFAULT NULL,
  p_limit INTEGER DEFAULT 50,
  p_filter TEXT DEFAULT 'all'

RETURNS TABLE (

  liked_at TIMESTAMPTZ,
  matching_status TEXT,
  -- ... song columns ...
  analysis_id UUID,
  analysis_content JSONB,
  analysis_model TEXT,
  analysis_created_at TIMESTAMPTZ,
  -- ... audio features ...
)
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  RETURN QUERY
  SELECT
    -- ...
    CASE
      WHEN mr_agg.total_results > 0 AND mr_agg.undecided_count > 0 THEN 'has_suggestions'
      WHEN mr_agg.total_results > 0 AND mr_agg.undecided_count = 0 THEN 'acted'
      WHEN ist.id IS NOT NULL THEN 'no_suggestions'
      ELSE 'pending'         -- ← conflates locked + pending
    END AS matching_status,
    sa.analysis AS analysis_content,  -- ← no entitlement gate
    -- ...
  FROM liked_song ls
  -- ... joins ...
  LEFT JOIN item_status ist ON ...
  WHERE ls.account_id = p_account_id
    AND ls.unliked_at IS NULL
    AND (p_filter = 'pending' AND ist.id IS NULL)   -- ← matches locked songs too

$$;
```

**Problems:**
- `matching_status = 'pending'` when `ist.id IS NULL` — this includes non-entitled (locked) songs
- `analysis_content` returned for all songs including non-entitled ones
- No `display_state` column in return type

## Changes Required

### 1. New migration: `supabase/migrations/20260405210000_liked_songs_page_billing_aware.sql`

Drop and recreate `get_liked_songs_page` with:
- Add `display_state TEXT` to RETURNS TABLE
- Use a `billing_facts` CTE (compute `has_unlimited_access` once per query)
- Compute per-song `is_entitled` via CTE or inline expression using the same logic as `is_account_song_entitled`
- Derive `display_state` via CASE:
  ```sql
  CASE
    WHEN NOT (
      bf.has_unlimited_access
      OR EXISTS (
        SELECT 1 FROM account_song_unlock asu
        WHERE asu.account_id = p_account_id
          AND asu.song_id = s.id
          AND asu.revoked_at IS NULL
      )
    ) THEN 'locked'
    WHEN sa.id IS NOT NULL THEN 'analyzed'
    WHEN EXISTS (
      SELECT 1 FROM job_failure jf
      INNER JOIN job j ON j.id = jf.job_id
      WHERE jf.item_id = ls.song_id
        AND jf.item_type = 'song'
        AND jf.error_type IN ('validation', 'unsupported', 'auth', 'permanent')
        AND j.account_id = p_account_id
    ) THEN 'failed'
    ELSE 'pending'
  END AS display_state
  ```
- Suppress analysis for locked songs:
  ```sql
  CASE WHEN <is_entitled> THEN sa.id ELSE NULL END AS analysis_id,
  CASE WHEN <is_entitled> THEN sa.analysis ELSE NULL END AS analysis_content,
  CASE WHEN <is_entitled> THEN sa.model ELSE NULL END AS analysis_model,
  CASE WHEN <is_entitled> THEN sa.created_at ELSE NULL END AS analysis_created_at,
  ```
- Fix `matching_status`: for locked songs, return `NULL`; for entitled songs, preserve existing CASE logic
- `p_filter = 'pending'` should exclude locked songs: `AND ist.id IS NULL AND <is_entitled>` — but filter changes are **out of scope for S3-11**; update the pending filter to exclude locked: `(p_filter = 'pending' AND ist.id IS NULL AND <is_entitled>)`

Migration must use `LANGUAGE plpgsql SECURITY DEFINER SET search_path = public`.

Use `DROP FUNCTION IF EXISTS get_liked_songs_page(UUID, TIMESTAMPTZ, INTEGER, TEXT)` before recreating.

**Note on performance:** Use the inline `billing_facts` CTE (computing `has_unlimited_access` once) rather than calling `is_account_song_entitled()` per row to avoid N+1 function calls. Replicate the entitlement logic inline (as shown in `20260405190000_billing_aware_selectors.sql`).

## Constraints

From DECISIONS.md §4 (Read-Model Invariants):

- **`locked`** = Not entitled, regardless of shared cache state. Missing `item_status` for a non-entitled song means `locked`, not `pending`.
- **`pending`** = Entitled, queued for processing, not yet started.
- **`analyzing`** = Entitled, processing in progress.
- **`analyzed`** = Entitled, content visible.
- **`failed`** = Entitled, terminal processing failure.
- **Matching status is a sub-dimension of `analyzed` songs only.** Locked songs must not expose matching status.
- **"Read models must filter by effective entitlement, not by `song_analysis` or `item_status` existence."** The RPC must gate on unlock/unlimited, not on artifact presence.
- **"Shared artifacts do not imply entitlement."** `song_analysis` rows are global (not account-scoped); existence does not mean the account may see that analysis.

From DECISIONS.md §2:
- `past_due` and `unpaid` subscription statuses do NOT grant unlimited access for `subscription` source.
- `self_hosted` grants access without Stripe state.
- Single canonical predicate — use the same logic as `is_account_song_entitled` (no variants).

From DECISIONS.md §3:
- When `BILLING_ENABLED=false`, accounts get `unlimited_access_source = 'self_hosted'` — these must see all songs as entitled.

From TERMINOLOGY.md / DECISIONS.md §6:
- `SongDisplayState` frozen as: `locked | pending | analyzing | analyzed | failed`
- No `any`, no non-null assertions, no unsafe type casts.

## Acceptance Criteria

- [ ] **Locked songs returned with `locked` state, no analysis text** — verify: song with no unlock row and no unlimited access → `display_state = 'locked'`, `analysis_content = NULL`, `analysis_id = NULL`
- [ ] **Entitled songs without `item_status` returned as `pending`** — verify: song with active unlock row, no `song_analysis` → `display_state = 'pending'`
- [ ] **Analyzed entitled songs returned with analysis text and matching status** — verify: song with unlock + `song_analysis` → `display_state = 'analyzed'`, analysis content non-null, `matching_status` populated
- [ ] **Self-hosted/unlimited users see all songs as entitled** — verify: account with `unlimited_access_source = 'self_hosted'` → no songs return `locked`
- [ ] **No analysis text leaks for locked songs** — verify: even if `song_analysis` row exists globally, locked songs return NULL `analysis_content`

## Risks

1. **`database.types.ts` manual update required.** The file is generated by `bun run gen:types`, which requires a live Supabase instance. The implementation agent must manually add `display_state: string` to the return type. If `supabase db reset` runs successfully, using `bun run gen:types` is preferred over manual edit.

2. **`analyzing` state not distinguishable from `pending` at song level.** The `job` table is account-level (not song-level). There is no reliable way to determine if a specific song is currently being analyzed without scanning job progress JSONB. The `billing_aware_selectors.sql` precedent is to simply not return `analyzing` — all non-`song_analysis`, non-failed entitled songs are treated as `pending`. The `analyzing` state can be added in S3-11 or a follow-up if per-song job state tracking is added.

3. **`p_filter = 'pending'` behavior change.** Currently `pending` filter returns `ist.id IS NULL`, which includes locked songs. After this migration, `pending` filter should exclude locked songs. This is technically within S3-07 scope (fixing the filter semantics), but the story also says "Feature type migration (S3-11) consumes new states." The implementation agent should update the pending filter here to exclude locked songs, since allowing locked songs in the pending filter is a direct bug.

4. **`matching_status` for locked songs.** Currently `matching_status` can be `'pending'` for non-entitled songs. After this migration, locked songs should return `NULL` for `matching_status`. The CASE in the SQL needs to wrap the existing matching logic with an entitlement check.

5. **Timestamp for migration file.** Use `20260405210000` as the prefix (one step after the latest existing migration `20260405200000_credit_and_pack_rpcs.sql`).
