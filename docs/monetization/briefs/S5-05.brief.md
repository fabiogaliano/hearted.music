# Context Snapshot: S5-05 Free Allocation on Completion

## Goal

When a user completes onboarding and stays free, auto-unlock up to 15 most-recent liked songs via `insert_song_unlocks_without_charge` with `source='free_auto'`.

---

## Codebase Snapshot

### `src/lib/domains/billing/unlocks.ts`

**Current state (complete implementation):**

```typescript
export type GrantFreeAllocationResult = {
  unlockedIds: string[];
};

export async function grantFreeAllocation(
  supabase: AdminSupabaseClient,
  accountId: string,
): Promise<Result<GrantFreeAllocationResult, UnlockError>> {
  const { data: likedSongs, error: likedError } = await supabase
    .from("liked_song")
    .select("song_id")
    .eq("account_id", accountId)
    .is("unliked_at", null)
    .order("liked_at", { ascending: false })
    .limit(FREE_ALLOCATION_LIMIT); // 15

  // ... error handling ...

  const candidateIds = (likedSongs ?? []).map((row) => row.song_id);

  if (candidateIds.length === 0) {
    return Result.ok({ unlockedIds: [] });
  }

  const { data: unlockRows, error: unlockError } = await supabase.rpc(
    "insert_song_unlocks_without_charge",
    {
      p_account_id: accountId,
      p_song_ids: candidateIds,
      p_source: "free_auto",
    },
  );

  // ... error handling ...

  const unlockedIds = (unlockRows ?? []).map((row) => row.song_id);

  if (unlockedIds.length > 0) {
    try {
      await applyLibraryProcessingChange(
        BillingChanges.songsUnlocked(accountId, unlockedIds),
      );
    } catch (err) {
      console.error(
        "[unlocks] Failed to apply library processing change for free allocation:",
        err,
      );
    }
  }

  return Result.ok({ unlockedIds });
}
```

**Current behavior:**
- Queries `liked_song` table for up to 15 most-recent active (unliked_at IS NULL) songs per account, ordered by liked_at DESC
- Calls `insert_song_unlocks_without_charge` RPC with `source='free_auto'` — does not deduct from credit_balance
- Calls `applyLibraryProcessingChange(BillingChanges.songsUnlocked(...))` to emit control-plane change
- Returns Result with unlockedIds array (songs actually inserted, excluding already-active ones)
- Returns empty array if no liked songs exist
- Does not check billing state or throw errors for insufficient conditions

**What it does:**
- Pure unlock orchestrator; delegates RPC call to database and control-plane notification to library-processing service
- Handles Result-based error propagation (no exceptions thrown)
- Fire-and-forget on `applyLibraryProcessingChange` failure — logs error but returns success

## Constraints

From **DECISIONS.md §1 (Product Invariants)**:

| Constraint | Detail |
|---|---|
| Free allocation size | 15 songs, one-time |
| Selection policy | "Auto-select up to 15 most-recent liked songs at onboarding completion" |
| Unused allocation | Forfeited if fewer than 15 liked songs exist; no carryforward |
| Source | `source='free_auto'` — never deducts purchased balance |
| Durability | Unlocks persist forever; normal subscription changes never revoke `free_auto` unlocks |

From **DECISIONS.md §2 (Architecture Invariants)**:

| Constraint | Detail |
|---|---|
| Control-plane boundary | Billing-driven work enters via typed `LibraryProcessingChange.songs_unlocked` only |
| Error handling | Prefer errors as values (Result type); thrown exceptions reserved for truly exceptional cases |
| Fire-and-forget enrichment jobs | When `applyLibraryProcessingChange` fails, log error but do not block completion |

From **DECISIONS.md §3 (Data Model Invariants)**:

| Constraint | Detail |
|---|---|
| Billing state of truth | `account_billing` is single source; `plan`, `subscription_status`, `credit_balance`, `unlimited_access_source` determine eligibility |
| Every account has billing row | Missing billing row is invariant violation, not valid signal |

From **TERMINOLOGY.md §2 (Canonical Terms)**:

| Constraint | Detail |
|---|---|
| SQL plan column values | `'free' \| 'quarterly' \| 'yearly'` (frozen) |
| Subscription status enum | Stripe-derived, includes 'none', 'active', 'past_due', etc. (frozen) |

## Open Questions / Risks

### 1. How to Detect "User Stayed Free"?

**Issue:** No plan choice is persisted in user_preferences. PlanSelectionStep.handleFree() navigates to ready, but there's no state to indicate which plan was selected.

**Current knowledge:**
- If user selected pack or unlimited, checkout flow redirects them away (to Stripe)
- If they return from Stripe checkout, their billing state is updated (subscription/credits created)
- If they navigate to ready without checkout, they "stayed free"
- At completion time, free users should have: `plan='free'` AND `subscriptionStatus='none'` AND `creditBalance=0` (and optionally `unlimitedAccess.kind='none'`)

**Ambiguity:**
- Should we check ALL of these conditions, or just `plan='free'`?
- What if a user starts with pack balance, then cancels, then completes onboarding? Should they still get free allocation?
- Should pack-having users be excluded even if they never purchased? (They wouldn't be, they'd have creditBalance > 0)

**Impact:** Determines how to implement "free completion" detection in markOnboardingComplete or ReadyStep.

### 2. Where Should grantFreeAllocation Be Called?

**Issue:** Story says "Update `markOnboardingComplete` or its caller". Two locations are possible:

**Option A: Inside markOnboardingComplete (server function)**
- Pros: Transactional guarantee; allocation happens atomically with completion timestamp
- Cons: Requires fetching billing state inside server function; ReadyStep is unaware; harder to debug
- Pattern: Similar to how `savePlaylistTargets` calls `applyLibraryProcessingChange`

**Option B: In ReadyStep.handleStart (before calling markOnboardingComplete)**
- Pros: Clear in-component logic; can show loading state; easier to debug
- Cons: Two separate server calls; not transactional; ReadyStep becomes billing-aware
- Pattern: Similar to how ReadyStep calls other server functions

**Current codebase pattern:**
- Most changes are triggered inside server functions (`savePlaylistTargets` → `applyLibraryProcessingChange`)
- But ReadyStep is already the caller of markOnboardingComplete and owns error handling

**Impact:** Architectural decision on where allocation logic lives and how errors propagate.

### 3. Should grantFreeAllocation Be Exposed as a Server Function?

**Current state:**
- `grantFreeAllocation` exists in `src/lib/domains/billing/unlocks.ts`
- It is NOT exported from `src/lib/server/billing.functions.ts`
- Only `requestSongUnlock` is server-exposed

**Unclear:**
- If allocation is called from ReadyStep component, it needs to be a server function
- If allocation is called from inside markOnboardingComplete (which is already a server function), it can use the orchestrator directly

**Impact:** Determines whether a new `grantFreeAllocation` server function is needed.

### 4. Error Handling Strategy

**Ambiguity:**
- If `grantFreeAllocation` fails (RPC error, DB error), should it:
  - Block onboarding completion? (User can't progress)
  - Log error and succeed anyway? (Allocation is lost but onboarding completes)
  - Retry automatically?

**Current pattern in codebase:**
- `applyLibraryProcessingChange` failures are caught and logged, but don't block (fire-and-forget)
- But `grantFreeAllocation` is the primary unlock mechanism for free users, not a secondary notification

**Impact:** User experience on allocation failures.

### 5. Billing State Availability at ReadyStep

**Issue:** ReadyStep currently does NOT load billing state. If allocation is called from ReadyStep.handleStart, billing state must be:
- Loaded before the button is clicked, OR
- Loaded in handleStart, OR
- Fetched implicitly by a server function

**Current ReadyStep props:**
```typescript
interface ReadyStepProps {
  syncStats: { songs: number; playlists: number };
}
```

No billing state is passed.

**Impact:** Whether ReadyStep needs to be modified to load or accept billing state.

### 6. Timing of Free Allocation in User Journey

**Scenario A: Allocation before navigation**
```
Ready Step → grantFreeAllocation → markOnboardingComplete → navigate("/")
```
- User sees allocation happen (loading state)
- Songs are immediately available in app

**Scenario B: Allocation after navigation**
```
Ready Step → markOnboardingComplete → navigate("/") → background job allocates
```
- App loads faster; allocation is async
- Songs appear after initial page load

**Current pattern in codebase:**
- Most enrichment work is async/background; control-plane jobs are enqueued
- But free allocation is a pre-requisite for first-time user experience

**Impact:** User-visible timing and loading states.

### 7. Multiple Calls to ReadyStep

**Issue:** What if a user navigates back to ReadyStep (browser back button, manual URL edit)? Should free allocation happen twice?

**Current behavior:**
- markOnboardingComplete is idempotent (upsert sets timestamp, idempotent on re-run)
- grantFreeAllocation is NOT idempotent (insert_song_unlocks_without_charge may reject duplicates)

**Unclear:**
- Is re-navigating to ready after completion even possible? (Route loader might prevent it)
- If so, should grantFreeAllocation be idempotent (check if already unlocked)?

**Impact:** Idempotency guarantee and re-entrant behavior.

### 8. Testing Coverage for Free Allocation

**Current tests:**
- `src/features/onboarding/__tests__/onboarding-flow.test.tsx` does NOT test plan-selection or free allocation
- No tests for markOnboardingComplete server function

**Question:**
- Should tests mock `grantFreeAllocation`?
- Should tests verify BillingChanges.songsUnlocked is emitted?
- Should tests verify enrichment jobs are enqueued for unlocked songs?

**Impact:** Test design and confidence in implementation.

### 9. BILLING_ENABLED=false Path

**Issue:** When `BILLING_ENABLED=false`, plan-selection is skipped and users go directly to ready (no free/pack/unlimited choice, just ready → completion).

**Current state:**
- S5-04 brief says free allocation is handled by S5-05
- But with BILLING_ENABLED=false, there's no plan-selection step
- Should these users also get free allocation?

**Likely answer:** Yes, they should still get 15 songs. But unclear whether:
- The same code path handles both?
- There's special logic for self_hosted unlimited users?
- The check should be `plan='free'` regardless of BILLING_ENABLED?

**Impact:** Whether free allocation logic must handle self-hosted/provider-disabled cases.

### 10. Unused Allocation Forfeiture Confirmation

**Issue:** Story says "Forfeited if fewer than 15 liked songs exist; no carryforward". 

**Current implementation:** `grantFreeAllocation` just unlocks however many liked songs exist (up to 15). If only 10 exist, 10 are unlocked.

**Question:**
- Is this the desired behavior (silence forfeiture)?
- Or should we audit/log how many were forfeited?

**Impact:** Observability and metrics.

## Summary

**Completed:**
- `grantFreeAllocation` orchestrator is fully implemented in `src/lib/domains/billing/unlocks.ts`
- `BillingChanges.songsUnlocked` is implemented and will be called by grantFreeAllocation
- `insert_song_unlocks_without_charge` RPC exists in database
- PlanSelectionStep navigates correctly to ready on free choice

**Missing:**
- Integration point: grantFreeAllocation is not called anywhere during onboarding completion
- No server function wrapper: grantFreeAllocation is only available as a domain orchestrator, not a server function
- Detection logic: No mechanism to identify "user stayed free" (would need billing state check)
- Persistence: No plan choice is saved; relies on billing state at completion time to detect free path
- ReadyStep coordination: No awareness of whether to trigger allocation

**Key unknowns:**
1. Where exactly should grantFreeAllocation be called: in markOnboardingComplete or in ReadyStep?
2. How to determine "user stayed free": check `plan='free'` only, or validate full billing state?
3. Error handling: should allocation failures block completion or be fire-and-forget?
4. Server function exposure: is a new server function wrapper needed?
5. Timing: should allocation block navigation or happen asynchronously?
