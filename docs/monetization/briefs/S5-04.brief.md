### `src/features/onboarding/hooks/useOnboardingNavigation.ts`

**Function signature (lines 49–69):**
```typescript
export function useOnboardingNavigation() {
	const navigate = useNavigate({ from: "/onboarding" });

	const goToStep = useCallback(
		async (
			step: OnboardingStep,
			options?: {
				phaseJobIds?: PhaseJobIds | null;
				syncStats?: { songs: number; playlists: number };
			},
		) => {
			try {
				// Save step to DB for resumability
				await saveOnboardingStep({ data: { step } });

				// Update URL and pass state
				navigate({
					search: (prev) => ({
						...prev,
						step,
					}),
					state: (prev) => ({
						...prev,
						...(options?.phaseJobIds !== undefined && {
							phaseJobIds: options.phaseJobIds,
						}),
						...(options?.syncStats !== undefined && {
							syncStats: options.syncStats,
						}),
					}),
				});
			} catch (error) {
				console.error("Failed to save onboarding step:", error);
				toast.error("Something went wrong. Please try again.");
				// Don't navigate if save fails
			}
		},
		[navigate],
	);

	return { goToStep };
}
```

Used in PlanSelectionStep to navigate:
- Free path: `await goToStep("ready")`
- Pack/Unlimited paths: should navigate to next step (S5-06) after checkout initiation

## Constraints

### Frozen from TERMINOLOGY.md §2 (Onboarding Monetization)

- **Onboarding step IDs:** `song-showcase`, `match-showcase`, `plan-selection` (frozen)
- **Free allocation:** one-time, up to 15 songs, `source='free_auto'` (S5-05 responsibility)
- **Pack offer:** `song_pack_500` (frozen offer ID), 500 songs + 25 instant unlocks, $5.99
- **Unlimited offers:** `unlimited_quarterly` ($14.99/quarter), `unlimited_yearly` ($39.99/yr, includes priority queue)
- **Pack bonus unlocks:** up to 25 most-recent liked songs at pack purchase (handled by billing service, not app)
- **User-facing action verb:** "Unlock"
- **Pack entry points hidden:** when account has active unlimited access

### Frozen from TERMINOLOGY.md §6 (Env Flags)

- `BILLING_ENABLED`: bool, default false — master billing integration toggle
- `QUARTERLY_PLAN_ENABLED`: bool, default false — feature flag for quarterly plan visibility
- No migration needed for new step — stored as TEXT in DB

### Frozen from DECISIONS.md §1 (Product Invariants)

- **Auto-skip behavior:** "when `BILLING_ENABLED=false`, onboarding auto-skips the `plan-selection` step"
- **Queue bands:** free (no balance) → `low`; any positive balance → `standard`; quarterly → `standard`; yearly → `priority`; self-hosted → `priority`
- **Pack-to-unlimited conversion discount:** unused purchased pack value becomes one-time proportional discount on initial unlimited invoice (calculated by billing service, display TBD)

### Frozen from DECISIONS.md §2 (Architecture Invariants)

- **Internal offer IDs only:** app sends `song_pack_500`, `unlimited_quarterly`, `unlimited_yearly`; only billing service resolves Stripe price IDs
- **Checkout idempotency:** `checkoutAttemptId` is UUID, part of signed request body, forwarded as Stripe `idempotency_key`
- **Provider-disabled handling:** when `BILLING_ENABLED=false`, return typed error, not thrown exception

## Open Questions / Risks

### 1. Discount Display for Pack-to-Unlimited Conversion

**Issue:** Story says "show any first-invoice discount from unused purchased pack value" but does not specify:
- How is the discount amount calculated or retrieved?
- Is it provided by `getBillingState()`, or calculated client-side from `creditBalance`?
- What is the UI copy for the discount (e.g., "$X credit applied" or percentage savings)?
- Only shown for unlimited path, or both pack and unlimited?

**Impact:** Without discount calculation logic, the unlimited option may appear without the conversion benefit, potentially making the offer less attractive on upgrades.

### 2. Checkout Flow After Button Click

**Issue:** Story does not specify what happens after pack/unlimited button click:
- Does the function call `createCheckoutSession()`, receive URL, then redirect immediately?
- Does it store `checkoutAttemptId` somewhere for recovery after redirect?
- Does it close the component / show loading state before redirect?
- What happens if checkout URL fetch fails mid-flow?

**Impact:** User experience during redirect to Stripe; error recovery if checkout service is down.

### 3. Free Path Navigation

**Issue:** Story says "Free: continue with 15 songs → navigates to `ready`" but:
- Does clicking "Free" immediately navigate to ready?
- Or does it trigger a server function to allocate the free songs first?
- S5-05 handles free allocation, but that may happen post-onboarding completion — does the plan-selection step need to coordinate?

**Impact:** Unknown whether free allocation happens before or after navigating to ready. Affects loading state and error handling.

### 4. Step Navigation on Error

**Issue:** `useOnboardingNavigation` throws `toast.error()` if `saveOnboardingStep` fails, and does not navigate. But plan-selection may call `createCheckoutSession` before navigation.

**Unclear:**
- If checkout call fails, should the component show error and stay on plan-selection?
- Or should it attempt to navigate anyway and let the next step handle recovery?
- Should errors prevent step-save entirely, or save step then show error?

**Impact:** Error handling strategy and error message clarity for users.

### 5. "Quarterly" Visibility in UI

**Issue:** Story says "`QUARTERLY_PLAN_ENABLED=false` hides quarterly option" but:
- Is the quarterly option a separate button, or part of an unlimited selector?
- If separate, is the space reserved (hidden but laid out) or removed entirely?
- What is the layout when quarterly is hidden (e.g., two buttons instead of three)?

**Impact:** UI layout and spacing; component may need responsive behavior.

### 6. Checkout ID Generation

**Issue:** `createCheckoutSession` requires `checkoutAttemptId` (UUID) as input, but story does not specify:
- Who generates this UUID — the component or server function?
- Is it generated per button click, or reused on retry?
- Should it be stored in browser state for recovery after redirect?

**Impact:** Idempotency guarantee and retry semantics.

### 7. Call Order: Save Step vs. Call Checkout

**Issue:** Current pattern in `useOnboardingNavigation` is: save step → navigate → new route loads. But checkout is different:

**Unclear:**
- Should plan-selection call `saveOnboardingStep("plan-selection")` before calling `createCheckoutSession`?
- Or should it call `createCheckoutSession` without step save, wait for redirect, and let post-checkout flow (S5-06) handle step advancement?
- If step is not saved before redirect, and user abandons checkout, how do they resume onboarding?

**Impact:** Resumability and step progression integrity.

### 8. Disabled Pack Entry Point

**Issue:** Story says pack entry points hidden "when user already has unlimited" but:
- Should the entire pack button be disabled/hidden, or just visually disabled?
- What UI pattern indicates "already have unlimited, can't buy pack"?
- Does this require loading `getBillingState()` in the component?

**Impact:** Component complexity and state management; need to fetch billing state at render time.

### 9. Pack Bonus Unlock Copy

**Issue:** Story mentions "25 instant unlocks" but:
- Is this the user-facing copy, or internal term?
- TERMINOLOGY.md says user-facing label is "Instant Unlocks"; confirm this is the right copy for the UI button/card.

**Impact:** Copy consistency with brand/design system.

### 10. Onboarding Flow After Plan-Selection

**Issue:** S5-06 (checkout branches) is blocked by this story, but story does not specify:
- Does the component navigate to a waiting step, or does S5-06 define the next step?
- Is there a "processing checkout" or "awaiting payment" step between plan-selection and ready?
- Or does checkout completion redirect directly to app (not via onboarding)?

**Impact:** Next step after plan-selection is undefined; flow may hang or loop.

### 11. Data Dependencies for Component Render

**Issue:** The step may need:
- Current billing state (to hide pack button if unlimited active)
- Config flags (BILLING_ENABLED, QUARTERLY_PLAN_ENABLED)
- Pricing data (amounts, currency)

**Unclear:** Should these be loaded via `getOnboardingData()` and passed through context, or loaded by the component itself?

**Impact:** Whether PlanSelectionStep is a pure component or stateful data fetcher.

### 12. Theme Usage

**Issue:** No existing onboarding step shows pricing options. The step will need to display three distinct plan cards/buttons.

**Unclear:** Are there existing card/button components in the codebase, or should the component build its own styling?

**Impact:** Implementation complexity and design consistency.

## Summary

The `PlanSelectionStep` component does not exist yet; it needs to:

1. **Render three plan options** (free, pack, unlimited) with pricing and feature descriptions
2. **Hide quarterly option** when `QUARTERLY_PLAN_ENABLED=false`
3. **Handle free path:** click free button → `useOnboardingNavigation().goToStep("ready")`
4. **Handle pack/unlimited paths:** click button → `createCheckoutSession(offer, uuid)` → receive URL → redirect to Stripe
5. **Respect provider-disabled mode:** auto-skip is handled by route loader; component only renders when `BILLING_ENABLED=true`
6. **Optionally show conversion discount** when upgrading from pack to unlimited (calculation TBD)
7. **Hide pack buttons** when account already has unlimited access (requires fetching billing state)

All supporting infrastructure (step enum, config placeholder, auto-skip logic, checkout functions, env config, billing domain types) is already in place. The component remains the only unimplemented piece.

Key unknowns: discount calculation, post-checkout flow (S5-06), free allocation trigger (S5-05), and error handling strategy.
