### HMAC Signing Utility (Already Implemented)
**File:** `src/lib/domains/billing/hmac.ts`

Provides two functions for outbound request signing:

```typescript
export async function signBridgeRequest(
	body: string,
	secret: string,
	timestampOverride?: number,
): Promise<{ timestamp: string; signature: string }>
```

Signature components (frozen from S4-01):
- **Timestamp header:** `X-Timestamp` (Unix epoch seconds)
- **Signature header:** `X-Signature` (hex-encoded HMAC-SHA256)
- **Signature format:** `hex(HMAC-SHA256(secret, "{timestamp}.{hex(SHA256(body))}"))` 
- **Clock skew window:** 5 minutes (300 seconds)

The function returns `{ timestamp, signature }` ready to set as HTTP headers.

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Offer ID Validation
- **Frozen canonical offer IDs:** `song_pack_500`, `unlimited_quarterly`, `unlimited_yearly` (TERMINOLOGY.md §2)
- Must validate incoming `offer` against these constants before calling billing service
- Reject invalid offers with typed error

### HMAC Signing Requirements
- **All app → billing requests must be HMAC-signed** (DECISIONS.md §2)
- Signature format: `hex(HMAC-SHA256(shared_secret, "{timestamp}.{hex(SHA256(body))}"))`
- Headers: `X-Timestamp` (epoch seconds), `X-Signature` (hex string)
- Clock-skew window: 5 minutes (300 seconds) — prevent replay attacks
- Use existing `signBridgeRequest()` function from `src/lib/domains/billing/hmac.ts`

### Checkout Idempotency
- **`checkoutAttemptId` is part of signed request body** (DECISIONS.md §2)
- Generated by app per checkout intent; reused across retries of same intent
- Billing service forwards it as Stripe `idempotency_key` to prevent duplicate Stripe charges

### Provider-Disabled Handling
- **When `BILLING_ENABLED=false`, return typed error, not thrown exception** (story scope)
- Error type must be discriminated (not a boolean or exception)
- Prevents UI from attempting to navigate to checkout/portal

### Internal Offer IDs Only
- **App sends internal offer IDs, not Stripe price IDs** (DECISIONS.md §2)
- Only `v1_hearted_brand/` resolves offer IDs to Stripe price IDs
- App never sees or imports Stripe SDK or price identifiers

### Environment Variable Frozen Names
- `BILLING_ENABLED` (bool, default false)
- `BILLING_SERVICE_URL` (optional URL)
- `BILLING_SHARED_SECRET` (optional string, min 1 char)
(TERMINOLOGY.md §6)

### Type Safety
- No `any`, no non-null assertions (`!`), no unsafe type casts (CLAUDE.md)
- Make illegal states unrepresentable (discriminated unions over booleans)
- Validate inputs at boundaries; keep internal state typed

## Open Questions / Risks

### 1. Return Type Shape for Checkout/Portal URLs
**Issue:** Story specifies "Returns `{ checkoutUrl }` or typed error" but doesn't define the TypeScript type alias.

**Unclear:**
- Should return type be a discriminated union (success | error) like `RequestSongUnlockResponse`?
- Or should error be propagated via `Result<T, E>` and then mapped to response type?
- What are the exact error cases that should be typed (invalid offer, provider-disabled, network failure)?

**Impact:** Return type shape affects whether functions throw exceptions or return discriminated response types.

### 2. Request Body Validation Schema
**Issue:** No Zod schema shown for `createCheckoutSession` input.

**Unclear:**
- Should `offer` be validated with `z.enum()` against offer constants?
- Should `checkoutAttemptId` be `z.string().uuid()`?
- Should `createPortalSession` accept `account_id` from context or take it as input?

**Impact:** Error handling for malformed requests (caught by schema vs. caught by billing service).

### 3. Network Error Handling
**Issue:** Story doesn't specify how to handle network failures when calling billing service.

**Unclear:**
- If billing service is unreachable, should function throw or return a typed error?
- Retry logic? Timeout duration?
- What response format does the billing service return on error? (status code, error field, etc.)

**Impact:** Robustness of checkout flow during service degradation.

### 4. Authentication for Portal vs. Checkout
**Issue:** `createCheckoutSession` takes `checkoutAttemptId` as input, but `createPortalSession` scope doesn't mention it.

**Unclear:**
- Does portal session also use `checkoutAttemptId` or similar idempotency key?
- Or is portal session request body just `{ account_id }`?
- How does billing service identify which account is requesting portal access?

**Impact:** Portal function signature and idempotency guarantee.

### 5. Handling of Missing Env Vars When BILLING_ENABLED=true
**Issue:** `BILLING_SERVICE_URL` and `BILLING_SHARED_SECRET` are optional in `src/env.ts`, but are required when `BILLING_ENABLED=true`.

**Unclear:**
- Should server functions validate that these vars exist when `BILLING_ENABLED=true`?
- Or is that validation a deployment concern?
- If missing, throw error or return typed "configuration error"?

**Impact:** Error message clarity and failure mode when env is misconfigured.

### 6. Account ID Validation
**Issue:** Request body includes `account_id`, which comes from authenticated session context.

**Unclear:**
- Should `account_id` be passed from `context.session.accountId` automatically, or taken as input?
- If taken from context, how does the input validator validate it doesn't come from the request body?

**Impact:** Security of account_id field against spoofing.

### 7. Timing-Safe Comparison in Signing
**Issue:** The `signBridgeRequest` function uses standard SHA256, but S4-01 HMAC verification uses timing-safe comparison for the signature.

**Unclear:**
- Is timing-safe comparison used when signing outbound requests, or only on verification?
- Story says "signing utility" but doesn't specify internal implementation details.

**Risk:** Theoretical timing-attack vulnerability if outbound signing doesn't match verification rigor (low practical impact, but should be consistent).

### 8. Request Body Serialization
**Issue:** HMAC signing operates on raw request body string, but `signBridgeRequest` expects `body: string`.

**Unclear:**
- Should body be `JSON.stringify()` with sorted keys for deterministic hashing?
- What happens if request body structure changes (object key order)?
- Does billing service re-serialize and re-hash to verify, or does it have the raw body?

**Impact:** HMAC verification failure if serialization is non-deterministic.

### 9. Existing `RequestSongUnlock` Error Handling
**Issue:** `requestSongUnlock` in billing.functions.ts returns a discriminated union response type, not a `Result<T, E>`.

**Unclear:**
- Should `createCheckoutSession` and `createPortalSession` follow the same pattern (discriminated response)?
- Or return `Result<T, E>` internally and map to response before returning?
- Consistency with existing billing functions in the same file?

**Impact:** API style consistency and error handling pattern.

### 10. Billing Service Response Format on Error
**Issue:** Story says "Returns `{ checkoutUrl }` or typed error" but doesn't specify HTTP status code or response body format from billing service.

**Unclear:**
- Does billing service return 400/401/500 with `{ error: string }`, or different format?
- How should client code parse and map billing service errors to app error types?

**Risk:** Mismatch between billing service error responses and app error handling.

### 11. Session/Cookie Management for Checkout Redirect
**Issue:** `createCheckoutSession` returns a URL to redirect to Stripe, but story doesn't mention session state.

**Unclear:**
- Does Stripe checkout session require browser cookies or state storage?
- Should app store `checkoutAttemptId` in session for reconciliation after redirect?
- How does the app handle return from Stripe checkout (success/cancel)?

**Impact:** Post-checkout flow integration (beyond S4-10 scope, but affects function design).

### 12. Rate Limiting / Abuse Prevention
**Issue:** Story scope doesn't mention rate limiting on checkout/portal session creation.

**Unclear:**
- Should these functions rate-limit per account or IP?
- Billing service responsibility or app responsibility?
- Affects function design if app must track attempt counts.

**Impact:** Abuse vector if users spam checkout attempts.

Key unknowns:

2. Complete error type set (provider-disabled, invalid offer, network failure, etc.)
3. Network error handling and retry strategy
4. Request body validation schema details
5. Billing service error response format and mapping
