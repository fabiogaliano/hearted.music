# Implementation Brief: S3-11 — Feature Type Migration: SongDisplayState

## Goal

Replace `UIAnalysisStatus` with `SongDisplayState` across all feature modules and update matching status to be a sub-dimension of `analyzed` only.

---

## Constraints

From **DECISIONS.md §4 (Read-Model Invariants)**:
- `SongDisplayState` values are frozen: `locked | pending | analyzing | analyzed | failed` — no `not_analyzed`
- `not_analyzed` no longer exists — it's been split into `locked` and `pending`
- **"Matching status (`has_suggestions | acted | no_suggestions`) is a sub-dimension of `analyzed` songs only"** — locked songs must not expose matching status
- Read models must filter by effective entitlement, not by `song_analysis` or `item_status` existence

From **DECISIONS.md §6 (Terminology Invariants)**:
- Frozen TS types: `SongDisplayState` (no `UIAnalysisStatus` — it is not a frozen term, it is being replaced)
- `Song display state` frozen values: `locked | pending | analyzing | analyzed | failed`

## Open Questions / Risks

### 1. `simulateEnrichment` return type after removing `uiAnalysisStatus`

`simulateEnrichment` in `src/stories/fixtures/index.ts` spreads `song` then overrides `uiAnalysisStatus`. After removing `uiAnalysisStatus` from `LikedSong`, the spread still works, but the override becomes a property that no longer exists on the type. The function must also handle `displayState` to simulate the non-enriched state — currently it doesn't set `displayState` in the override object.

### 2. `fixtures.json` does not have `displayState` field

`src/stories/fixtures/fixtures.json` has 25 `uiAnalysisStatus` entries and zero `displayState` entries. The file is consumed as `allLikedSongs: LikedSong[]`. If `displayState` becomes a required field on `LikedSong` and `uiAnalysisStatus` is removed, the cast `data.likedSongs as LikedSong[]` would be unsound at runtime (field missing). Either `build-fixtures.ts` must be re-run after updating, or the file must be manually updated, or the cast must be replaced with runtime validation.

### 3. `build-fixtures.ts` adds `displayState` but currently outputs only `uiAnalysisStatus`

The script at line 64 only writes `uiAnalysisStatus`. If `uiAnalysisStatus` is removed from `LikedSong`, `build-fixtures.ts` would fail to typecheck (it writes a `LikedSong` shape but is missing `displayState`). The script must be updated to emit `displayState` instead. Note the script uses `any` for the `ls` variable in the mapping — TypeScript won't catch this without annotation.

### 4. `src/lib/data/landing-songs.ts` sets `uiAnalysisStatus: "analyzing"` for non-detail songs

The value `"analyzing"` is not a valid `SongDisplayState` value (frozen values are `locked | pending | analyzing | analyzed | failed` — wait, `analyzing` IS valid). But `"analyzing"` is a valid `SongDisplayState`. The `uiAnalysisStatus` field on the return object would need to be removed; the `displayState` field already correctly returns `"pending"` for non-detail songs, not `"analyzing"`.

### 5. `uiAnalysisStatus` is currently a required field on `LikedSong`

All consumers of the `LikedSong` type currently construct objects with `uiAnalysisStatus`. Removing the field from the type definition will cause TypeScript errors at all construction sites. The relevant construction sites are: `mapLikedSongPageRow` in `liked-songs.functions.ts`, `toLikedSong` in `landing-songs.ts`, `simulateEnrichment` return in `stories/fixtures/index.ts`, `build-fixtures.ts`, and the test fixture in `useSongExpansion.test.tsx`. All must be updated.

### 6. Matching status gate — no TypeScript enforcement exists yet

The constraint "matching status is a sub-dimension of `analyzed` songs only" is currently expressed only via the SQL RPC (which returns `NULL` `matching_status` for locked songs) and the `parseMatchingStatus` helper (which returns `null` for unexpected values). There is no TypeScript-level discriminated union enforcing that `matching_status` is only non-null when `displayState === "analyzed"`. The story says to update this, but no existing matching types reference `UIAnalysisStatus`, and the `LikedSong` interface would need structural changes (e.g., a discriminated union on `displayState`) to enforce this at the type level. The scope of this constraint enforcement is ambiguous — it may mean only removing `uiAnalysisStatus` and leaving the existing `matching_status: MatchingStatus | null` on `LikedSong` as-is (since SQL already gates it).

### 7. `fixtures.json` `uiAnalysisStatus` values are all `"analyzed"` — no `not_analyzed` entries exist

All 25 fixture songs have `"uiAnalysisStatus": "analyzed"`. None have `"not_analyzed"`. This means removing the field from JSON requires no value migration, but the file must still be regenerated or manually patched to add `displayState` and remove `uiAnalysisStatus`.
