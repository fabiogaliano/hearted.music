# Implementation Brief: S1-07 Subscription Lifecycle RPCs

## Goal

Implement three RPCs that manage the subscription lifecycle state on `account_billing`:

- `activate_subscription` — sets subscription fields and grants unlimited access
- `deactivate_subscription` — reverts to free plan, clears subscription-sourced unlimited access
- `update_subscription_state` — updates lifecycle fields only (status, period-end, cancel flag)

All three call `reprioritize_pending_jobs_for_account` as their final step.

```sql
CREATE OR REPLACE FUNCTION reprioritize_pending_jobs_for_account(
  p_account_id UUID
) RETURNS INTEGER
LANGUAGE sql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
  WITH resolved_band AS (
    SELECT 
      CASE
        WHEN ab.plan = 'yearly' 
             AND ab.unlimited_access_source = 'subscription' 
             AND ab.subscription_status = 'active' THEN 2
        WHEN ab.unlimited_access_source = 'self_hosted' THEN 2
        WHEN ab.plan = 'quarterly' 
             AND ab.unlimited_access_source = 'subscription' 
             AND ab.subscription_status = 'active' THEN 1
        WHEN ab.unlimited_access_source IS NULL 
             AND ab.credit_balance > 0 THEN 1
        ELSE 0
      END AS priority_value
    FROM account_billing ab
    WHERE ab.account_id = p_account_id
  ),
  updated_jobs AS (
    UPDATE job
    SET queue_priority = (SELECT priority_value FROM resolved_band)
    WHERE account_id = p_account_id
      AND type IN ('enrichment', 'match_snapshot_refresh')
      AND status = 'pending'
    RETURNING id
  )
  SELECT COUNT(*)::INTEGER FROM updated_jobs;
$$;
```

### Function 1: `activate_subscription`

Sets all subscription fields including `unlimited_access_source = 'subscription'`, then reprioritizes.

```sql
CREATE OR REPLACE FUNCTION activate_subscription(
  p_account_id             UUID,
  p_plan                   TEXT,
  p_stripe_subscription_id TEXT,
  p_stripe_customer_id     TEXT,
  p_subscription_period_end TIMESTAMPTZ
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  UPDATE account_billing
  SET
    plan                    = p_plan,
    stripe_subscription_id  = p_stripe_subscription_id,
    stripe_customer_id      = p_stripe_customer_id,
    subscription_status     = 'active',
    subscription_period_end = p_subscription_period_end,
    unlimited_access_source = 'subscription',
    cancel_at_period_end    = FALSE
  WHERE account_id = p_account_id;

  PERFORM reprioritize_pending_jobs_for_account(p_account_id);
END;
$$;
```

**Notes:**
- `cancel_at_period_end` reset to `FALSE` on activation (correct for new/renewed subscriptions).
- Using `LANGUAGE plpgsql` to call `PERFORM` on the void-equivalent reprioritize RPC within a multi-statement body. `LANGUAGE sql` cannot use `PERFORM`; it would require `SELECT reprioritize_pending_jobs_for_account(p_account_id)` as the last statement, which works too — but plpgsql is cleaner and matches complex RPCs already in the repo (e.g., `unlock_songs_for_account`).
- Idempotent: calling with the same state twice performs a no-op update and a no-op reprioritize.

### Function 2: `deactivate_subscription`

Reverts plan to `'free'`, clears `unlimited_access_source` only when its current value is `'subscription'`. Does not restore converted pack value.

```sql
CREATE OR REPLACE FUNCTION deactivate_subscription(
  p_account_id UUID
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  UPDATE account_billing
  SET
    plan                    = 'free',
    unlimited_access_source = CASE
                                WHEN unlimited_access_source = 'subscription' THEN NULL
                                ELSE unlimited_access_source
                              END
  WHERE account_id = p_account_id;

  PERFORM reprioritize_pending_jobs_for_account(p_account_id);
END;
$$;
```

**Notes:**
- The `CASE` expression preserves `self_hosted` (and any future values) while clearing only `subscription`-sourced unlimited access. This is the critical invariant from the story: `deactivate_subscription` must NOT clear `unlimited_access_source` when source is `self_hosted`.
- `plan` is not validated against enum values here — the `CHECK` constraint on `account_billing.plan` will reject invalid values if a bug is introduced.
- `stripe_subscription_id`, `stripe_customer_id`, `subscription_status`, `subscription_period_end`, `cancel_at_period_end` are intentionally left untouched. These fields preserve the Stripe audit trail. The lifecycle status is updated separately via `update_subscription_state`.
- Idempotent: calling twice on a `free` account with NULL `unlimited_access_source` is a no-op.

### Function 3: `update_subscription_state`

Updates lifecycle-only fields. Does not change `plan` or `unlimited_access_source`.

```sql
CREATE OR REPLACE FUNCTION update_subscription_state(
  p_account_id              UUID,
  p_subscription_status     TEXT,
  p_subscription_period_end TIMESTAMPTZ,
  p_cancel_at_period_end    BOOLEAN
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  UPDATE account_billing
  SET
    subscription_status     = p_subscription_status,
    subscription_period_end = p_subscription_period_end,
    cancel_at_period_end    = p_cancel_at_period_end
  WHERE account_id = p_account_id;

  PERFORM reprioritize_pending_jobs_for_account(p_account_id);
END;
$$;
```

**Notes:**
- `p_subscription_status` is validated by the `CHECK` constraint on `account_billing.subscription_status`. Invalid values will raise a constraint violation before the reprioritize call.
- Intentionally does NOT touch `plan`, `unlimited_access_source`, `stripe_subscription_id`, `stripe_customer_id`, `credit_balance`, or any other fields.
- Used by: `invoice.payment_failed` (→ `past_due`), `customer.subscription.updated` (cancel flag changes), renewal period-end refresh (→ new `subscription_period_end`).
- Idempotent: calling with identical values is a no-op update.

## Risks and Ambiguities

### 1. `activate_subscription` always resets `cancel_at_period_end = FALSE`

The function resets `cancel_at_period_end` to `FALSE` unconditionally. This is correct for new subscription creation and renewal events. However, if the billing service ever calls `activate_subscription` on a subscription that was previously marked for cancellation (but then the user reconsidered), the flag reset is the desired outcome. If a caller needs to preserve the cancel flag, it should call `update_subscription_state` instead.

### 2. `deactivate_subscription` leaves Stripe audit fields intact

`stripe_subscription_id`, `stripe_customer_id`, `subscription_status`, `subscription_period_end`, and `cancel_at_period_end` are not cleared by `deactivate_subscription`. The story does not specify clearing these fields, and they serve as audit trail. The billing service typically calls `update_subscription_state` (with `subscription_status='canceled'`) before or after `deactivate_subscription` in its webhook handlers, providing a complete picture.

If clearing Stripe refs on deactivation is desired, it would need an explicit story requirement — it is not called for here.

### 3. `update_subscription_state` accepts any `subscription_status` string

Validation is delegated to the `CHECK` constraint on `account_billing.subscription_status`. An invalid status value (e.g. a new Stripe status added in the future) will raise a PostgreSQL constraint violation. This is the correct pattern: the SQL layer enforces the frozen value set, and the billing service layer must be updated if Stripe adds new statuses.

### 4. No explicit account existence check

None of the three RPCs validate that the `account_id` exists in `account` or has a corresponding `account_billing` row. If the account is missing, `UPDATE ... WHERE account_id = p_account_id` will silently match 0 rows and return without error. Per DECISIONS.md §3, every account must have an `account_billing` row — a missing row is an invariant violation, not a valid mode. Callers should ensure the account is provisioned before invoking these RPCs.

### 5. Concurrent activation

`activate_subscription` does not lock `account_billing` with `SELECT ... FOR UPDATE`. For subscription lifecycle events this is acceptable: Stripe delivers webhook events sequentially per subscription, and the billing service is expected to process them serially. Unlike balance deductions, concurrent activations for the same account are not a valid scenario in normal operation.

### 6. Language choice: `plpgsql` vs `sql`

Pure `LANGUAGE sql` functions cannot use `PERFORM`; they would need `SELECT reprioritize_pending_jobs_for_account(p_account_id)` as the final statement in a CTE or subquery. `LANGUAGE plpgsql` with `PERFORM` is cleaner and consistent with `unlock_songs_for_account` in `20260405160000_core_unlock_rpcs.sql`. Either works — `plpgsql` is chosen here for readability.
