### Function 1: `grant_credits`

Adds operational/replacement credits to `credit_balance`. Never creates `pack_credit_lot` rows.

**Parameters:**
- `p_account_id UUID`
- `p_amount INTEGER` — positive, number of credits to add
- `p_reason TEXT` — `'replacement_grant'` or `'admin_adjustment'`
- `p_stripe_event_id TEXT DEFAULT NULL` — optional audit reference

**Returns:** `INTEGER` (new balance)

```sql
CREATE OR REPLACE FUNCTION grant_credits(
  p_account_id      UUID,
  p_amount          INTEGER,
  p_reason          TEXT,
  p_stripe_event_id TEXT DEFAULT NULL
) RETURNS INTEGER
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_balance INTEGER;
BEGIN
  -- Validate reason: only operational grant reasons are allowed
  -- Must NOT be 'pack_purchase' — that would create upgrade-convertible pack value
  IF p_reason NOT IN ('replacement_grant', 'admin_adjustment') THEN
    RAISE EXCEPTION 'grant_credits: invalid reason ''%'' (must be replacement_grant or admin_adjustment)',
      p_reason;
  END IF;

  IF p_amount <= 0 THEN
    RAISE EXCEPTION 'grant_credits: amount must be positive, got %', p_amount;
  END IF;

  -- Lock account_billing before reading balance (prevents concurrent mutations)
  SELECT ab.credit_balance INTO v_balance
  FROM account_billing ab
  WHERE ab.account_id = p_account_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'grant_credits: account_billing not found for account %', p_account_id;
  END IF;

  -- Add credits to balance
  UPDATE account_billing
  SET credit_balance = credit_balance + p_amount
  WHERE account_id = p_account_id;

  -- Write immutable ledger row
  INSERT INTO credit_transaction (account_id, amount, balance_after, reason, stripe_event_id)
  VALUES (p_account_id, p_amount, v_balance + p_amount, p_reason, p_stripe_event_id);

  RETURN v_balance + p_amount;
END;
$$;
```

**Key invariants:**
- No `pack_credit_lot` row created → granted credits are operational balance, not eligible for upgrade conversion
- `SELECT ... FOR UPDATE` on `account_billing` before balance read
- Reason is restricted to prevent accidental misuse as pack fulfillment path

### Function 2: `fulfill_pack_purchase`

Idempotent pack fulfillment. Creates a `pack_credit_lot` row, increments balance, grants up to 25 bonus unlocks, reprioritizes.

**Parameters:**
- `p_account_id UUID`
- `p_stripe_event_id TEXT` — idempotency key (maps to `pack_credit_lot.stripe_event_id UNIQUE`)
- `p_offer_id TEXT` — e.g., `'song_pack_500'`
- `p_credits INTEGER` — credits to grant (e.g., `500`)
- `p_price_cents INTEGER` — purchase price in cents (e.g., `599`)

**Returns:** `JSONB` — `{ "new_balance": INTEGER, "bonus_unlocked_song_ids": UUID[] }`

```sql
CREATE OR REPLACE FUNCTION fulfill_pack_purchase(
  p_account_id      UUID,
  p_stripe_event_id TEXT,
  p_offer_id        TEXT,
  p_credits         INTEGER,
  p_price_cents     INTEGER
) RETURNS JSONB
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_balance          INTEGER;
  v_bonus_candidates UUID[];
  v_bonus_ids        UUID[];
BEGIN
  -- 1. Attempt idempotent lot insert
  --    ON CONFLICT DO NOTHING: if stripe_event_id already exists, insert is suppressed.
  --    NOT FOUND branch handles the duplicate-fulfillment no-op path.
  INSERT INTO pack_credit_lot (
    account_id, stripe_event_id, offer_id, original_credits, remaining_credits, price_cents
  ) VALUES (
    p_account_id, p_stripe_event_id, p_offer_id, p_credits, p_credits, p_price_cents
  ) ON CONFLICT (stripe_event_id) DO NOTHING;

  IF NOT FOUND THEN
    -- Already fulfilled: return current balance and bonus unlocks for this event
    SELECT ab.credit_balance INTO v_balance
    FROM account_billing ab
    WHERE ab.account_id = p_account_id;

    SELECT ARRAY(
      SELECT asu.song_id
      FROM account_song_unlock asu
      WHERE asu.account_id = p_account_id
        AND asu.granted_stripe_event_id = p_stripe_event_id
        AND asu.source = 'pack'
    ) INTO v_bonus_ids;

    RETURN jsonb_build_object(
      'new_balance',             v_balance,
      'bonus_unlocked_song_ids', to_jsonb(v_bonus_ids)
    );
  END IF;

  -- 2. Lock account_billing before balance mutation
  SELECT ab.credit_balance INTO v_balance
  FROM account_billing ab
  WHERE ab.account_id = p_account_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'fulfill_pack_purchase: account_billing not found for account %', p_account_id;
  END IF;

  -- 3. Increment credit_balance by purchased credits
  UPDATE account_billing
  SET credit_balance = credit_balance + p_credits
  WHERE account_id = p_account_id;

  -- 4. Write immutable ledger row
  INSERT INTO credit_transaction (account_id, amount, balance_after, reason, stripe_event_id)
  VALUES (p_account_id, p_credits, v_balance + p_credits, 'pack_purchase', p_stripe_event_id);

  -- 5. Select up to 25 bonus candidates: most-recent currently liked songs not already unlocked
  SELECT ARRAY(
    SELECT ls.song_id
    FROM liked_song ls
    WHERE ls.account_id = p_account_id
      AND ls.unliked_at IS NULL
      AND NOT EXISTS (
        SELECT 1 FROM account_song_unlock asu
        WHERE asu.account_id = p_account_id
          AND asu.song_id = ls.song_id
          AND asu.revoked_at IS NULL
      )
    ORDER BY ls.liked_at DESC
    LIMIT 25
  ) INTO v_bonus_candidates;

  -- 6. Grant bonus unlocks (source='pack', no balance deduction, tagged with pack event)
  IF cardinality(v_bonus_candidates) > 0 THEN
    SELECT ARRAY(
      SELECT song_id
      FROM insert_song_unlocks_without_charge(
        p_account_id,
        v_bonus_candidates,
        'pack',
        p_stripe_event_id
      )
    ) INTO v_bonus_ids;
  ELSE
    v_bonus_ids := '{}';
  END IF;

  -- 7. Reprioritize pending jobs (balance increased → may move low→standard)
  PERFORM reprioritize_pending_jobs_for_account(p_account_id);

  RETURN jsonb_build_object(
    'new_balance',             v_balance + p_credits,
    'bonus_unlocked_song_ids', to_jsonb(v_bonus_ids)
  );
END;
$$;
```

**Key invariants:**
- `INSERT ... ON CONFLICT DO NOTHING` on `pack_credit_lot` is the idempotency gate; second call with same `stripe_event_id` returns early without touching balance or unlock rows
- `SELECT ... FOR UPDATE` on `account_billing` happens AFTER the lot INSERT (lot is a new row; no deadlock risk with concurrent `unlock_songs_for_account` which locks `account_billing` first then lots)
- Bonus unlocks use `insert_song_unlocks_without_charge` with `source='pack'` and `p_granted_stripe_event_id = p_stripe_event_id` to tag them for later reversal
- Bonus selection caps at 25 and excludes songs with any active unlock (not just `source='pack'`)

### Function 3: `reverse_pack_entitlement`

Reverses a pack purchase: zeroes the lot, subtracts unspent credits from balance, revokes bonus unlocks and any pack unlocks for credits already spent, writes ledger row, reprioritizes.

**Parameters:**
- `p_account_id UUID`
- `p_pack_stripe_event_id TEXT` — identifies the `pack_credit_lot` to reverse
- `p_stripe_event_id TEXT` — the refund/chargeback Stripe event ID (audit trail)
- `p_reason TEXT` — `'refund'` or `'chargeback'`

**Returns:** `JSONB` — `{ "credits_reversed": INTEGER, "revoked_song_ids": UUID[] }`

**Reason mapping:**

| `p_reason` | `credit_transaction.reason` | `revoked_reason` on unlock |
|---|---|---|
| `'refund'` | `'refund'` | `'refund'` |
| `'chargeback'` | `'chargeback_reversal'` | `'chargeback'` |

```sql
CREATE OR REPLACE FUNCTION reverse_pack_entitlement(
  p_account_id           UUID,
  p_pack_stripe_event_id TEXT,
  p_stripe_event_id      TEXT,
  p_reason               TEXT
) RETURNS JSONB
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_lot            RECORD;
  v_balance        INTEGER;
  v_credits_to_sub INTEGER;
  v_credits_spent  INTEGER;
  v_txn_reason     TEXT;
  v_revoke_reason  TEXT;
  v_bonus_revoked  UUID[];
  v_extra_revoked  UUID[];
BEGIN
  -- 1. Validate reason and derive downstream reason codes
  IF p_reason = 'refund' THEN
    v_txn_reason    := 'refund';
    v_revoke_reason := 'refund';
  ELSIF p_reason = 'chargeback' THEN
    v_txn_reason    := 'chargeback_reversal';
    v_revoke_reason := 'chargeback';
  ELSE
    RAISE EXCEPTION 'reverse_pack_entitlement: invalid reason ''%'' (must be refund or chargeback)',
      p_reason;
  END IF;

  -- 2. Lock account_billing before balance mutation
  SELECT ab.credit_balance INTO v_balance
  FROM account_billing ab
  WHERE ab.account_id = p_account_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'reverse_pack_entitlement: account_billing not found for account %',
      p_account_id;
  END IF;

  -- 3. Lock the pack_credit_lot (consistent lock order: account_billing → lot)
  SELECT pcl.id, pcl.original_credits, pcl.remaining_credits
  INTO v_lot
  FROM pack_credit_lot pcl
  WHERE pcl.account_id = p_account_id
    AND pcl.stripe_event_id = p_pack_stripe_event_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'reverse_pack_entitlement: pack_credit_lot not found for event %',
      p_pack_stripe_event_id;
  END IF;

  -- 4. Compute reversal amounts
  --    credits_to_sub: unspent lot value still in credit_balance → subtract this
  --    credits_spent:  lot value already spent on unlock rows → revoke those unlocks
  v_credits_to_sub := v_lot.remaining_credits;
  v_credits_spent  := v_lot.original_credits - v_lot.remaining_credits;

  -- 5. Zero out the lot (marks it as fully reversed)
  UPDATE pack_credit_lot
  SET remaining_credits = 0
  WHERE id = v_lot.id;

  -- 6. Subtract unspent credits from balance
  --    Balance invariant: credit_balance >= sum(remaining_credits), so this never goes negative
  UPDATE account_billing
  SET credit_balance = credit_balance - v_credits_to_sub
  WHERE account_id = p_account_id;

  -- 7. Write immutable ledger row (amount = 0 when all credits were already spent; written anyway for audit)
  INSERT INTO credit_transaction (
    account_id, amount, balance_after, reason, stripe_event_id, metadata
  ) VALUES (
    p_account_id,
    -v_credits_to_sub,
    v_balance - v_credits_to_sub,
    v_txn_reason,
    p_stripe_event_id,
    jsonb_build_object('pack_stripe_event_id', p_pack_stripe_event_id)
  );

  -- 8. Revoke bonus unlocks tagged with this pack's stripe_event_id (idempotent: WHERE revoked_at IS NULL)
  WITH bonus_revoke AS (
    UPDATE account_song_unlock
    SET revoked_at              = now(),
        revoked_reason          = v_revoke_reason,
        revoked_stripe_event_id = p_stripe_event_id
    WHERE account_id = p_account_id
      AND granted_stripe_event_id = p_pack_stripe_event_id
      AND source = 'pack'
      AND revoked_at IS NULL
    RETURNING song_id
  )
  SELECT ARRAY(SELECT song_id FROM bonus_revoke) INTO v_bonus_revoked;

  -- 9. Revoke credits_spent newest active source='pack' unlocks (for balance already spent)
  --    Newest-first mirrors FIFO lot consumption in reverse.
  --    Only revokes 'pack' source; never touches 'free_auto', 'unlimited', 'self_hosted', or 'admin'.
  --    Bonus unlocks from step 8 are already revoked (revoked_at IS NOT NULL), so excluded.
  IF v_credits_spent > 0 THEN
    WITH extra_candidates AS (
      SELECT id
      FROM account_song_unlock
      WHERE account_id = p_account_id
        AND source = 'pack'
        AND revoked_at IS NULL
      ORDER BY created_at DESC
      LIMIT v_credits_spent
    ),
    extra_revoke AS (
      UPDATE account_song_unlock
      SET revoked_at              = now(),
          revoked_reason          = v_revoke_reason,
          revoked_stripe_event_id = p_stripe_event_id
      WHERE id IN (SELECT id FROM extra_candidates)
      RETURNING song_id
    )
    SELECT ARRAY(SELECT song_id FROM extra_revoke) INTO v_extra_revoked;
  ELSE
    v_extra_revoked := '{}';
  END IF;

  -- 10. Reprioritize pending jobs (balance decreased → may move standard→low)
  PERFORM reprioritize_pending_jobs_for_account(p_account_id);

  RETURN jsonb_build_object(
    'credits_reversed', v_credits_to_sub,
    'revoked_song_ids', to_jsonb(v_bonus_revoked || v_extra_revoked)
  );
END;
$$;
```

## Acceptance Criteria

- [ ] `grant_credits` increases `credit_balance` without creating a `pack_credit_lot` row
- [ ] `grant_credits` rejects invalid reasons (anything other than `replacement_grant`/`admin_adjustment`)
- [ ] `grant_credits` writes a `credit_transaction` row with correct `amount`, `balance_after`, `reason`
- [ ] `fulfill_pack_purchase` is idempotent — second call with same `stripe_event_id` returns current state without mutations
- [ ] Pack fulfillment creates exactly one `pack_credit_lot` row with `original_credits = remaining_credits = p_credits`
- [ ] Bonus unlocks are capped at 25 and only target songs with active likes and no active unlock
- [ ] Bonus unlocks have `source='pack'` and `granted_stripe_event_id = p_stripe_event_id`
- [ ] `fulfill_pack_purchase` writes `credit_transaction` with `reason='pack_purchase'`
- [ ] `reverse_pack_entitlement` subtracts `lot.remaining_credits` from `credit_balance` and zeroes the lot
- [ ] `reverse_pack_entitlement` revokes bonus unlocks tagged with `p_pack_stripe_event_id`
- [ ] `reverse_pack_entitlement` revokes `credits_spent` newest active `source='pack'` unlocks (where `credits_spent = original_credits - remaining_credits`)
- [ ] Full pack refund (all 500 credits spent): revokes 500 + 25 = 525 unlock rows
- [ ] `reverse_pack_entitlement` never revokes `free_auto`, `unlimited`, `self_hosted`, or `admin` unlocks
- [ ] `reverse_pack_entitlement` writes `credit_transaction` with correct `reason` based on `p_reason`
- [ ] `reprioritize_pending_jobs_for_account` called as final step in `fulfill_pack_purchase` and `reverse_pack_entitlement`
- [ ] All three RPCs are `SECURITY DEFINER` with `SET search_path = public`
- [ ] `supabase db reset` completes cleanly

## Risks and Ambiguities

### 1. `grant_credits` does not call `reprioritize_pending_jobs_for_account`

The story explicitly lists `reprioritize_pending_jobs_for_account` as a final step only for `fulfill_pack_purchase` and `reverse_pack_entitlement`. However, `grant_credits` increases `credit_balance`, which can move an account from the `low` queue band to `standard` (per DECISIONS.md §1: "any non-unlimited account with positive purchased balance → standard").

**Decision:** Follow the story spec and omit reprioritize from `grant_credits`. If the billing service or admin tooling calls `grant_credits`, it is responsible for calling `reprioritize_pending_jobs_for_account` separately if needed. This is a safe omission since `replacement_grant` and `admin_adjustment` are rare operational paths, not high-throughput billing events.

### 2. `reverse_pack_entitlement` is not intrinsically idempotent

Unlike `fulfill_pack_purchase` (which uses `pack_credit_lot.stripe_event_id UNIQUE` as a gate), `reverse_pack_entitlement` has no natural idempotency guard. A second call with the same `p_pack_stripe_event_id` after the lot is already zeroed would:
- Subtract 0 from `credit_balance` (safe, no-op)
- Write a `credit_transaction` with `amount=0` (harmless but spurious)
- Revoke 0 bonus unlocks (idempotent via `WHERE revoked_at IS NULL`)
- But `v_credits_spent = original_credits - 0 = original_credits` → would try to revoke `original_credits` MORE active pack unlocks

**Decision:** The billing service handles idempotency at the `billing_webhook_event` level before calling this RPC, per DECISIONS.md §2 (Bridge pattern). The second-call risk is real but is guarded by the external idempotency layer. Note this explicitly in code comments. A future hardening story could add a `reversed_at` column to `pack_credit_lot` as an additional in-RPC guard.

### 3. Bonus unlock candidate selection races with concurrent unlocks

Between the `INSERT ... ON CONFLICT DO NOTHING` (lot insert) and the bonus candidate `SELECT` (step 5), another `unlock_songs_for_account` call could unlock some of the bonus candidates. Those songs would then have `revoked_at IS NULL` and would be excluded from the bonus selection. This is correct behavior — the cap naturally adjusts. However, if `unlock_songs_for_account` runs concurrently and acquires the `account_billing` lock before step 2 of `fulfill_pack_purchase`, both could proceed (the unlock sees 0 balance from the old lot → insufficient balance exception). This is safe.

More precisely: the lot INSERT happens before the `account_billing` lock in `fulfill_pack_purchase`. This is intentional — the lot is a new row with no existing lock. No deadlock is possible because:
- `unlock_songs_for_account` locks `account_billing` first, then loops over existing lots
- `fulfill_pack_purchase` inserts a new lot (no lock), then locks `account_billing`

These lock sequences do not deadlock.

### 4. Bonus unlocks use `insert_song_unlocks_without_charge` which can re-activate revoked rows

`insert_song_unlocks_without_charge` re-activates revoked rows (sets `revoked_at = NULL`). The bonus candidate selection filters `WHERE revoked_at IS NULL` (active unlocks) — so revoked songs ARE eligible for bonus unlock re-grant. This matches the story intent: "non-unlocked currently liked songs". A revoked song is effectively not unlocked, so it should be eligible.

### 5. Multi-lot accounts and `credits_spent` revocation attribution

`v_credits_spent = lot.original_credits - lot.remaining_credits` counts credits that were spent from this lot via FIFO consumption. However, when we revoke `credits_spent` newest active pack unlocks in step 9, we cannot distinguish which specific songs were unlocked from THIS lot versus other lots.

The "newest active pack unlocks" heuristic is the closest approximation available without per-lot unlock attribution. In the common single-pack case, it's exact. In the multi-pack case, it's a reasonable proxy: FIFO consumption means the oldest lot's credits were spent first, so the songs unlocked from a specific lot tend to be among the newer unlocks.

This is the approach sanctioned by the story: "revokes newest active `source='pack'` unlocks".

### 6. `fulfill_pack_purchase` idempotent return: bonus IDs from DB vs candidates

In the idempotent (second call) path, bonus IDs are fetched from `account_song_unlock WHERE granted_stripe_event_id = p_stripe_event_id AND source = 'pack'`. This returns all pack unlocks tagged with this event, whether they are currently active (`revoked_at IS NULL`) or have since been revoked. The caller should be aware that the returned IDs may include subsequently-revoked songs. This is acceptable for the idempotent return: it represents what was fulfilled, not what is currently active.

### 7. `credits_to_sub = 0` ledger row

When all pack credits have been spent before the refund, `v_credits_to_sub = 0` and a `credit_transaction` with `amount = 0` is written. This is an unusual ledger row but is written for audit completeness. The `credit_balance >= 0` CHECK constraint will not be violated (subtracting 0 is safe). The `balance_after` equals the current balance (unchanged).
