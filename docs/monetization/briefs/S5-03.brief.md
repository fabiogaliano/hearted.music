### Playlist Profiling Trigger (Existing Infrastructure)

**File: `src/lib/server/onboarding.functions.ts` (lines 295–326)**

The `savePlaylistTargets` function already triggers profiling:
```typescript
export const savePlaylistTargets = createServerFn({ method: "POST" })
	.middleware([authMiddleware])
	.inputValidator(playlistIdsInputSchema)
	.handler(async ({ data, context }): Promise<{ success: true }> => {
		const { session } = context;
		const playlistsResult = await getPlaylists(session.accountId);
		// ... validation ...
		const updates = playlistsResult.value.map((playlist) => {
			const shouldBeTarget = data.playlistIds.includes(playlist.id);
			return setPlaylistTarget(playlist.id, shouldBeTarget);
		});
		// ... persist updates ...
		await applyLibraryProcessingChange(
			OnboardingChanges.targetSelectionConfirmed(session.accountId),
		);
		return { success: true };
	});
```

**Profiling flow**:
1. `OnboardingChanges.targetSelectionConfirmed()` is dispatched
2. This change is processed by `applyLibraryProcessingChange()` in library-processing service
3. The reconciler creates an `ensure_match_snapshot_refresh_job` effect
4. This job runs `executeMatchSnapshotRefresh()` which includes the `playlist_profiling` stage
5. Stage loads target playlists and computes/caches profiles via `loadTargetPlaylistProfiles()`

**Status**: Profiling trigger **already exists and works**. S5-03 does not need to add it — it just needs to coordinate the UI to let profiling complete before displaying results.

### Match Snapshot Refresh Orchestrator

**File: `src/lib/workflows/match-snapshot-refresh/orchestrator.ts` (lines 180–230)**

```typescript
await maybeDevDelay(stageDelayMs);

	accountId,
	profilingService,
);

```

After profiling completes, the orchestrator runs:
1. Candidate loading (entitled songs)
2. Song matching against profiles
3. Snapshot publishing

**Timeline**: Profiling typically completes within seconds, followed by matching. The full job can take 10–30 seconds depending on library size.

### Demo Song Fetching (Existing)

**File: `src/lib/server/onboarding.functions.ts` (lines 328–371)**

The `getDemoSongShowcase()` function fetches pre-seeded demo song:
```typescript
export const getDemoSongShowcase = createServerFn({ method: "GET" })
	.middleware([authMiddleware])
	.handler(async (): Promise<DemoSongData | null> => {
		const demoSongId = env.DEMO_SONG_ID;
		if (!demoSongId) {
			return null;
		}

		const supabase = createAdminSupabaseClient();
		const { data: song, error: songError } = await supabase
			.from("song")
			.select("name, artists, album_name, image_url, genres")
			.eq("id", demoSongId)
			.single();

		if (songError || !song) {
			console.warn("Demo song not found:", demoSongId, songError?.message);
			return null;
		}

		const analysisResult = await getAnalysis(demoSongId);
		// ... validation ...
		return {
			song: { name, artists, albumName, imageUrl, genres },
			analysis: analysisResult.value.analysis,
		};
	});
```

**Demo song ID source**: `env.DEMO_SONG_ID` (from `.env` or env vars; defaults to `undefined`)

## Open Questions / Risks

### 1. **Queue Priority for Onboarding**

The story says "uses `priority` queue band for the onboarding matching path" but the current code uses the account's billing-derived band (free tier → `low`).

**Question**: Should S5-03:
- Explicitly override queue band to `priority` for match snapshot refresh jobs triggered during onboarding?
- Or use the account's natural band and rely on short runtime (~2–5s) anyway?

**Risk**: If profiling/matching runs at `low` priority behind other work, it could exceed the ~10–15s timeout, triggering the fallback unnecessarily. Implementation agent should clarify intent with product/eng.

### 2. **Timeout Fallback Implementation**

The story says "Falls back to canned result after ~10–15s timeout" but does not specify:
- Is the fallback a hardcoded demo match result (pre-built)?
- Or a generic "matching in progress" message?
- Should the component retry after fallback, or allow user to continue?
- What constitutes a timeout — wall-clock time, job completion check, or HTTP response timeout?

**Risk**: Without a concrete fallback strategy, the component cannot gracefully handle slow profiling/matching. Implementation agent must define:
1. Timeout trigger (e.g., 15 seconds after entering match-showcase step)
2. Fallback content (canned result, error message, generic success message)
3. User action after fallback (proceed to plan-selection, retry, skip to ready)

### 3. **Demo Song Matching Data Source**

The story says "shows a live match against the user's real playlists using the demo song" but the match infrastructure is snapshot-based:
- Match snapshots are computed asynchronously after playlist selection
- A snapshot may not be ready immediately when match-showcase step is reached
- `getMatchingSession()` returns `null` if no snapshot exists

**Question**: Should S5-03:
- Poll for the match snapshot until it's ready (with timeout)?
- Fetch match results for the demo song once snapshot is published?
- Or use a different approach (real-time matching API call)?

**Risk**: Snapshot may still be computing during onboarding. The timeout fallback must handle this gracefully.

### 4. **Match Results for Demo Song**

The current `getSongMatches()` function:
- Requires a `snapshotId` parameter
- Operates on snapshot-scoped data
- Filters by entitlement at read time
- Returns playlist suggestions sorted by score

**Question**: Can this function be reused for demo song matching, or does S5-03 need a specialized variant?

**Assumption**: The function should work as-is — demo song is already in the database, and the match snapshot includes match results for all songs. Implementation agent should verify.

### 5. **Component File Location**

The story specifies: `src/features/onboarding/components/MatchShowcaseStep.tsx` (new)

**Status**: File does not exist. Implementation agent must create it.

### 6. **FlagPlaylistsStep Navigation Change**

The current code (line 111) navigates to `"ready"` after saving playlists:
```typescript
await goToStep("ready", { syncStats });
```

S5-03 must change this to navigate through the new steps:
```typescript
await goToStep("song-showcase", { syncStats });
```

**Risk**: This is a breaking change in existing flow. Existing tests or manual workflows that expect `ready` step after flag-playlists will fail.
