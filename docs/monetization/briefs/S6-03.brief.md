## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Read-Model Invariants (DECISIONS.md ┬з4)

**`SongDisplayState` values frozen:**
- `locked` = Not entitled, regardless of shared cache state
- `pending` = Entitled, queued for processing
- `analyzing` = Entitled, processing in progress
- `analyzed` = Entitled, content visible
- `failed` = Entitled, terminal processing failure

**Locked songs must:**
- Never show analysis text or content
- Never show matching status (sub-dimension of `analyzed` only)
- Be visually distinct from other states

**Matching status rule:**
> "Matching status (`has_suggestions | acted | no_suggestions`) is a sub-dimension of `analyzed` songs only."

**Shared artifacts do not imply entitlement:**
> "Read models must filter by effective entitlement, not by `song_analysis` or `item_status` existence."

### User-Facing Language (TERMINOLOGY.md ┬з2)

- **User-facing action verb:** "Unlock"
- **Zero-state CTA:** "Unlock more songs"
- **Internal code:** use `locked`, `unlock`, `entitlement`

### Effective Entitlement Predicate (DECISIONS.md ┬з2)

A song is entitled when:
- Account has active `account_song_unlock` row with `revoked_at IS NULL`, **OR**
- Account has active unlimited access (`unlimited_access_source IS NOT NULL` with valid lifecycle state)

**Note:** This is the single canonical predicate. It's enforced at the SQL RPC level by S3-07; UI need not recompute it.

### Self-Hosted / Provider-Disabled (DECISIONS.md ┬з3)

When `BILLING_ENABLED=false`:
- Accounts get explicit `unlimited_access_source = 'self_hosted'`
- Accounts have `unlimited_access_source = 'self_hosted'` тЖТ all songs entitled
- **"Provider-disabled: no locked songs should appear (all entitled)"** тАФ per story scope

## Summary

S6-03 requires adding visual treatment to locked songs in the liked songs list and detail view. The data structure (`displayState`) and data filtering (analysis/matching_status suppression) are already in place from S3-07 and S3-11. The implementation agent must:

1. **SongCard:** Add visual distinction (muted colors, lock icon, or similar) to locked song rows
2. **SongDetailPanel/PanelContent:** Suppress analysis content for locked songs; show appropriate message or UI state
3. **Affordance:** Decide on unlock/explore affordance type and placement (may be placeholder for S6-03 if paywall is S6-05)
4. **Testing:** Ensure locked and non-locked songs render distinctly

Key unknowns: exact visual treatment, affordance interaction design, and whether affordance is functional (navigates) or informational (tooltip) in S6-03.
