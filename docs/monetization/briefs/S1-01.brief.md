# Implementation Brief: S1-01 Core Billing Tables Migration

## Goal
Create the foundational billing tables (`account_billing`, `account_song_unlock`, `credit_transaction`) with constraints, indexes, and RLS policies.

## Current State

No billing tables exist yet. The target foreign key tables are present:

**supabase/migrations/20260116155214_create_account.sql:**
```sql
CREATE TABLE account (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  spotify_id TEXT UNIQUE NOT NULL,
  email TEXT,
  display_name TEXT,
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT now() NOT NULL
);
```

**supabase/migrations/20260116160000_create_song.sql:**
```sql
CREATE TABLE song (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  spotify_id TEXT UNIQUE NOT NULL,
  isrc TEXT,
  name TEXT NOT NULL,
  -- ... other columns
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT now() NOT NULL
);
```

The repo follows a consistent RLS pattern:
```sql
ALTER TABLE tablename ENABLE ROW LEVEL SECURITY;
CREATE POLICY "tablename_deny_all" ON tablename FOR ALL USING (false);
```

Tables with `updated_at` columns get automatic update triggers using the existing `update_updated_at_column()` function.

## Changes Required

Create a new migration file `supabase/migrations/{timestamp}_billing_core_tables.sql` with:

1. **`account_billing` table** - exactly as specified in V2 plan § Schema:
   - All columns with exact names and types
   - CHECK constraints for `plan`, `credit_balance`, `unlimited_access_source`, `subscription_status`
   - Defaults: `plan='free'`, `credit_balance=0`, `subscription_status='none'`, `cancel_at_period_end=FALSE`
   - Foreign key: `account_id REFERENCES account(id) ON DELETE CASCADE`
   - Unique constraints on `stripe_customer_id` and `stripe_subscription_id`
   - `updated_at` trigger (table has both `created_at` and `updated_at`)

2. **`account_song_unlock` table** - exactly as specified in V2 plan § Schema:
   - All columns with exact names and types  
   - CHECK constraints for `source` and `revoked_reason`
   - Foreign keys: `account_id REFERENCES account(id) ON DELETE CASCADE`, `song_id REFERENCES song(id) ON DELETE CASCADE`
   - `UNIQUE(account_id, song_id)` constraint
   - Index: `idx_account_song_unlock_account ON account_song_unlock(account_id)`
   - No `updated_at` column or trigger (append-only table)

3. **`credit_transaction` table** - exactly as specified in V2 plan § Schema:
   - All columns with exact names and types
   - CHECK constraint for `reason` 
   - Foreign key: `account_id REFERENCES account(id) ON DELETE CASCADE`
   - Index: `idx_credit_txn_account ON credit_transaction(account_id, created_at DESC)`
   - No `updated_at` column or trigger (immutable ledger)

4. **RLS policies** for all three tables:
   - `ALTER TABLE {tablename} ENABLE ROW LEVEL SECURITY;`
   - `CREATE POLICY "{tablename}_deny_all" ON {tablename} FOR ALL USING (false);`

## Constraints

From DECISIONS.md and TERMINOLOGY.md (frozen terms):

- **Plan values**: `'free' | 'quarterly' | 'yearly'`
- **Subscription status values**: `'none' | 'active' | 'past_due' | 'canceled' | 'unpaid' | 'incomplete' | 'incomplete_expired'`
- **Unlock sources**: `'free_auto' | 'pack' | 'unlimited' | 'self_hosted' | 'admin'`
- **Revocation reasons**: `'refund' | 'chargeback' | 'admin'`
- **Credit transaction reasons**: `'song_unlock' | 'pack_purchase' | 'credit_conversion' | 'credit_conversion_reversal' | 'replacement_grant' | 'refund' | 'chargeback_reversal' | 'admin_adjustment'`
- **Unlimited access sources**: `'subscription' | 'self_hosted'`
- **Table and column names**: Exactly as specified in V2 plan Schema section
- **Index names**: `idx_account_song_unlock_account` and `idx_credit_txn_account`
- **RLS pattern**: Must match existing repo convention of deny-all policies with service_role bypass

## Acceptance Criteria

From S1-01 story:
- [ ] Migration applies cleanly on a fresh Supabase instance  
- [ ] All CHECK constraints reject invalid values (tested via `INSERT` attempts)
- [ ] `UNIQUE` constraints prevent duplicate rows
- [ ] RLS enabled; direct `anon`/`authenticated` access denied
- [ ] `service_role` can read/write all three tables
- [ ] Foreign keys reference `account(id)` and `song(id)` with `ON DELETE CASCADE`

## Verification Steps

1. `supabase db reset` - completes without errors
2. Manual insert/select tests via `service_role` - succeed
3. Manual insert via `anon` role - denied
4. Invalid value inserts (wrong plan, negative credit_balance, etc.) - rejected by CHECK constraints
5. Duplicate `(account_id, song_id)` insert into `account_song_unlock` - rejected by UNIQUE constraint

## Risks

None identified. The schema is fully specified in the V2 plan with exact DDL, foreign key targets exist, and the RLS pattern is established in the repo.
