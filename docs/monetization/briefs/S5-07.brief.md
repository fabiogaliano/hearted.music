### `src/lib/server/onboarding.functions.ts`

**`markOnboardingComplete` function (lines 200–230):**

```typescript
export const markOnboardingComplete = createServerFn({
	method: "POST",
})
	.middleware([authMiddleware])
	.handler(async ({ context }): Promise<{ success: true }> => {
		const { session } = context;

		const result = await completeOnboarding(session.accountId);

		if (Result.isError(result)) {
			throw new OnboardingError("complete_onboarding", result.error);
		}

		const supabase = createAdminSupabaseClient();
		const billingResult = await readBillingState(supabase, session.accountId);

		if (Result.isOk(billingResult)) {
			const billing = billingResult.value;
			const isFree =
				billing.plan === "free" &&
				!hasUnlimitedAccess(billing) &&
				billing.creditBalance === 0;

			if (isFree) {
				const allocationResult = await grantFreeAllocation(
					supabase,
					session.accountId,
				);
				if (Result.isError(allocationResult)) {
					console.error(
						"[onboarding] Free allocation failed:",
						allocationResult.error,
					);
				}
			}
		} else {
			console.error(
				"[onboarding] Failed to read billing state for free allocation:",
				billingResult.error,
			);
		}

		return { success: true };
	});
```

**Current behavior:**
- Sets `onboarding_completed_at` timestamp via `completeOnboarding()`
- Reads billing state and checks if user is free-plan (no unlimited access, no credit balance)
- **If free**: calls `grantFreeAllocation()` to unlock up to 15 most-recent liked songs
- Logs errors but returns success regardless (fire-and-forget for allocation failures)
- Does NOT return billing state to ReadyStep; allocation is handled server-side only

### `src/lib/domains/billing/queries.ts`

**`readBillingState` function (complete):**

```typescript
export async function readBillingState(
	supabase: AdminSupabaseClient,
	accountId: string,
): Promise<Result<BillingState, DbError>> {
	const result = await fromSupabaseMaybe(
		supabase
			.from("account_billing")
			.select("*")
			.eq("account_id", accountId)
			.single(),
	);

	if (Result.isError(result)) return result;

	const row = result.value;

	if (row === null) {
		// Self-healing: insert default free row on missing data
		const { error: insertError } = await supabase
			.from("account_billing")
			.insert({ account_id: accountId })
			.select()
			.single();

		if (insertError && insertError.code !== "23505") {
			return Result.err(new DatabaseError({...}));
		}

		return Result.ok(FREE_DEFAULT);
	}

	const plan = parsePlan(row.plan);
	const unlimitedAccess = parseUnlimitedAccess(row.unlimited_access_source);
	// ... parse other fields and resolve queueBand ...

	return Result.ok({ plan, creditBalance, subscriptionStatus, cancelAtPeriodEnd, unlimitedAccess, queueBand });
}
```

**Current behavior:**
- Reads `account_billing` row for the account
- Returns normalized BillingState with plan, creditBalance, unlimitedAccess, subscriptionStatus
- Self-heals if no row exists (creates default free row)
- Already used by `markOnboardingComplete` to detect free-plan users

### `src/lib/domains/billing/unlocks.ts`

**`grantFreeAllocation` function (snippet):**

```typescript
export async function grantFreeAllocation(
	supabase: AdminSupabaseClient,
	accountId: string,
): Promise<Result<GrantFreeAllocationResult, UnlockError>> {
	const { data: likedSongs, error: likedError } = await supabase
		.from("liked_song")
		.select("song_id")
		.eq("account_id", accountId)
		.is("unliked_at", null)
		.order("liked_at", { ascending: false })
		.limit(15);  // FREE_ALLOCATION_LIMIT

	// ... error handling ...

	const candidateIds = (likedSongs ?? []).map((row) => row.song_id);

	const { data: unlockRows, error: unlockError } = await supabase.rpc(
		"insert_song_unlocks_without_charge",
		{
			p_account_id: accountId,
			p_song_ids: candidateIds,
			p_source: "free_auto",
		},
	);

	// ... error handling ...

	// Emit control-plane change
	const unlockedIds = (unlockRows ?? []).map((row) => row.song_id);
	if (unlockedIds.length > 0) {
		try {
			await applyLibraryProcessingChange(
				BillingChanges.songsUnlocked(accountId, unlockedIds),
			);
		} catch (err) {
			console.error("[unlocks] Failed to apply library processing change:", err);
		}
	}

	return Result.ok({ unlockedIds });
}
```

**Current behavior:**
- Fetches up to 15 most-recent liked songs (ordered by `liked_at` DESC)
- Calls `insert_song_unlocks_without_charge` RPC with `source='free_auto'`
- Emits `BillingChanges.songsUnlocked(...)` to control-plane
- Already fully implemented and called by `markOnboardingComplete`

| Constraint | Impact |
|---|---|
| Provider-disabled path must not show billing UI or purchase surfaces | Copy must adapt; no checkout buttons shown in provider-disabled mode |
| Self-hosted accounts have `unlimited_access_source='self_hosted'` | Full-library processing starts automatically; no plan selection needed |

| Effective entitlement predicate | A song is entitled when `unlock row with revoked_at IS NULL` OR `active unlimited access` |
| Free allocation is one-time at onboarding completion | No re-trigger on navigation or revisit |

## Open Questions / Risks

### 1. **Where to load billing state for ReadyStep**

**Issue:** ReadyStep must display billing-aware copy, but currently doesn't receive billing state.

**Current code path:**
- `markOnboardingComplete` server function reads billing state server-side only
- ReadyStep has no access to the user's plan or unlimited access status

**Ambiguity:**
- Should billing state be loaded in ReadyStep component (client-side server function call)?
- Or should it be loaded in the route loader and passed via Onboarding context?
- Or should a new server function be created to expose just the plan/access info for copy selection?

**Impact:** Determines whether ReadyStep becomes billing-aware or delegates to parent/route.

### 2. **Copy variants — how to structure**

**Issue:** Story specifies copy for free, pack, unlimited, and self-hosted, but doesn't specify exact text.

**Story guidance:**
- Free: "Exploring your 15 songs…" (or similar)
- Pack: "Exploring your selected songs…"
- Unlimited: "Going through every song…" (current hardcoded copy)
- Self-hosted: "Going through every song…"

**Ambiguity:**
- What exact text should each variant use? (Story says "or similar")
- Should pack copy vary based on purchased balance (500 vs remaining)?
- Should copy be a message or an action description?

**Impact:** User-visible messaging and product tone.

### 3. **Self-hosted account entitlement detection**

**Issue:** Self-hosted users should see unlimited copy and get full processing, but how is this detected?

**Current code:**
- S2-03 sets `unlimited_access_source='self_hosted'` on account creation
- `BillingState.unlimitedAccess.kind === "self_hosted"` can be checked

**Ambiguity:**
- Is `hasUnlimitedAccess(billing) === true` sufficient for copy variant detection?
- Or do we need to distinguish between `subscription` and `self_hosted` for different messages?
- The story says both unlimited and self-hosted should show "Going through every song…" so likely yes

**Impact:** Copy logic in ReadyStep.

### 4. **Pack users in onboarding**

**Issue:** Story mentions pack users but pack purchase happens via PlanSelectionStep (external checkout).

**Current flow:**
- User selects "Song Pack" in PlanSelectionStep
- Redirects to Stripe checkout
- After payment, user returns and polling detects billing state change
- User navigates to ready with updated billing state (creditBalance > 0)

**Ambiguity:**
- Is a pack-having user who reaches ReadyStep expected to exist, or are they always on the unlimited path after pack redirect?
- If they do reach ReadyStep, are they considered "pack users" (creditBalance > 0) or should copy only show for newly-purchased balance?
- Should pack users be shown different copy if they already have explored/unlocked songs?

**Impact:** Whether ReadyStep copy needs dynamic pack handling or if pack users are filtered out earlier.

### 5. **Provider-disabled flow verification**

**Issue:** Story says "verify provider-disabled onboarding path works end to end", but this may already be working.

**Current implementation:**
- Route loader auto-skips plan-selection when `BILLING_ENABLED=false`
- Self-hosted accounts have unlimited access from account creation
- Full-library processing starts on content activation (not gated by onboarding completion)

**Ambiguity:**
- What specifically needs to be "verified"? Is this a test validation or a manual user flow check?
- Are there known bugs in the provider-disabled path, or is this exploratory verification?
- Should verification include both the auto-skip behavior AND the full-processing start?

**Impact:** Testing scope and manual verification checklist.

### 6. **Polling state after checkout in provider-disabled mode**

**Issue:** PlanSelectionStep won't be shown when `BILLING_ENABLED=false`, but what if a user somehow reaches it?

**Current behavior:**
- PlanSelectionStep only renders when loaded; route loader blocks access
- No explicit "this step is disabled" UI

**Ambiguity:**
- Is it sufficient to rely on route loader auto-skip, or should the component itself have a guard?

### 7. **Timing of full-library processing for self-hosted users**

**Issue:** Self-hosted accounts have unlimited access from account creation, so full-library processing can start immediately.

**Current code:**

**Ambiguity:**

- Does self-hosted processing need a second reprioritization call after onboarding?
- Or is the account-creation reprioritization sufficient?

**Impact:** Whether ReadyStep completion needs additional processing triggers for self-hosted accounts.

### 8. **Copy durability and migration scope**
