## Changes Required

### 1. Create `src/lib/domains/billing/unlocks.ts` (new file)

Define result and error types, then implement the two orchestration functions.

**Typed error union:**
```typescript
export type UnlockError =
  | { kind: "insufficient_balance"; required: number; available: number }
  | { kind: "invalid_songs"; songIds: string[] }
  | { kind: "unlimited_access_active" }
  | { kind: "db_error"; cause: DbError };
```

**`requestSongUnlock(supabase, accountId, songIds)`:**
1. Call `readBillingState(supabase, accountId)` — if DB error, return `{ kind: "db_error" }`.
2. If `hasUnlimitedAccess(billingState)` → return `{ kind: "unlimited_access_active" }`.
3. Call `supabase.rpc("unlock_songs_for_account", { p_account_id: accountId, p_song_ids: songIds })`.
4. If RPC error message contains "insufficient balance" → return `{ kind: "insufficient_balance", required: songIds.length, available: billingState.creditBalance }`.
5. If RPC error message contains "not currently liked" → return `{ kind: "invalid_songs", songIds }`.
6. If other RPC error → return `{ kind: "db_error" }`.
7. Parse JSON result: `{ newly_unlocked_song_ids, already_unlocked_song_ids }`.
8. If `newly_unlocked_song_ids.length > 0`, call `applyLibraryProcessingChange(BillingChanges.songsUnlocked(accountId, newly_unlocked_song_ids))`.
9. Return `Result.ok({ newlyUnlockedIds, alreadyUnlockedIds, remainingBalance })`.

**`grantFreeAllocation(supabase, accountId)`:**
1. Query `liked_song` for up to 15 most-recent active (unliked_at IS NULL) song IDs, ordered by `liked_at DESC`.
2. If none, return `Result.ok({ unlockedIds: [] })`.
3. Call `supabase.rpc("insert_song_unlocks_without_charge", { p_account_id: accountId, p_song_ids: candidateIds, p_source: "free_auto" })`.
4. Collect returned `song_id` values.
5. If any were unlocked, call `applyLibraryProcessingChange(BillingChanges.songsUnlocked(accountId, unlockedIds))`.
6. Return `Result.ok({ unlockedIds })`.

**Full return types:**

export type GrantFreeAllocationResult = {

**Note on liked song query:** Query directly via `supabase.from("liked_song")` in `unlocks.ts`, filtered by `account_id = accountId`, `unliked_at IS NULL`, ordered `liked_at DESC`, limited to 15. This avoids importing from the library domain queries file (avoids cross-domain coupling). ### 2. Modify `src/lib/server/billing.functions.ts`

Add `requestSongUnlock` server function. Keep existing `getBillingState` untouched.

**New function:**
```typescript
import { z } from "zod";
import { requestSongUnlock as orchestrateUnlock } from "@/lib/domains/billing/unlocks";

const RequestSongUnlockSchema = z.object({
  songIds: z.array(z.string().uuid()).min(1).max(500),
});

export type RequestSongUnlockResponse =
  | { success: true; newlyUnlockedIds: string[]; alreadyUnlockedIds: string[]; remainingBalance: number }
  | { success: false; error: "insufficient_balance"; required: number; available: number }
  | { success: false; error: "invalid_songs"; songIds: string[] }
  | { success: false; error: "unlimited_access_active" }
  | { success: false; error: "internal_error" };

export const requestSongUnlock = createServerFn({ method: "POST" })
  .middleware([authMiddleware])
  .inputValidator((data) => RequestSongUnlockSchema.parse(data))
  .handler(async ({ data, context }): Promise<RequestSongUnlockResponse> => {
    const supabase = createAdminSupabaseClient();
    const result = await orchestrateUnlock(supabase, context.session.accountId, data.songIds);

    if (Result.isError(result)) {
      const err = result.error;
      if (err.kind === "insufficient_balance") {
        return { success: false, error: "insufficient_balance", required: err.required, available: err.available };
      }
      if (err.kind === "invalid_songs") {
        return { success: false, error: "invalid_songs", songIds: err.songIds };
      }
      if (err.kind === "unlimited_access_active") {
        return { success: false, error: "unlimited_access_active" };
      }
      return { success: false, error: "internal_error" };
    }

    return {
      success: true,
      newlyUnlockedIds: result.value.newlyUnlockedIds,
      alreadyUnlockedIds: result.value.alreadyUnlockedIds,
      remainingBalance: result.value.remainingBalance,
    };
  });
```

## Constraints

From DECISIONS.md:

- **All-or-nothing for net-new unlocks** (§7): the RPC enforces this; do not partial-fill in TypeScript layer.
- **`requestSongUnlock` must not be callable when account has active unlimited access** (§1 "Unavailable while unlimited active"): guard in `requestSongUnlock` by checking `hasUnlimitedAccess(billingState)` before calling the RPC.
- **Free allocation: `source='free_auto'`**, does not deduct balance, forfeits unused slots — `insert_song_unlocks_without_charge` with `p_source='free_auto'`, no balance check needed.
- **Free allocation size: 15 songs** (§1): LIMIT 15 on liked_song query.
- **Errors as typed values, not thrown exceptions** (AGENTS.md): `requestSongUnlock` and `grantFreeAllocation` return `Result<..., UnlockError>`.
- **No `any`, no non-null assertions** (AGENTS.md).
- **No barrel exports** (CLAUDE.md).
- **Control plane**: only emit `BillingChanges.songsUnlocked` — never `unlimitedActivated` or `candidateAccessRevoked` from this module.
- **`unlock_songs_for_account` already handles all-or-nothing and balance deduction** — the TS layer must not duplicate that logic.

## Risks

1. **`unlock_songs_for_account` return type is `Json`** (not a typed struct). After the RPC call, the result must be narrowed before accessing `newly_unlocked_song_ids` and `already_unlocked_song_ids`. Use a type guard or parse with Zod inline. Do not use `as` without a runtime check.

2. **RPC error discrimination**: `unlock_songs_for_account` raises PostgreSQL exceptions with messages like `"insufficient balance (credit_balance=..."` and `"not currently liked"`. The TS layer discriminates errors by checking `error.message` string patterns. This is fragile if the RPC message wording changes; document the coupling. An alternative would be to check Postgres error codes, but the RPCs raise generic exceptions without custom SQL state codes.

3. **`applyLibraryProcessingChange` failure is fire-and-forget**: The existing pattern in other server functions (e.g., `onboarding.functions.ts`) awaits `applyLibraryProcessingChange` but does not surface its failure to the caller. Follow the same pattern — log the error but return the unlock result as success regardless.

4. **Liked song query in `grantFreeAllocation`**: The story says "up to 15 most-recent liked songs". The liked_song query must filter `unliked_at IS NULL` (active likes only). The `liked_song` table is queried directly via the `AdminSupabaseClient` parameter — consistent with how `readBillingState` receives the client. This avoids importing from `library/liked-songs/queries.ts` which creates its own client internally (those functions don't accept a client parameter).

5. **`remainingBalance` computation**: After `unlock_songs_for_account` succeeds, `remainingBalance = billingState.creditBalance - newly_unlocked_song_ids.length`. The `billingState` was read before the RPC call; this arithmetic is safe because the RPC deducts exactly `cardinality(net_new)` from balance. Read `billingState` before calling the RPC so balance is known.
