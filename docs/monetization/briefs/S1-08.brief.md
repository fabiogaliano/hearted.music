### Reservation model (how prepare interacts with unlock_songs_for_account)

`unlock_songs_for_account` (in `20260405160000_core_unlock_rpcs.sql`) already implements the reservation-aware spendable check:

```sql
-- Reads allocation rows to compute what's reserved
SELECT COALESCE(SUM(scca.reserved_credits), 0) INTO v_reserved
FROM subscription_credit_conversion_allocation scca
JOIN subscription_credit_conversion scc ON scca.conversion_id = scc.id
WHERE scc.account_id = p_account_id
  AND scc.status = 'pending';

-- Spendable = credit_balance − reserved
IF (v_balance - v_reserved) < v_cost THEN
  RAISE EXCEPTION 'insufficient balance ...';
END IF;
```

This means `prepare` must use a **virtual reservation model**: it creates allocation rows without physically reducing `remaining_credits` on lots or `credit_balance`. The reservation is enforced by the spendable check in `unlock_songs_for_account`, which filters `status = 'pending'` allocations. Once a conversion is `applied` or `released`, `status != 'pending'`, so the allocation rows no longer count against spendable balance.

---

## Reservation Model: Full Lifecycle

| Operation | `remaining_credits` on lots | `credit_balance` | Allocation rows | Conversion status |
|---|---|---|---|---|
| `prepare` (new) | **unchanged** | **unchanged** | Created | `pending` |
| `prepare` (reuse) | **unchanged** | **unchanged** | **unchanged** | `pending` |
| `link` | unchanged | unchanged | unchanged | `pending` |
| `release` | **unchanged** | **unchanged** | Unchanged (status change lifts virtual reservation) | `released` |
| `apply` | **Reduced** by `reserved_credits` per lot | **Reduced** by `converted_credits` | Unchanged | `applied` |
| `reverse` | **Increased** by `reserved_credits` per lot | **Increased** by `converted_credits` | Unchanged | `reversed` |

**Key insight for `release`:** The story says "restores reserved credits to lots and credit_balance". Because the reservation is virtual (allocation rows + `status = 'pending'` filter), setting status to `released` automatically lifts the reservation — `unlock_songs_for_account` no longer subtracts these allocations from spendable. No physical column changes are needed on lots or `credit_balance`. The allocation rows remain on the DB row (they cascade-delete only if the conversion row is deleted) but become invisible to the spendable check.

### Function 1: `prepare_subscription_upgrade_conversion`

```sql
CREATE OR REPLACE FUNCTION prepare_subscription_upgrade_conversion(
  p_account_id UUID,
  p_target_plan TEXT
) RETURNS TABLE(converted_credits INTEGER, discount_cents INTEGER, conversion_id UUID)
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_conversion_id     UUID;
  v_converted_credits INTEGER;
  v_discount_cents    INTEGER;
  v_lot               RECORD;
  v_lot_discount      INTEGER;
BEGIN
  -- 1. Check for existing pending conversion (reuse if present)
  SELECT scc.id, scc.converted_credits, scc.discount_cents
  INTO v_conversion_id, v_converted_credits, v_discount_cents
  FROM subscription_credit_conversion scc
  WHERE scc.account_id = p_account_id
    AND scc.status = 'pending'
  FOR UPDATE;  -- lock to prevent concurrent operations on this conversion

  IF FOUND THEN
    RETURN QUERY SELECT v_converted_credits, v_discount_cents, v_conversion_id;
    RETURN;
  END IF;

  -- 2. Compute converted_credits and discount_cents from open lots (FIFO)
  v_converted_credits := 0;
  v_discount_cents    := 0;

  -- 3. Create the conversion row
  INSERT INTO subscription_credit_conversion (
    account_id, target_plan, status, converted_credits, discount_cents
  ) VALUES (
    p_account_id, p_target_plan, 'pending', 0, 0  -- will update after lot scan
  )
  RETURNING id INTO v_conversion_id;

  -- 4. Lock open lots FIFO and create allocation rows
  FOR v_lot IN
    SELECT id, remaining_credits, original_credits, price_cents
    FROM pack_credit_lot
    WHERE account_id = p_account_id
      AND remaining_credits > 0
    ORDER BY created_at ASC, id ASC
    FOR UPDATE
  LOOP
    -- Pro-rated discount: floor(remaining_credits / original_credits * price_cents)
    v_lot_discount := (v_lot.remaining_credits * v_lot.price_cents) / v_lot.original_credits;

    INSERT INTO subscription_credit_conversion_allocation (
      conversion_id, pack_credit_lot_id, reserved_credits, reserved_discount_cents
    ) VALUES (
      v_conversion_id, v_lot.id, v_lot.remaining_credits, v_lot_discount
    );

    v_converted_credits := v_converted_credits + v_lot.remaining_credits;
    v_discount_cents    := v_discount_cents    + v_lot_discount;
  END LOOP;

  -- 5. Update conversion row with computed totals
  UPDATE subscription_credit_conversion
  SET converted_credits = v_converted_credits,
      discount_cents    = v_discount_cents
  WHERE id = v_conversion_id;

  RETURN QUERY SELECT v_converted_credits, v_discount_cents, v_conversion_id;
END;
$$;
```

**Notes:**
- `FOR UPDATE` on the existing pending row prevents two concurrent callers from both passing the "not found" check and creating duplicate rows. The partial unique index on `(account_id) WHERE status='pending'` provides the database-level safety net; the in-RPC check + lock ensures deterministic reuse.
- If zero open lots exist, the function inserts a conversion with `converted_credits=0, discount_cents=0` and no allocation rows. This is valid (zero-discount upgrade).
- Integer division in PostgreSQL truncates toward zero (same as `floor` for positive values), giving the correct pro-rated floor.
- Allocation rows have `ON DELETE CASCADE` from conversion, so they are cleaned up if the conversion row is ever deleted.

### Function 2: `link_subscription_upgrade_checkout`

```sql
CREATE OR REPLACE FUNCTION link_subscription_upgrade_checkout(
  p_conversion_id       UUID,
  p_checkout_session_id TEXT
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  UPDATE subscription_credit_conversion
  SET checkout_session_id = p_checkout_session_id
  WHERE id = p_conversion_id
    AND status = 'pending';

  IF NOT FOUND THEN
    RAISE EXCEPTION 'link_subscription_upgrade_checkout: conversion % not found or not pending',
      p_conversion_id;
  END IF;
END;
$$;
```

**Notes:**
- Does not use `SELECT ... FOR UPDATE` before the UPDATE because the UPDATE itself is atomic on the row. The state check (`AND status = 'pending'`) is part of the UPDATE predicate.
- The unique index `idx_subscription_credit_conversion_checkout_session` (on `checkout_session_id WHERE IS NOT NULL`) will reject duplicate checkout session IDs across conversions.

### Function 3: `release_subscription_upgrade_conversion`

```sql
CREATE OR REPLACE FUNCTION release_subscription_upgrade_conversion(
  p_conversion_id UUID
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  -- Lock the conversion row before checking/mutating status
  PERFORM id
  FROM subscription_credit_conversion
  WHERE id = p_conversion_id
  FOR UPDATE;

  UPDATE subscription_credit_conversion
  SET status = 'released'
  WHERE id = p_conversion_id
    AND status = 'pending';

  IF NOT FOUND THEN
    RAISE EXCEPTION 'release_subscription_upgrade_conversion: conversion % not found or not pending',
      p_conversion_id;
  END IF;

  -- No lot or credit_balance changes: the reservation is virtual.
  -- Setting status to 'released' causes unlock_songs_for_account to stop
  -- counting this conversion's allocation rows in its spendable check.
END;
$$;
```

**Notes:**
- Because the reservation is virtual, this function only changes the status. The allocation rows remain in the DB but are now invisible to `unlock_songs_for_account`'s `status = 'pending'` filter, so the reserved credits become spendable again automatically.
- `SELECT ... FOR UPDATE` before the UPDATE prevents concurrent release + apply racing.
- State transition enforced: only `pending → released`. Any other status raises an exception.

### Function 4: `apply_subscription_upgrade_conversion`

```sql
CREATE OR REPLACE FUNCTION apply_subscription_upgrade_conversion(
  p_conversion_id          UUID,
  p_stripe_subscription_id TEXT,
  p_stripe_invoice_id      TEXT,
  p_applied_stripe_event_id TEXT
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_account_id        UUID;
  v_converted_credits INTEGER;
  v_balance           INTEGER;
  v_alloc             RECORD;
BEGIN
  -- 1. Lock conversion row and read account/credit values
  SELECT scc.account_id, scc.converted_credits
  INTO v_account_id, v_converted_credits
  FROM subscription_credit_conversion scc
  WHERE scc.id = p_conversion_id
    AND scc.status = 'pending'
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'apply_subscription_upgrade_conversion: conversion % not found or not pending',
      p_conversion_id;
  END IF;

  -- 2. Lock account_billing before balance mutation
  SELECT ab.credit_balance INTO v_balance
  FROM account_billing ab
  WHERE ab.account_id = v_account_id
  FOR UPDATE;

  -- 3. Consume reserved credits from each lot (physically reduce remaining_credits)
  FOR v_alloc IN
    SELECT scca.pack_credit_lot_id, scca.reserved_credits
    FROM subscription_credit_conversion_allocation scca
    WHERE scca.conversion_id = p_conversion_id
    ORDER BY scca.pack_credit_lot_id  -- deterministic lock order to prevent deadlock
    FOR UPDATE
  LOOP
    UPDATE pack_credit_lot
    SET remaining_credits = remaining_credits - v_alloc.reserved_credits
    WHERE id = v_alloc.pack_credit_lot_id;
  END LOOP;

  -- 4. Deduct credit_balance
  UPDATE account_billing
  SET credit_balance = credit_balance - v_converted_credits
  WHERE account_id = v_account_id;

  -- 5. Write ledger row
  INSERT INTO credit_transaction (
    account_id, amount, balance_after, reason, stripe_event_id, metadata
  ) VALUES (
    v_account_id,
    -v_converted_credits,
    v_balance - v_converted_credits,
    'credit_conversion',
    p_applied_stripe_event_id,
    jsonb_build_object('conversion_id', p_conversion_id)
  );

  -- 6. Mark conversion applied with Stripe references
  UPDATE subscription_credit_conversion
  SET status                  = 'applied',
      stripe_subscription_id  = p_stripe_subscription_id,
      stripe_invoice_id       = p_stripe_invoice_id,
      applied_stripe_event_id = p_applied_stripe_event_id
  WHERE id = p_conversion_id;
END;
$$;
```

**Notes:**
- `SELECT ... FOR UPDATE` on the conversion row before locking `account_billing` — consistent lock ordering avoids deadlock with other RPCs.
- Lot locks use `ORDER BY pack_credit_lot_id` (stable UUID ordering) so concurrent apply calls across different conversions sharing lots acquire locks in the same order.
- `v_balance - v_converted_credits` for `balance_after` uses the balance read at step 2 (before the UPDATE), which is the correct balance captured at ledger write time.
- If `v_converted_credits = 0` (no lots at prepare time), the loop is a no-op and `credit_balance` is unchanged. The ledger row records `amount=0` — caller can handle this.

### Function 5: `reverse_subscription_upgrade_conversion`

```sql
CREATE OR REPLACE FUNCTION reverse_subscription_upgrade_conversion(
  p_conversion_id   UUID,
  p_stripe_event_id TEXT
) RETURNS VOID
LANGUAGE plpgsql
VOLATILE
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_account_id        UUID;
  v_converted_credits INTEGER;
  v_balance           INTEGER;
  v_alloc             RECORD;
BEGIN
  -- 1. Lock conversion row and verify it is in applied state
  SELECT scc.account_id, scc.converted_credits
  INTO v_account_id, v_converted_credits
  FROM subscription_credit_conversion scc
  WHERE scc.id = p_conversion_id
    AND scc.status = 'applied'
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'reverse_subscription_upgrade_conversion: conversion % not found or not applied',
      p_conversion_id;
  END IF;

  -- 2. Lock account_billing before balance mutation
  SELECT ab.credit_balance INTO v_balance
  FROM account_billing ab
  WHERE ab.account_id = v_account_id
  FOR UPDATE;

  -- 3. Restore exact allocated lot amounts (reverse of apply's lot consumption)
  FOR v_alloc IN
    SELECT scca.pack_credit_lot_id, scca.reserved_credits
    FROM subscription_credit_conversion_allocation scca
    WHERE scca.conversion_id = p_conversion_id
    ORDER BY scca.pack_credit_lot_id  -- consistent lock order
    FOR UPDATE
  LOOP
    UPDATE pack_credit_lot
    SET remaining_credits = remaining_credits + v_alloc.reserved_credits
    WHERE id = v_alloc.pack_credit_lot_id;
  END LOOP;

  -- 4. Restore credit_balance
  UPDATE account_billing
  SET credit_balance = credit_balance + v_converted_credits
  WHERE account_id = v_account_id;

  -- 5. Write ledger row
  INSERT INTO credit_transaction (
    account_id, amount, balance_after, reason, stripe_event_id, metadata
  ) VALUES (
    v_account_id,
    v_converted_credits,
    v_balance + v_converted_credits,
    'credit_conversion_reversal',
    p_stripe_event_id,
    jsonb_build_object('conversion_id', p_conversion_id)
  );

  -- 6. Mark conversion reversed
  UPDATE subscription_credit_conversion
  SET status = 'reversed'
  WHERE id = p_conversion_id;
END;
$$;
```

**Notes:**
- Restores `remaining_credits` exactly per allocation row — idempotent if the lot's `remaining_credits + reserved_credits <= original_credits` (the CHECK constraint `remaining_credits <= original_credits` will reject if restoration would exceed original; this would indicate a data integrity problem).
- State transition: `applied → reversed` only.
- After reversal, the lots are usable again and `credit_balance` is restored.

## Acceptance Criteria

- [ ] `prepare` creates conversion + allocation rows with correct `converted_credits` and `discount_cents` computed from open lots (FIFO)
- [ ] Second `prepare` for same account (pending already exists) reuses the existing pending conversion and returns its values
- [ ] `link` attaches `checkout_session_id` to a pending conversion; raises exception if not pending
- [ ] `release` sets status to `released`; the virtual reservation is lifted (allocation rows remain but `status != 'pending'` removes them from the spendable check); raises exception if not pending
- [ ] `apply` reduces `remaining_credits` on each allocated lot by `reserved_credits`; reduces `credit_balance` by `converted_credits`; writes `credit_transaction` with `reason='credit_conversion'`; sets `stripe_subscription_id`, `stripe_invoice_id`, `applied_stripe_event_id` on conversion; raises exception if not pending
- [ ] `reverse` restores `remaining_credits` on each allocated lot by `reserved_credits`; increments `credit_balance` by `converted_credits`; writes `credit_transaction` with `reason='credit_conversion_reversal'`; sets status `reversed`; raises exception if not applied
- [ ] State transitions enforce valid sequences: only `pending→applied`, `pending→released`, `applied→reversed`
- [ ] `unlock_songs_for_account` cannot spend reserved credits while a `pending` conversion holds them
- [ ] `supabase db reset` completes without errors

## Risks and Ambiguities

### 1. Discount computation rounding

The formula `(reserved_credits * price_cents) / original_credits` uses integer division (truncation toward zero). For a 150/500 lot at $5.99 (599 cents): `(150 * 599) / 500 = 89850 / 500 = 179` (exact). For 50/500: `(50 * 599) / 500 = 29950 / 500 = 59`. This gives the user the correct floor amount (never rounds up in their favor beyond what they paid for).

### 2. Zero-lot prepare

If the account has no open lots (e.g., all credits were manually granted with no lot rows), `prepare` creates a conversion row with `converted_credits=0, discount_cents=0` and zero allocation rows. This is technically valid but useless (zero discount). The billing service should check `converted_credits > 0` before proceeding with the conversion.

### 3. Deadlock prevention in lot locking

`apply` and `reverse` lock allocation rows with `ORDER BY pack_credit_lot_id`. Since UUIDs are fixed random values, this ordering is stable. Two concurrent operations affecting the same lots will acquire row locks in the same order, preventing deadlock. The lock on `account_billing` is acquired separately (before lot locks), which is consistent with the pattern in `unlock_songs_for_account` (which also locks `account_billing` before lots).

### 4. `prepare` reuse and target_plan mismatch

If a pending conversion exists with `target_plan='yearly'` and the caller requests `target_plan='quarterly'`, the function returns the existing `'yearly'` conversion (reuse wins). The billing service must call `release_subscription_upgrade_conversion` first if the user changes their plan selection, then call `prepare` again. This is not enforced in the RPC — it's the caller's responsibility.

### 5. `apply` with `converted_credits=0`

If `prepare` found zero open lots, `apply` will write a `credit_transaction` with `amount=0`. This is a valid (though semantically empty) ledger entry. The `remaining_credits` and `credit_balance` are unchanged. The conversion status is still set to `applied` and Stripe references are recorded.

### 6. `remaining_credits` CHECK constraint on `reverse`

The `pack_credit_lot` table has `CHECK (remaining_credits >= 0 AND remaining_credits <= original_credits)`. After `apply` consumes credits, `remaining_credits = 0`. After `reverse` restores them, `remaining_credits = original_reserved`. If the account was partially drained by other unlocks between prepare and apply (impossible — the spendable check prevents it), the restored value could exceed `original_credits`. This is an invariant violation that the CHECK constraint will surface as a DB error, which is the correct behavior (fail loudly).

### 7. `link` does not validate conversion_id owner

`link_subscription_upgrade_checkout` does not verify that the conversion belongs to the caller's account. It only checks `status = 'pending'`. This is acceptable because: (a) the RPC is `SECURITY DEFINER` and callable only by `service_role`, and (b) `conversion_id` is a UUID generated by `prepare`, which is already account-scoped. The billing service is responsible for passing the correct `conversion_id`.
