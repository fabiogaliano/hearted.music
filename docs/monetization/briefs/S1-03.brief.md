# Implementation Brief: S1-03 Event/Idempotency Tables + Measurement Columns

## Goal

Create the idempotency and event-tracking tables (`billing_webhook_event`, `billing_activation`, `billing_bridge_event`) and add measurement columns to `song_analysis`.

## Current State

**Dependencies satisfied:** S1-01 and S1-02 are implemented in existing migrations:
- `supabase/migrations/20260405100000_billing_core_tables.sql` (S1-01 core tables)
- `supabase/migrations/20260405110000_billing_pack_conversion_tables.sql` (S1-02 pack/conversion tables)

**Current song_analysis table structure:**
```sql
CREATE TABLE song_analysis (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  song_id UUID NOT NULL REFERENCES song(id) ON DELETE CASCADE,
  analysis JSONB NOT NULL,
  model TEXT NOT NULL,
  prompt_version TEXT,
  tokens_used INTEGER,      -- existing measurement column
  cost_cents INTEGER,       -- existing measurement column
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT now() NOT NULL
);
```

**Existing RLS infrastructure:**
- `song_analysis` already has RLS enabled with deny-all policy: `"song_analysis_deny_all"`
- `update_updated_at_column()` function exists for auto-updating timestamps
- Consistent RLS pattern: `ENABLE ROW LEVEL SECURITY` + deny-all policies

**Migration naming pattern:** `{YYYYMMDDHHMMSS}_{descriptive_name}.sql`

**Target tables missing:** All three event/idempotency tables need to be created.

## Changes Required

Create a new migration file: `supabase/migrations/20260405120000_billing_event_tables_and_measurement_cols.sql` containing:

### 1. `billing_webhook_event` table

```sql
CREATE TABLE billing_webhook_event (
  stripe_event_id TEXT PRIMARY KEY,
  status         TEXT NOT NULL 
                 CHECK (status IN ('processing', 'processed', 'failed')),
  processed_at   TIMESTAMPTZ,
  error_message  TEXT,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_billing_webhook_event_status_created 
  ON billing_webhook_event(status, created_at);
```

### 2. `billing_activation` table

```sql
CREATE TABLE billing_activation (
  id                         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id                 UUID NOT NULL REFERENCES account(id) ON DELETE CASCADE,
  kind                       TEXT NOT NULL 
                             CHECK (kind IN ('unlimited_period_activated')),
  stripe_subscription_id     TEXT NOT NULL,
  subscription_period_end    TIMESTAMPTZ NOT NULL,
  stripe_event_id           TEXT NOT NULL,
  created_at                TIMESTAMPTZ NOT NULL DEFAULT now(),

  UNIQUE(account_id, kind, stripe_subscription_id, subscription_period_end)
);

CREATE INDEX idx_billing_activation_account 
  ON billing_activation(account_id, created_at);
```

### 3. `billing_bridge_event` table

```sql
CREATE TABLE billing_bridge_event (
  stripe_event_id TEXT PRIMARY KEY,
  event_kind      TEXT NOT NULL 
                  CHECK (event_kind IN (
                    'pack_fulfilled',
                    'unlimited_activated', 
                    'pack_reversed',
                    'unlimited_period_reversed',
                    'subscription_deactivated'
                  )),
  processed_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_billing_bridge_event_kind_processed 
  ON billing_bridge_event(event_kind, processed_at);
```

### 4. RLS policies for event tables

```sql
-- Enable RLS (service_role bypasses)
ALTER TABLE billing_webhook_event ENABLE ROW LEVEL SECURITY;
ALTER TABLE billing_activation ENABLE ROW LEVEL SECURITY;
ALTER TABLE billing_bridge_event ENABLE ROW LEVEL SECURITY;

-- RLS policies (deny direct anon/authenticated access, managed via service_role)
CREATE POLICY "billing_webhook_event_deny_all" ON billing_webhook_event FOR ALL USING (false);
CREATE POLICY "billing_activation_deny_all" ON billing_activation FOR ALL USING (false);
CREATE POLICY "billing_bridge_event_deny_all" ON billing_bridge_event FOR ALL USING (false);
```

### 5. Updated timestamp trigger for billing_webhook_event

```sql
-- Auto-update updated_at timestamp for billing_webhook_event (has updated_at column)
CREATE TRIGGER billing_webhook_event_updated_at
  BEFORE UPDATE ON billing_webhook_event
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();
```

### 6. Add measurement columns to song_analysis

```sql
-- Add measurement columns to song_analysis (all nullable, IF NOT EXISTS)
ALTER TABLE song_analysis 
  ADD COLUMN IF NOT EXISTS provider TEXT,
  ADD COLUMN IF NOT EXISTS input_tokens INTEGER,
  ADD COLUMN IF NOT EXISTS output_tokens INTEGER,
  ADD COLUMN IF NOT EXISTS cost_usd NUMERIC(10, 8);
```

## Constraints from DECISIONS.md and TERMINOLOGY.md

**Frozen constraint values:**
- `billing_webhook_event.status` values: `'processing', 'processed', 'failed'` (frozen in TERMINOLOGY.md §5)
- `billing_activation.kind` values: `'unlimited_period_activated'` (frozen in TERMINOLOGY.md §5)
- `billing_bridge_event.event_kind` values: `'pack_fulfilled', 'unlimited_activated', 'pack_reversed', 'unlimited_period_reversed', 'subscription_deactivated'` (frozen in TERMINOLOGY.md §5)
- Table and column names exactly as specified in V2 plan (frozen)

**Business rules:**
- `billing_activation` NOT NULL on `stripe_subscription_id` and `subscription_period_end` (PostgreSQL NULL-distinctness in UNIQUE)
- `billing_webhook_event` and `billing_bridge_event` use `stripe_event_id` as PK for idempotency
- `song_analysis` measurement columns are for operational COGS only; billing must never depend on them

**Architectural invariants:**
- RLS enabled with deny-all policies (service_role bypasses for RPCs)
- Use `IF NOT EXISTS` for song_analysis columns to prevent migration failures on re-runs

## Risks

**Low risk:** The schema is fully specified in the V2 plan with exact DDL. Foreign key targets exist from S1-01. The RLS pattern is established. The `IF NOT EXISTS` clause prevents issues with existing song_analysis columns.

**Potential issue:** The existing `song_analysis` table has `tokens_used` and `cost_cents` columns that overlap conceptually with the new measurement columns, but the story explicitly requires the new columns for operational COGS visibility.
