export const BillingChanges = {

    accountId: string,

### 3. Create `src/lib/domains/billing/bridge-events.ts` (new file)

```typescript
/**
 * Typed payloads for billing-service -> app bridge calls.
 *
 * These shapes are the stable interface between v1_hearted_brand (billing service)
 * and v1_hearted (app). No Stripe-specific types may appear here.
 *
 * Consumed by the bridge ingress handler (Phase 4).
 */

/** Payload for a pack fulfillment bridge call. */
export interface PackFulfilledBridgePayload {
  /** Stripe event ID — used as bridge idempotency key. */
  stripeEventId: string;
  accountId: string;
  /** Song IDs that were unlocked as pack bonus unlocks. */
  bonusUnlockedSongIds: string[];
}

/** Payload for an unlimited access activation bridge call. */
export interface UnlimitedActivatedBridgePayload {
  /** Stripe event ID — used as bridge idempotency key. */
  stripeEventId: string;
  accountId: string;
}

/** Payload for an access revocation outcome bridge call. */
export interface RevocationOutcomeBridgePayload {
  /** Stripe event ID — used as bridge idempotency key. */
  stripeEventId: string;
  accountId: string;
  /**
   * The event kind that produced this revocation.
   * Maps to billing_bridge_event.event_kind frozen values.
   */
  eventKind:
    | "pack_reversed"
    | "unlimited_period_reversed"
    | "subscription_deactivated";
}
```

## Constraints

From DECISIONS.md §2 (Control-plane responsibility) and TERMINOLOGY.md §5:

- **Change kind names are frozen**: `songs_unlocked`, `unlimited_activated`, `candidate_access_revoked`
- **The control plane never learns why access changed, only that it changed** — `candidate_access_revoked` carries no song IDs or refund reasons; `unlimited_activated` carries no Stripe subscription ID
- **`songs_unlocked` carries song IDs**; `unlimited_activated` is account-wide; `candidate_access_revoked` does not carry song IDs
- **No Stripe-specific types in `library-processing` code** — `bridge-events.ts` lives in `domains/billing/`, not in `library-processing/`; the billing change helper (`changes/billing.ts`) imports only from `../types`
- **No barrel exports** (CLAUDE.md) — no `index.ts`; consumers import directly from specific file paths
- **No `any`, no non-null assertions** (AGENTS.md)
- `candidate_access_revoked` must NOT include refund/chargeback reasons or song IDs — doing so would cross the control-plane boundary

## Risks

1. **S2-04 coordination**: Both S2-04 and S2-05 may touch `types.ts`. Each adds distinct, non-overlapping union variants, so sequential application is safe — but if run in parallel the edits must be merged without conflict.

2. **`RevocationOutcomeBridgePayload.eventKind` values**: The three revocation event kinds (`pack_reversed`, `unlimited_period_reversed`, `subscription_deactivated`) are drawn from the frozen `billing_bridge_event.event_kind` CHECK constraint values (TERMINOLOGY.md §5). `unlimited_activated` is excluded because it maps to `UnlimitedActivatedBridgePayload`. This split is correct.

3. **No existing consumers of `applyLibraryProcessingChange` handle the new variants**: The reconciler will receive them but has no cases for them yet — Phase 3 wires the reconciler (S3-06). Until then, unhandled variants in the reconciler's switch/match will surface as TypeScript exhaustiveness errors if the reconciler uses exhaustive checking. Verify the reconciler doesn't break after adding the new variants.
