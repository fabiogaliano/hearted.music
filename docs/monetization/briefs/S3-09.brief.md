## Changes Required

### 1. New migration: `supabase/migrations/20260405220000_billing_aware_dashboard_stats.sql`

Replace `count_analyzed_songs_for_account` with an entitlement-aware version using the `billing_facts` CTE pattern:

```sql
-- S3-09: Billing-aware dashboard stats
-- Replaces count_analyzed_songs_for_account with entitlement-gated version.
-- Entitled = active unlock row OR unlimited access (subscription active OR self_hosted).

CREATE OR REPLACE FUNCTION count_analyzed_songs_for_account(p_account_id UUID)
RETURNS BIGINT
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  WITH billing_facts AS (
    SELECT COALESCE(
      (
        SELECT
          unlimited_access_source IS NOT NULL
          AND (
            unlimited_access_source = 'self_hosted'
            OR (
              unlimited_access_source = 'subscription'
              AND subscription_status = 'active'
            )
          )
        FROM account_billing
        WHERE account_id = p_account_id
      ),
      false
    ) AS has_unlimited_access
  )
  SELECT COUNT(DISTINCT ls.song_id)
  FROM liked_song ls
  CROSS JOIN billing_facts bf
  WHERE ls.account_id = p_account_id
    AND ls.unliked_at IS NULL
    -- Must have analysis
    AND EXISTS (SELECT 1 FROM song_analysis sa WHERE sa.song_id = ls.song_id)
    -- Must be entitled
    AND (
      bf.has_unlimited_access
      OR EXISTS (
        SELECT 1 FROM account_song_unlock asu
        WHERE asu.account_id = p_account_id
          AND asu.song_id = ls.song_id
          AND asu.revoked_at IS NULL
      )
    );
$$;
```

This is a `CREATE OR REPLACE` on the existing function name — no type signature change, so `database.types.ts` does not need updating.

### 2. Update `fetchMatchPreviews` in `src/lib/server/dashboard.functions.ts`

Filter `undecided` songs to only those the account is entitled to. The entitlement check must be performed at read time against `account_song_unlock` and `account_billing` tables.

**Approach:** After `getUndecidedSongs` returns, fetch entitled song IDs for the account from the snapshot candidates, then filter undecided to the intersection. Use the existing `select_entitled_data_enriched_liked_song_ids` RPC (from `20260405190000_billing_aware_selectors.sql`) — it returns liked songs that are both data-enriched AND entitled.

Add a new query helper to `src/lib/domains/taste/song-matching/queries.ts` (or `dashboard.functions.ts` inline via supabase RPC call):

```typescript
// In fetchMatchPreviews, after getting undecided songs:
const supabase = createAdminSupabaseClient();

// Get entitled song IDs for this account
const { data: entitledRows, error: entitledError } = await supabase.rpc(
  "select_entitled_data_enriched_liked_song_ids",
  { p_account_id: accountId },
);

const entitledSet = new Set(
  (!entitledError && entitledRows ? entitledRows : []).map(
    (r: { song_id: string }) => r.song_id
  )
);

// Filter undecided to entitled only
const entitledUndecided = undecided.filter((s) => entitledSet.has(s.songId));
```

Then sort and slice `entitledUndecided` instead of `undecided`.

**Full updated `fetchMatchPreviews`:**

```typescript
async function fetchMatchPreviews(accountId: string): Promise<MatchPreview[]> {
  const snapshotResult = await getLatestMatchSnapshot(accountId);
  if (Result.isError(snapshotResult) || !snapshotResult.value) return [];

  const supabase = createAdminSupabaseClient();

  const [undecided, newSongIds, entitledResult] = await Promise.all([
    getUndecidedSongs(snapshotResult.value.id, accountId),
    getNewItemIds(accountId, "song"),
    supabase.rpc("select_entitled_data_enriched_liked_song_ids", {
      p_account_id: accountId,
    }),
  ]);

  if (Result.isError(newSongIds) || undecided.length === 0) return [];

  const entitledSet = new Set(
    (!entitledResult.error && entitledResult.data
      ? entitledResult.data
      : []
    ).map((r: { song_id: string }) => r.song_id),
  );

  const newSet = new Set(newSongIds.value);
  const sorted = undecided
    .filter((s) => entitledSet.has(s.songId))
    .toSorted((a, b) => {
      const aNew = newSet.has(a.songId) ? 1 : 0;
      const bNew = newSet.has(b.songId) ? 1 : 0;
      if (aNew !== bNew) return bNew - aNew;
      if (b.maxScore !== a.maxScore) return b.maxScore - a.maxScore;
      return a.songId.localeCompare(b.songId);
    });

  const topIds = sorted.slice(0, 3).map((s) => s.songId);

  const { data, error } = await supabase
    .from("song")
    .select("id, image_url")
    .in("id", topIds);

  if (error || !data) return [];

  const imageMap = new Map(data.map((s) => [s.id, s.image_url]));
  return topIds
    .map((id, i) => {
      const image = imageMap.get(id);
      return image ? { id: i + 1, image } : null;
    })
    .filter((p): p is MatchPreview => p !== null);
}
```

## Constraints

From **DECISIONS.md §4 (Read-Model Invariants)**:

> **Shared artifacts do not imply entitlement**
> - `song_analysis` rows are global (not account-scoped); existence does not mean the account may see that analysis
> - Read models must filter by effective entitlement, not by `song_analysis` or `item_status` existence
> - **Dashboard stats (`analyzedPercent`, etc.) must be billing-aware**

> **Effective entitlement predicate** (§2): A song is entitled when either an `account_song_unlock` row exists with `revoked_at IS NULL` OR the account has active unlimited access.
> `past_due` and `unpaid` statuses do NOT grant unlimited access.

From **DECISIONS.md §3 (Data Model Invariants)**:

> When `BILLING_ENABLED=false`, account provisioning grants explicit `self_hosted` unlimited access via `unlimited_access_source = 'self_hosted'`. SQL never infers deployment mode from missing rows.

This means **self-hosted users have `unlimited_access_source = 'self_hosted'`** in their `account_billing` row — the entitlement predicate handles them transparently. No special-case code is needed for self-hosted; the updated RPC already returns the correct count via `has_unlimited_access = true`.

**Frozen terminology:** `is_account_song_entitled`, `account_song_unlock`, `account_billing`, `unlimited_access_source`, `subscription_status`.

## Risks

1. **`select_entitled_data_enriched_liked_song_ids` requires all 4 artifacts** (audio features, genres, analysis, embedding). A song that is entitled and analyzed but lacks audio features or genres would be excluded from match previews even if it should appear. This is consistent with the selector's existing semantics (it's the match refresh candidate selector), but may be more conservative than needed for dashboard previews. **Mitigation:** If this is too conservative, a simpler entitlement-only filter can be added as a new RPC, but using the existing selector is consistent with current architecture.

2. **`database.types.ts` is generated** — the `CREATE OR REPLACE` keeps the same signature (`Args: { p_account_id: string }, Returns: number`), so no manual type update is needed. If a future `supabase gen types` run regenerates the file, the type remains valid.

3. **`fetchMatchPreviews` now makes 3 parallel calls** instead of 2. The added `select_entitled_data_enriched_liked_song_ids` call is `STABLE` (read-only), so it's safe to parallelize.

4. **No existing test for `fetchDashboardStats`** — the `src/lib/server/__tests__/` directory contains tests for jobs and playlists but not dashboard. A new test file is required per acceptance criteria.
