## Goal

Replace the hardcoded `resolveQueuePriority()` stub with a real implementation that derives the queue band from `BillingState`, and update the call site in `service.ts` to pass billing state.

## Changes Required

### 1. Update `src/lib/workflows/library-processing/queue-priority.ts`

Replace the async stub with a sync function that accepts `BillingState` and returns `state.queueBand`.

**Replace:**
```typescript
export async function resolveQueuePriority(
  _accountId: string,
): Promise<QueueBand> {
  // No entitlement table yet — default to free/baseline band
  return "low";
}
```

**With:**
```typescript
import type { BillingState } from "@/lib/domains/billing/state";

/**
 * Derives the queue band from resolved billing state.
 * Billing domain owns the band-derivation logic (BillingState.queueBand);
 * this is the scheduler-facing adapter that reads it.
 */
export function resolveQueuePriority(state: BillingState): QueueBand {
  return state.queueBand;
}
```

- Signature change: `async (accountId: string) => Promise<QueueBand>` → `(state: BillingState) => QueueBand`
- No longer async — `BillingState` is already resolved by the caller
- Remains a thin adapter; band logic lives in billing domain

### 2. Update `src/lib/workflows/library-processing/service.ts`

**Add import** for billing query and client (client already imported):
```typescript
import { readBillingState } from "@/lib/domains/billing/queries";
import type { BillingState } from "@/lib/domains/billing/state";
```

**Replace the call site in `executeEffect`:**

Old:
```typescript
const band = await resolveQueuePriority(effect.accountId);
const queuePriority = bandToNumeric(band);
```

New:
```typescript
const supabase = createAdminSupabaseClient();
const billingResult = await readBillingState(supabase, effect.accountId);
const band = resolveQueuePriority(
  Result.isOk(billingResult) ? billingResult.value : FREE_DEFAULT_BILLING_STATE,
);
const queuePriority = bandToNumeric(band);
```

The `FREE_DEFAULT_BILLING_STATE` fallback ensures the function degrades to `"low"` (same as current behaviour) if the billing read fails, without throwing or disrupting the effect execution path.

Define the fallback constant at module scope in `service.ts` (or import it from `queries.ts` if S2-02 exports it — check whether `FREE_DEFAULT` / equivalent is exported from `queries.ts`):

```typescript
import type { BillingState } from "@/lib/domains/billing/state";

const FREE_DEFAULT_BILLING_STATE: BillingState = {
  plan: "free",
  creditBalance: 0,
  subscriptionStatus: "none",
  cancelAtPeriodEnd: false,
  unlimitedAccess: { kind: "none" },
  queueBand: "low",
};
```

> **Note:** If S2-02 exports a `FREE_DEFAULT` constant from `queries.ts`, import it instead of re-declaring. Check at implementation time. If not exported, declare locally in `service.ts`.

## Constraints

From DECISIONS.md §1 (Queue bands — frozen):

| Account state | Band |
|---|---|
| Free, no purchased balance | `low` |
| Any non-unlimited account with positive purchased balance | `standard` |
| Quarterly unlimited | `standard` |
| Yearly unlimited | `priority` |
| `self_hosted` unlimited | `priority` |

From DECISIONS.md §2 (Architecture):
> "queue-priority.ts remains a thin adapter — plan/offer mapping stays in billing domain"

From DECISIONS.md §6 (Terminology — frozen):
- `QueueBand`: `"low" | "standard" | "priority"` — frozen values
- TS type `QueueBand` is frozen in this file

From AGENTS.md:
- No `any`, no non-null assertions, no unsafe type casts
- Errors as values — propagate `Result` failures explicitly, do not swallow

The band mapping itself is owned entirely by `BillingState.queueBand` (computed in S2-02 `readBillingState`). This story does **not** re-implement the mapping — it reads the pre-computed value.

## Risks

1. **S2-02 may or may not export `FREE_DEFAULT`**: The S2-02 brief defines the constant internally. If it's not exported, declare it locally in `service.ts`. Check the actual `queries.ts` file at implementation time — do not duplicate if it's already exported.

2. **`createAdminSupabaseClient()` called per effect**: `executeEffect` is called in a loop in `applyLibraryProcessingChange`. Creating a new Supabase client per call is slightly wasteful. The existing code already calls `createAdminSupabaseClient()` inside `executeEffect` indirectly (for target-song enrichment check), so this is consistent with the existing pattern and acceptable for v1.

3. **Dependency ordering**: This story cannot compile until S2-01 (`state.ts`) and S2-02 (`queries.ts`) are merged. The `import type { BillingState }` and `import { readBillingState }` will fail at typecheck until those files exist.

4. **`resolveQueuePriority` signature change is breaking**: Any other callers of `resolveQueuePriority` (if any exist beyond `service.ts`) must be updated. Current grep shows only one call site in `service.ts`. Verify with `grep -r "resolveQueuePriority"` before implementing.
