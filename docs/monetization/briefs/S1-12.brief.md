## Risks and Ambiguities

### 1. Types were more out-of-date than the story expected

The story assumed only S1-06 RPCs needed adding. In practice, the file was missing 14 RPCs
from S1-05 through S1-11. This is because `gen:types` was last run only after S1-01 through
S1-03 were applied locally. The regeneration resolved all gaps at once — no incremental risk.

### 2. `fulfill_pack_purchase` and `reverse_pack_entitlement` return `Json` not a typed record

The Supabase type generator maps PostgreSQL `RETURNS JSONB` to `Json` (the generic union
type). Phase 2 TypeScript callers will need to runtime-narrow the JSON payloads from these
functions. This is standard Supabase behavior for JSON-returning functions and not a defect
in the generated types.

### 3. Pre-existing typecheck failures are not this story's responsibility

`v1_hearted_brand/tools/pricing-calculator/` has 7 pre-existing TypeScript errors from
Recharts type API changes. These predate this story and are out of scope per the story's
constraint ("fix any compile errors in existing code caused by the type changes"). The errors
are not caused by the billing type changes.

### 4. `gen:types` is idempotent

Re-running `bun run gen:types` any number of times produces the same output as long as the
local Supabase schema is unchanged. Callers should re-run whenever new migrations are applied.
