## Changes Required

### 1. `src/lib/workflows/library-processing/reconciler.ts` — add three cases

Insert after the `playlist_management_session_flushed` case, before the closing `}` of the switch:

```typescript
case "songs_unlocked": {
  // Newly unlocked songs need enrichment; they may also become candidates
  enrichment = advanceRequestedAt(enrichment, requestMarker);
  if (hasTargetPlaylists) {
    matchSnapshotRefresh = advanceRequestedAt(matchSnapshotRefresh, requestMarker);
  }
  break;
}

case "unlimited_activated": {
  // Full-library enrichment required; candidate set expands account-wide
  enrichment = advanceRequestedAt(enrichment, requestMarker);
  if (hasTargetPlaylists) {
    matchSnapshotRefresh = advanceRequestedAt(matchSnapshotRefresh, requestMarker);
  }
  break;
}

case "candidate_access_revoked": {
  // Candidate set shrank — refresh snapshot only, no new enrichment work
  matchSnapshotRefresh = advanceRequestedAt(matchSnapshotRefresh, requestMarker);
  break;
}
```

**Rationale:**
- `songs_unlocked`: unlocked songs are now eligible for Phase B/C enrichment → advance enrichment. If targets exist, these songs may become candidates → advance match refresh.
- `unlimited_activated`: all liked songs are now eligible for full processing → advance enrichment. Candidate set expands → advance match refresh if targets exist.
- `candidate_access_revoked`: no enrichment needed (access shrank, not grew). Snapshot is stale because revoked songs must be removed from match results → advance match refresh unconditionally (revocation is account-wide; stale snapshot must be refreshed regardless of whether targets are currently configured).

Note on `candidate_access_revoked` not gating on `hasTargetPlaylists`: unlike enrichment-driven refresh, a revocation makes existing snapshot contents wrong regardless of target state. This is distinct from `library_synced` liked-songs-removed which does gate on hasTargetPlaylists — that case only matters if there are playlists to match against. Revocation must flush stale results unconditionally.

### 2. `src/lib/workflows/library-processing/__tests__/reconciler.test.ts` — add billing describe block

Add `describe("billing changes", ...)` after the existing describe blocks, following the exact test patterns already in the file:

**`songs_unlocked`:**
- Advances enrichment regardless of `hasTargetPlaylists`

- Does NOT advance enrichment (`state.enrichment.requestedAt` stays null)

- Emits `ensure_match_snapshot_refresh_job`
- Emits no `ensure_enrichment_job`

No Stripe-specific logic may appear in `reconciler.ts` or `service.ts`.

## Risks

1. **`candidate_access_revoked` refresh gating**: The story says "refresh only" without specifying whether `hasTargetPlaylists` gates it. The brief recommends unconditional (no gate) because revocation makes existing snapshots wrong regardless of target state. If the implementation agent chooses to gate it, tests must document and verify the chosen behavior.

2. **Silent no-ops before this story**: The three billing variants currently fall through the switch silently. No TypeScript error surfaces because the switch has no exhaustive `default` check. This is safe pre-Phase-4 but must be resolved before bridge wiring.

3. **Exhaustiveness**: Consider adding `change satisfies never` in a `default` branch after all cases. This would catch future unhandled variants at compile time and aligns with "make illegal states unrepresentable." Only add if the current union is fully covered (all ten variants have explicit cases after this story).
