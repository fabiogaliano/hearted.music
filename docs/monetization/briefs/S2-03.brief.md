## Changes Required

### Modify `src/lib/domains/library/accounts/queries.ts`

1. **Add import** for `env` from `@/env` (needed to read `BILLING_ENABLED`).
2. **Add import** for `Result` value (not just type) from `"better-result"`.
3. **Add import** for `DatabaseError` from `@/lib/shared/errors/database`.
4. **Replace `createAccountForBetterAuthUser`** with an updated version that:
   - Inserts the `account` row (same as before), returning `Result<Account, DbError>`
   - On success, inserts an `account_billing` row with `ON CONFLICT DO NOTHING`
   - When `BILLING_ENABLED=false`: sets `unlimited_access_source = 'self_hosted'` on the billing row
   - When `BILLING_ENABLED=true`: inserts with defaults only (`plan='free'`, `credit_balance=0`, `unlimited_access_source=null`)
   - After the self-hosted billing write, calls `reprioritize_pending_jobs_for_account` via Supabase RPC
   - Returns the original `Account` from the first insert (return type unchanged)

The updated function body:

```typescript
export async function createAccountForBetterAuthUser(
  data: CreateBetterAuthAccountData,
): Promise<Result<Account, DbError>> {
  const supabase = createAdminSupabaseClient();

  // Insert the account row
  const accountResult = await fromSupabaseSingle(
    supabase
      .from("account")
      .insert({
        better_auth_user_id: data.better_auth_user_id,
        email: data.email,
        display_name: data.display_name,
      })
      .select()
      .single(),
  );

  if (Result.isError(accountResult)) return accountResult;

  const account = accountResult.value;

  // Always provision an account_billing row. ON CONFLICT DO NOTHING for idempotency.
  const billingInsert = env.BILLING_ENABLED
    ? { account_id: account.id }
    : { account_id: account.id, unlimited_access_source: "self_hosted" as const };

  const { error: billingError } = await supabase
    .from("account_billing")
    .insert(billingInsert)
    .select()
    .single();

  // Ignore conflict (23505) — row already exists, invariant preserved
  if (billingError && billingError.code !== "23505") {
    return Result.err(
      new DatabaseError({
        code: billingError.code,
        message: `Failed to provision account_billing for account ${account.id}: ${billingError.message}`,
      }),
    );
  }

  // In self-hosted mode, reprioritize any already-pending jobs for this account
  if (!env.BILLING_ENABLED) {
    await supabase.rpc("reprioritize_pending_jobs_for_account", {
      p_account_id: account.id,
    });
  }

  return Result.ok(account);
}
```

**Note on `ON CONFLICT DO NOTHING` via Supabase client:** The Supabase JS client does not expose `ON CONFLICT DO NOTHING` directly on `insert()`. The equivalent is to catch the unique-constraint error code `23505` and ignore it — which is the same pattern used in `src/lib/domains/billing/queries.ts` for the self-healing insert. Use the same approach here.

**Note on `billingInsert` typing:** `unlimited_access_source` in `TablesInsert<"account_billing">` is typed as `string | null | undefined`. Casting with `as const` on the string literal is sufficient; no `any` is required.

## Constraints

From DECISIONS.md and TERMINOLOGY.md that apply:

- **Every account gets an `account_billing` row in every deployment; missing row is an invariant violation** (DECISIONS.md §3)
- **`self_hosted` is orthogonal to `plan`; those accounts keep `plan='free'` and `credit_balance=0`** (DECISIONS.md §3) — so the self-hosted insert only sets `unlimited_access_source`, relies on DB column defaults for `plan` and `credit_balance`
- **SQL never infers deployment mode from missing rows** (DECISIONS.md §3)
- **Provisioning must be idempotent** (story scope) — use `ON CONFLICT DO NOTHING` semantics (catch `23505`)
- **Reprioritize after self-hosted billing write** (DECISIONS.md §5): "App-layer `self_hosted` provisioning calls the reprioritization RPC immediately after the billing write"
- **No `any`, no non-null assertions** (AGENTS.md)
- **Errors as values** (AGENTS.md) — propagate `DatabaseError` from billing insert failure; do not swallow
- **Unlock source values frozen**: `self_hosted` (TERMINOLOGY.md §5)

## Risks

1. **`insert().select().single()` vs bare `insert()`**: The Supabase client's `.insert()` without `.select()` returns `{ data: null, error }` when it succeeds. For the billing insert we don't need the returned row, but calling `.select().single()` causes a `PGRST116` not-found error on conflict if `ON CONFLICT DO NOTHING` silently suppresses the insert. The safer pattern (matching what `src/lib/domains/billing/queries.ts` already does) is to call `insert(...).select().single()` and treat error code `23505` as success — but that won't handle `DO NOTHING` cleanly either. **Use `insert()` without `.select()` and check for `error.code !== "23505"`**. Alternatively, use the `upsert` with `ignoreDuplicates: true` if the Supabase client version supports it. The existing `queries.ts` pattern catches `23505` explicitly on the plain `.insert().select().single()` path; that same pattern works here.

2. **`reprioritize_pending_jobs_for_account` failure is not fatal**: The story does not require the reprioritization call to block account creation. A newly created account will have no pending jobs, so the RPC will return 0 and is effectively a no-op at creation time. If the RPC call errors, it is safe to ignore — log if possible but don't fail the account creation. The implementation above uses `await` but ignores the return value; any RPC error will be swallowed at the `await` boundary. This is intentional (the account row is already written and committed).

3. **Import of `env` creates a module-level dependency**: `env` is already used in other files in `src/lib/domains/`. Adding it to `queries.ts` is consistent with the pattern and introduces no circular dependency.

4. **`Result` import**: The existing file imports `type { Result }` (type-only import). The updated function needs `Result.isError()` as a value, so the import must change from `import type { Result }` to `import { Result }`. Check the existing import style and update accordingly.
