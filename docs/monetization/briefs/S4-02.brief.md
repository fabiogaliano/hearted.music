**File: `v1_hearted_brand/src/middleware/hmac-auth.ts`**
```typescript
export function hmacAuth(secret: string): MiddlewareHandler {
	const encoder = new TextEncoder();
	const secretKey = encoder.encode(secret);

	return async (c, next) => {
		const timestamp = c.req.header("X-Timestamp");
		const signature = c.req.header("X-Signature");

		if (!timestamp || !signature) {
			return c.json({ error: "Missing authentication headers" }, 401);
		}

		const requestTime = Number(timestamp);
		if (Number.isNaN(requestTime)) {
			return c.json({ error: "Invalid timestamp" }, 401);
		}

		const now = Date.now() / 1000;
		if (Math.abs(now - requestTime) > CLOCK_SKEW_MS / 1000) {
			return c.json({ error: "Request timestamp expired" }, 401);
		}

		const body = await c.req.raw.clone().text();
		const bodyHash = toHex(
			await crypto.subtle.digest("SHA-256", encoder.encode(body)),
		);

		const message = `${timestamp}.${bodyHash}`;
		const key = await crypto.subtle.importKey(
			"raw",
			secretKey,
			{ name: "HMAC", hash: "SHA-256" },
			false,
			["sign"],
		);
		const expected = toHex(
			await crypto.subtle.sign("HMAC", key, encoder.encode(message)),
		);

		if (!timingSafeEqual(expected, signature)) {
			return c.json({ error: "Invalid signature" }, 401);
		}

		await next();
	};
}
```
- Validates HMAC signature format: `X-Signature: hex(HMAC-SHA256(secret, timestamp.bodyHash))`
- 5-minute clock-skew window enforced
- Timing-safe comparison prevents timing attacks

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### Offer ID Validation (TERMINOLOGY.md §2)
- **Canonical internal offer ID for pack:** `song_pack_500`
- S4-02 must validate that incoming `offer_id` equals exactly `"song_pack_500"`
- Reject with error if any other offer ID is sent

### Stripe Price ID Resolution (DECISIONS.md §2)
- **"Only the billing service resolves Stripe price IDs"**
- The billing service (not the app) must know the mapping from internal offer ID to Stripe price ID
- The app never sees Stripe price IDs; it sends stable offer IDs
- Stripe identifiers are opaque audit columns in shared schema

### Stripe Customer Reuse (DECISIONS.md §2)
- **"Stripe customer reuse prevents duplicate customers"**
- Before calling `checkout.sessions.create()`, check if `account_billing.stripe_customer_id` exists
- If exists, reuse it (pass `customer: stripe_customer_id` to Stripe)
- If null, let Stripe create a new customer (omit `customer` param, or set `customer_creation: "always"`)

### Checkout Idempotency (DECISIONS.md §2)
- **"`checkout_attempt_id` must be part of the signed request body"**
- Forward `checkout_attempt_id` as Stripe's `idempotency_key` on `checkout.sessions.create()`
- Idempotency key ensures duplicate requests with the same ID return the same session
- **"Reused across retries of same intent; new UUID if offer changes"** — app responsibility

### Request/Response Format (story scope)
- **Request body:** `{ account_id: UUID, offer_id: "song_pack_500", checkout_attempt_id: UUID }`
- **Response:** `{ checkout_url: string }`
- HMAC signature in headers (X-Timestamp, X-Signature) — already validated by middleware

### Metadata on Checkout Session (story scope)
- Set `metadata: { account_id }` on the created Checkout Session
- Metadata persists through Stripe and can be read in webhook payloads (S4-05 uses this)

### Frozen Env Var Names (TERMINOLOGY.md §6)
- `BILLING_SHARED_SECRET` — already validated in env.ts
- `STRIPE_SECRET_KEY` — already validated in env.ts
- `APP_BRIDGE_URL` — already validated (used by S4-05, not S4-02)
- None of these can be renamed

### Bridge Event Kinds (TERMINOLOGY.md §6, frozen for future context)
- Future S4-05 will emit `pack_fulfilled` bridge event to app
- S4-02 does not emit events; that's S4-05's responsibility

## Open Questions / Risks

### 1. Stripe Price ID Mapping
**Issue:** The story says "Resolve Stripe price ID from internal offer ID" but does not specify where the mapping is stored.

**Options:**
- Environment variable (e.g., `STRIPE_PRICE_ID_SONG_PACK_500`)
- Hardcoded constant in config (e.g., `const priceIds = { song_pack_500: "price_..." }`)
- Database lookup (query a `stripe_products` table, though not mentioned in schema)

**Current state:** No mapping exists in the codebase. Stripe product/price must be created manually in Stripe test dashboard beforehand (as listed in story dependencies).

**Risk:** If implementation agent hardcodes the price ID without clear constants, it becomes opaque and hard to update if Stripe product is recreated. If using env vars, must match deployment config.

### 2. Stripe Checkout Session Mode
**Issue:** Story mentions "Stripe test-mode product" but doesn't specify whether `checkout.sessions.create()` should be in `mode: "payment"` or `mode: "subscription"`.

**Assumption:** Song pack is a one-time purchase, so mode should be `"payment"` (not `"subscription"`).

**Risk:** Calling with wrong mode will fail or create wrong checkout type.

### 3. Supabase Lookup of Account Billing
**Issue:** Story says "Reuse `account_billing.stripe_customer_id` if exists" but endpoint only receives `account_id`.

**Implementation requirement:** Must query Supabase to fetch `account_billing` row by `account_id` to check if `stripe_customer_id` exists. If null, omit customer from Stripe call (Stripe will auto-create).

**Current pattern:** S4-01 provides `supabase` client global; no RPC or helper function yet for this lookup. Implementation must write the query.

### 4. Error Handling for Invalid Offer ID
**Issue:** Story says "Rejects invalid offer IDs" but does not specify response format.

**Options:**
- 400 with JSON error body (consistent with middleware errors)
- 422 "Unprocessable Entity"
- Different status code for different error types (validation vs server errors)

**Current pattern:** HMAC auth returns 401 with `{ error: string }` format.

**Risk:** App-side (S4-10) expects consistent error responses. Format must be documented or inferred from other endpoints.

### 5. Stripe Checkout Return URL
**Issue:** Story does not specify where user is redirected after Stripe payment.

**Context:** S4-02 creates the session; S4-05 receives the webhook. But the redirect URL must be passed to `checkout.sessions.create()` via `success_url` and `cancel_url`. These are typically set to app URLs.

**Unclear:** Should these URLs be:
- Hardcoded in billing service (pointing to production/staging app)?
- Passed in the request body from the app?
- Environment variables?

**Current state:** Not mentioned in story scope or acceptance criteria. Likely handled in implementation based on standard Stripe patterns.

### 6. Stripe Customer Creation Fallback
**Issue:** If Stripe customer creation fails (API error, quota exceeded), how should the endpoint respond?

**Options:**
- Return error response with failure reason
- Fail the entire checkout attempt
- Attempt retry logic

**Current pattern:** No error handling pattern established yet in S4-01.

### 7. App-Side Retry Behavior
**Issue:** Story says "`checkout_attempt_id` reused across retries of the same intent" (app responsibility), but implementation must handle it correctly.

**Example:** If app sends the same `checkout_attempt_id` twice, Stripe idempotency key will return the same session. The endpoint must return the same `checkout_url` both times.

**Risk:** If implementation caches by `checkout_attempt_id` on the service side and that cache is cleared, behavior could be inconsistent.

### 8. No Explicit Test Mode Flag
**Issue:** Stripe SDK can work with test mode keys automatically, but no explicit test/prod mode flag is set in env.

**Current state:** If `STRIPE_SECRET_KEY` is a test key, everything works in test mode. If it's a live key, requests go to production. No safeguard against accidental live requests.

**Acceptable for now:** Trust that deployment configs separate test and live API keys by environment.

### 9. Database Service Role Permissions
**Issue:** Service-role Supabase client is unfiltered by RLS. Story doesn't explicitly verify that RLS policies on `account_billing` allow service-role reads.

**Current schema:** RLS policy on `account_billing`: `CREATE POLICY "account_billing_deny_all" ON account_billing FOR ALL USING (false);` — but service role always bypasses RLS.

**Safe:** Service role can read/write any table. No risk here.

### 10. Timing of Stripe Customer Lookup vs Creation
**Issue:** If multiple requests for the same account arrive in parallel before the first one has created and written the `stripe_customer_id`, race condition could result in duplicate Stripe customers.

**Current state:** No explicit locking mentioned. `stripe_customer_id` has a UNIQUE constraint, so one of the INSERTs will fail. Question: which request wins, and what does the loser do?

**Typical pattern:** Use Supabase `INSERT ... ON CONFLICT ... DO UPDATE` or re-fetch after failure to get the winner's ID. Not specified in story.

## Summary

S4-02 is the first external-facing endpoint of the billing service. It receives HMAC-signed requests from the app, validates the offer ID, looks up or creates a Stripe customer, and creates a Checkout Session. The main unknowns are:

1. **Where is the Stripe price ID mapping?** (env var, constant, or TBD)
2. **How are errors structured?** (consistent with S4-01 or different?)
3. **How are success/cancel URLs provided?** (hardcoded, env, or request param?)
4. **How is race condition on customer creation handled?** (implicit via Stripe idempotency or explicit locking?)

All can be resolved by implementation agent using standard Stripe + Hono patterns; none are blockers.
