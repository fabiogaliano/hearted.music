## Changes Required

### 1. Create `src/lib/domains/billing/queries.ts`

Implement `readBillingState(supabase, accountId)`:

```typescript
import { Result } from "better-result";
import { DatabaseError, type DbError } from "@/lib/shared/errors/database";
import type { AdminSupabaseClient } from "@/lib/data/client";
import type { BillingState, UnlimitedAccess } from "@/lib/domains/billing/state";
import { STRIPE_STATUS_TO_NORMALIZED } from "@/lib/domains/billing/state";

/**
 * Reads and normalizes billing state for an account.
 *
 * Self-healing: if no account_billing row exists (invariant violation),
 * inserts a default free row via INSERT ... ON CONFLICT DO NOTHING, then
 * returns the free default. The insert is best-effort; if it races with
 * a concurrent insert the CONFLICT means we just return the default.
 */
export async function readBillingState(
  supabase: AdminSupabaseClient,
  accountId: string,
): Promise<Result<BillingState, DbError>> {
  // ...
}
```

**Normalization logic:**
1. `SELECT * FROM account_billing WHERE account_id = accountId`
2. If no row found: `INSERT INTO account_billing (account_id) VALUES (accountId) ON CONFLICT DO NOTHING` ‚Üí return free default `BillingState`
3. Map raw row ‚Üí `BillingState`:
   - `plan` ‚Üí cast to `BillingPlan` via `STRIPE_STATUS_TO_NORMALIZED` map pattern (known safe values from CHECK constraint)
   - `creditBalance` ‚Üê `credit_balance`
   - `subscriptionStatus` ‚Üê `STRIPE_STATUS_TO_NORMALIZED[row.subscription_status] ?? "none"`, then if result is `"active"` and `cancel_at_period_end === true` ‚Üí override to `"ending"`
   - `cancelAtPeriodEnd` ‚Üê `cancel_at_period_end`
   - `unlimitedAccess` ‚Üê derive from `unlimited_access_source`:
     - `null` ‚Üí `{ kind: "none" }`
     - `"subscription"` ‚Üí `{ kind: "subscription" }`
     - `"self_hosted"` ‚Üí `{ kind: "self_hosted" }`
     - any other string (defensive) ‚Üí `{ kind: "none" }`
   - `queueBand` ‚Üê resolve from billing state using same logic as `reprioritize_pending_jobs_for_account` SQL:
     - `plan === "yearly" && unlimitedAccess.kind === "subscription" && subscriptionStatus === "active"` ‚Üí `"priority"`
     - `unlimitedAccess.kind === "self_hosted"` ‚Üí `"priority"`
     - `plan === "quarterly" && unlimitedAccess.kind === "subscription" && subscriptionStatus === "active"` ‚Üí `"standard"`
     - `unlimitedAccess.kind === "none" && creditBalance > 0` ‚Üí `"standard"`
     - else ‚Üí `"low"`

**Free default constant** (used by self-healing):
```typescript
const FREE_DEFAULT: BillingState = {
  plan: "free",
  creditBalance: 0,
  subscriptionStatus: "none",
  cancelAtPeriodEnd: false,
  unlimitedAccess: { kind: "none" },
  queueBand: "low",
};
```

**Important:** The self-healing insert must use `ON CONFLICT DO NOTHING` so it's safe under concurrent calls. Do NOT insert any credit balance or unlimited access ‚Äî return minimal free defaults.

The function takes a `supabase: AdminSupabaseClient` parameter (not creating the client internally) to enable testing with mock clients.

### 2. Create `src/lib/server/billing.functions.ts`

```typescript
import { createServerFn } from "@tanstack/react-start";
import { authMiddleware } from "@/lib/platform/auth/auth.middleware";
import { createAdminSupabaseClient } from "@/lib/data/client";
import { readBillingState } from "@/lib/domains/billing/queries";
import type { BillingState } from "@/lib/domains/billing/state";

export const getBillingState = createServerFn({ method: "GET" })
  .middleware([authMiddleware])
  .handler(async ({ context }): Promise<BillingState> => {
    const supabase = createAdminSupabaseClient();
    const result = await readBillingState(supabase, context.session.accountId);
    
    if (Result.isError(result)) {
      throw new Error("Failed to load billing state");
    }
    
    return result.value;
  });
```

## Constraints

From DECISIONS.md and TERMINOLOGY.md:

- **Missing row is a bug to repair** (DECISIONS.md ¬ß3): "missing row is an invariant violation, not a valid mode signal" ‚Äî self-heal via `INSERT ... ON CONFLICT DO NOTHING`
- **Self-healing must not grant purchased balance** (DECISIONS.md ¬ß3): default insert has `credit_balance = 0` (DB default), no `unlimited_access_source`
- **Subscription status normalization** (story + DECISIONS.md ¬ß2):
  - `active + cancel_at_period_end=true` ‚Üí `"ending"`
  - `past_due` / `unpaid` ‚Üí `"past_due"`
  - `canceled | incomplete | incomplete_expired` ‚Üí `"none"`
- **`past_due` does NOT grant unlimited access** (DECISIONS.md ¬ß2)
- **`self_hosted` grants unlimited regardless of Stripe state** (DECISIONS.md ¬ß2, ¬ß3)
- **Queue band mapping** (DECISIONS.md ¬ß1):
  - free, no balance ‚Üí `"low"`
  - positive purchased balance (non-unlimited) ‚Üí `"standard"` ‚Äî NOTE: story says "positive purchased balance" but `credit_balance` is the only balance field on the row; `queueBand` resolution here uses `credit_balance > 0` (consistent with `reprioritize_pending_jobs_for_account` SQL which also checks `credit_balance > 0 AND unlimited_access_source IS NULL`)
  - quarterly unlimited active ‚Üí `"standard"`
  - yearly unlimited active ‚Üí `"priority"`
  - `self_hosted` ‚Üí `"priority"`
- **No barrel exports** (CLAUDE.md): no `index.ts`, import directly from file path
- **No `any`, no non-null assertions** (AGENTS.md)
- **Errors as values** (AGENTS.md): `readBillingState` returns `Result<BillingState, DbError>`; the server function throws for the HTTP boundary

## Risks

1. **`queueBand` for "positive purchased balance"**: DECISIONS.md ¬ß1 says "any non-unlimited account with positive purchased balance" ‚Üí standard. The SQL RPC (`reprioritize_pending_jobs_for_account`) uses `credit_balance > 0 AND unlimited_access_source IS NULL`. The TS implementation should mirror this exactly. Note: `credit_balance` includes both purchased lot credits AND operational grants (replacement/admin), so it slightly over-counts "purchased" balance. This is the same behaviour as the SQL RPC and is intentional for v1.

2. **`plan` field as `BillingPlan` cast**: The DB type has `plan: string`. A defensive cast via a validation helper or explicit narrowing is safer than a raw `as BillingPlan`. Consider a helper `function isBillingPlan(v: string): v is BillingPlan` or use `STRIPE_STATUS_TO_NORMALIZED` pattern. The CHECK constraint guarantees valid values but TypeScript doesn't know that.

3. **`unlimitedAccess` for unknown `unlimited_access_source` values**: The DB type is `string | null`. Add a defensive fallback to `{ kind: "none" }` for any unexpected string value to avoid runtime errors if the schema ever extends the CHECK constraint values.

4. **Self-healing race condition**: Two concurrent requests for a new account will both attempt the insert; `ON CONFLICT DO NOTHING` makes this safe. The second caller gets 0 rows affected from the insert but can immediately re-read (or simply return the free default, since the row was just created with free defaults). Simplest approach: attempt insert, then read; or: read first, if null insert+return default.
