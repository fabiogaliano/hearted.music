### Completed Dependency: S2-02 (`getBillingState` server function)

**Location:** `src/lib/server/billing.functions.ts`

**Existing implementation:**
```typescript
export const getBillingState = createServerFn({ method: "GET" })
  .middleware([authMiddleware])
  .handler(async ({ context }): Promise<BillingState> => {
    const supabase = createAdminSupabaseClient();
    const result = await readBillingState(supabase, context.session.accountId);

    if (Result.isError(result)) {
      throw new Error("Failed to load billing state");
    }

    return result.value;
  });
```

**Key properties:**
- Returns `Promise<BillingState>` (never null/undefined if successful)
- Throws on error (database failure); no typed error return
- Already uses `authMiddleware` so can be called from route loaders
- Can be awaited directly in `beforeLoad` without query client (is a server function, not a server query)

## Constraints from DECISIONS.md and TERMINOLOGY.md

From `docs/monetization/DECISIONS.md`:

- **Queue bands** (§1): Free, no purchased balance → `"low"`; any non-unlimited with positive balance → `"standard"`; quarterly unlimited → `"standard"`; yearly unlimited → `"priority"`; self_hosted → `"priority"`

From `docs/monetization/TERMINOLOGY.md`:

- **Frozen plan values:** `free`, `quarterly`, `yearly` (§2, §5)
- **Frozen offer IDs:** `song_pack_500`, `unlimited_quarterly`, `unlimited_yearly` (§2, §5)
- **User-facing plan language:** 
  - Free plan: no special label in user-facing copy beyond "Free"
  - Quarterly: "3-Month Unlimited" (per story)
  - Yearly: "Backstage Pass" (per story, brand name, soft/descriptive)
  - Pack: referenced as "Song Pack" in onboarding; sidebar should show balance
- **Frozen type names:** `BillingState`, `BillingPlan`, `UnlimitedAccess`, `SongDisplayState` (§5)
- **Frozen unlimited access source values:** `null` (none), `'subscription'`, `'self_hosted'` (§5)
- **Provider-disabled deployment:** When `BILLING_ENABLED=false`, accounts get `unlimited_access_source='self_hosted'` and should show "Unlimited" or similar in UI

## Open Questions / Risks

### 1. Plan Label Mapping Not Fully Specified

The story specifies plan labels as:
- `"Free Plan"`
- `"Pack (X songs to explore)"` (X = creditBalance)
- `"3-Month Unlimited"` (for quarterly)
- `"Backstage Pass"` (for yearly)
- Unnamed "Unlimited" for self_hosted/provider-disabled

**Question:** Where should the label-generation logic live?
- Option A: In `route.tsx` inline, derive label from `billingState` before passing to Sidebar
- Option B: Extract to a helper function in a utility or domain module (e.g., `src/lib/domains/billing/display.ts`)
- Option C: In Sidebar component itself (less preferred — requires passing both `plan` and `creditBalance` separately)

Implementation agent must decide. The brief does not prescribe location.

### 2. Balance Display Conditional Logic

The story says:
- Show balance for pack users (`plan === "free"` AND `creditBalance > 0`)
- Hide for unlimited (`hasUnlimitedAccess(state) === true`)
- Hide for self_hosted (provider-disabled)

**Question:** Should balance appear as part of the `userPlan` string (e.g., `"Pack (500 songs to explore)"`) or as a separate UI element?

Story wording ("add balance display to sidebar") suggests separate display, but exact placement (same line as plan label, below, etc.) is unspecified. Implementation agent must decide.

### 3. Self-Hosted Label Ambiguity

For a self_hosted account (`unlimited_access_source === "self_hosted"`), the story mentions `"Unlimited"` or similar but does not specify exact wording. Also:
- Self_hosted accounts can have `plan === "free"` AND `unlimited_access_source === "self_hosted"` simultaneously (DECISIONS.md §3)
- No `creditBalance` for self_hosted (always 0 on free default)

**Risk:** Implementation must distinguish self_hosted unlimited from subscription unlimited in UI label. Currently no distinct label exists in the story.

### 4. Error Handling for `getBillingState()`

`getBillingState()` throws `Error("Failed to load billing state")` on DB failure. 

**Question:** How should route `beforeLoad` handle this error?
- Propagate and let TanStack Router handle (redirect to error boundary)?
- Catch and return a fallback (e.g., free default)?
- Retry?

Story does not specify. Implementation agent must decide error strategy before route `beforeLoad` can be written.

### 5. Route Context Type Safety

The route context is currently typed implicitly by the return type of `beforeLoad`. Adding `billingState: BillingState` will extend the context. 

**Question:** Should a new explicit context type be declared?
- Current pattern: inferred from return type
- Adding a type alias would improve IDE support but is not required

Implementation agent may choose to add an explicit `RouteContext` type for clarity.

### 6. Query Caching vs Direct Call

The story does not specify whether `getBillingState()` should be:
- Called directly in `beforeLoad` (no caching)
- Cached via `queryClient.ensureQueryData()` (reused across navigations)

Current dependencies (`session`, `account`, `onboarding`) all use `ensureQueryData()` with configurable `staleTime`. 

**Question:** Should billing state follow the same pattern with a cache TTL, or should it always be fresh?

Implementation agent must decide based on billing data freshness requirements.

### 7. Sidebar Props Contract Change

Adding balance display or a separate plan label may require extending `SidebarProps`:
- Option A: Pass `BillingState` directly to Sidebar (makes component aware of billing domain)
- Option B: Pass derived `userPlan: string` and `userBalance?: number` (keeps Sidebar domain-agnostic)
- Option C: Compute label+balance in `route.tsx`, pass as single string (e.g., `"Pack (500 songs)"`)

Story does not prescribe. Implementation agent must decide. Option B maintains current separation of concerns.
