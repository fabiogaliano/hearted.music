# Implementation Brief: S3-05 Match Refresh Candidate Filtering

## Goal

Replace the ungated `select_data_enriched_liked_song_ids` RPC call in the match snapshot refresh candidate-loading step with the billing-aware `select_entitled_data_enriched_liked_song_ids` RPC, so revoked and locked songs are excluded from matching candidates.

---

## Current State

### `src/lib/workflows/enrichment-pipeline/batch.ts` (relevant excerpt)

```typescript
/**
 * Returns liked song IDs that have all 4 shared data artifacts.
 * Used by match snapshot refresh for candidate loading.
 */
export async function getDataEnrichedSongIds(
	accountId: string,
): Promise<string[]> {
	const supabase = createAdminSupabaseClient();

	const { data, error } = await supabase.rpc(
		"select_data_enriched_liked_song_ids",
		{ p_account_id: accountId },
	);

	if (error) {
		throw new Error(`Failed to select data-enriched songs: ${error.message}`);
	}

	return (data ?? []).map((row: { song_id: string }) => row.song_id);
}
```

This function is the sole call site for `select_data_enriched_liked_song_ids` in `src/`. It is called by the orchestrator.

## Changes Required

### 1. Add `getEntitledDataEnrichedSongIds` to `src/lib/workflows/enrichment-pipeline/batch.ts`

Add a new exported function alongside the existing `getDataEnrichedSongIds` (do not remove or modify the existing one):

```typescript
/**
 * Returns liked song IDs that have all 4 shared data artifacts AND effective entitlement.
 * Used by match snapshot refresh for billing-aware candidate loading.
 * Revoked and locked songs are excluded.
 */
export async function getEntitledDataEnrichedSongIds(
	accountId: string,
): Promise<string[]> {
	const supabase = createAdminSupabaseClient();

	const { data, error } = await supabase.rpc(
		"select_entitled_data_enriched_liked_song_ids",
		{ p_account_id: accountId },
	);

	if (error) {
		throw new Error(
			`Failed to select entitled data-enriched songs: ${error.message}`,
		);
	}

	return (data ?? []).map((row: { song_id: string }) => row.song_id);
}
```

## Constraints

From DECISIONS.md and TERMINOLOGY.md (as applied by S1-11):

- **Effective entitlement** = `revoked_at IS NULL` (active unlock row) OR unlimited access (`self_hosted` or `subscription` with `subscription_status = 'active'`).
- `past_due`/`unpaid` do NOT grant unlimited for `subscription` source.
- `self_hosted` grants unlimited unconditionally.
- Match-refresh selector requires all 4 shared artifacts (audio_features, genres, analysis, embedding) **plus entitlement**. Does NOT require `item_status`.
- Revoked songs (`revoked_at IS NOT NULL`, no unlimited access) must be excluded.
- The `select_entitled_data_enriched_liked_song_ids` RPC is already implemented (S1-11) and typed (S1-12) — no DB changes needed.

---

## Acceptance Criteria

- [ ] Match refresh uses the billing-aware selector — orchestrator calls `select_entitled_data_enriched_liked_song_ids` (via `getEntitledDataEnrichedSongIds`), not the old `select_data_enriched_liked_song_ids`.
- [ ] Revoked songs excluded — a song with `revoked_at IS NOT NULL` and no unlimited access is not a match candidate.
- [ ] Locked songs excluded — a song with no unlock row and no unlimited billing access is excluded.
- [ ] Self-hosted/unlimited songs included — accounts with `unlimited_access_source = 'self_hosted'` or active subscription see their fully-enriched songs as candidates.
- [ ] Project compiles — `bun run typecheck` passes with no new errors in `src/`.
