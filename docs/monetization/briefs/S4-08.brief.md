### HMAC Auth Middleware Pattern (from S4-01)

**File:** `v1_hearted_brand/src/middleware/hmac-auth.ts` (reference implementation)

The billing service defines this pattern. The app must verify incoming signatures using the same logic:

```typescript
/**
 * HMAC auth using SHA-256.
 *
 * Expected headers:
 *   X-Timestamp   — Unix epoch seconds when request was signed
 *   X-Signature   — hex(HMAC-SHA256(secret, timestamp + "." + hex(SHA256(body))))
 *
 * Rejects requests with missing headers, expired timestamps, or invalid signatures.
 * Clock skew window: 5 minutes.
 */
function hmacAuth(secret: string): MiddlewareHandler {
  const encoder = new TextEncoder();
  const secretKey = encoder.encode(secret);

  return async (c, next) => {
    const timestamp = c.req.header("X-Timestamp");
    const signature = c.req.header("X-Signature");

    if (!timestamp || !signature) {
      return c.json({ error: "Missing authentication headers" }, 401);
    }

    const requestTime = Number(timestamp);
    if (Number.isNaN(requestTime)) {
      return c.json({ error: "Invalid timestamp" }, 401);
    }

    const now = Date.now() / 1000;
    const CLOCK_SKEW_MS = 5 * 60 * 1000;
    if (Math.abs(now - requestTime) > CLOCK_SKEW_MS / 1000) {
      return c.json({ error: "Request timestamp expired" }, 401);
    }

    const body = await c.req.raw.clone().text();
    const bodyHash = toHex(
      await crypto.subtle.digest("SHA-256", encoder.encode(body)),
    );

    const message = `${timestamp}.${bodyHash}`;
    const key = await crypto.subtle.importKey(
      "raw",
      secretKey,
      { name: "HMAC", hash: "SHA-256" },
      false,
      ["sign"],
    );
    const expected = toHex(
      await crypto.subtle.sign("HMAC", key, encoder.encode(message)),
    );

    if (!timingSafeEqual(expected, signature)) {
      return c.json({ error: "Invalid signature" }, 401);
    }

    await next();
  };
}
```

**Key points:**
- Signature format: `hex(HMAC-SHA256(secret, "{timestamp}.{hex(SHA256(body))}"))` 
- Headers: `X-Timestamp` (Unix epoch seconds), `X-Signature` (hex string)
- Clock skew: 5 minutes (300 seconds)
- Timing-safe comparison to prevent side-channel attacks

## Constraints (Frozen from DECISIONS.md & TERMINOLOGY.md)

### HMAC Auth (DECISIONS.md §2)
- **Signature components:** timestamp header + raw request body hash (SHA256) + shared secret
- **Shared secret:** Must match `BILLING_SHARED_SECRET` in both repos (v1_hearted and v1_hearted_brand)
- **Clock-skew window:** 5 minutes (300 seconds) for replay protection
- **Header names:** `X-Timestamp` (Unix epoch seconds) and `X-Signature` (hex-encoded HMAC-SHA256)
- **Message format:** `{timestamp}.{hex(SHA256(body))}`

### Bridge Event Idempotency (DECISIONS.md §2)
- **Key:** `stripe_event_id` (PRIMARY KEY in `billing_bridge_event`)
- **Mechanism:** `INSERT ... ON CONFLICT DO NOTHING`
- **Check:** Query the insert result; duplicate deliveries must be detected and no-op'd before proceeding to handlers
- **Success response:** Always return 2xx for duplicates (so billing service doesn't retry)

### Bridge Event Kinds (TERMINOLOGY.md §6, frozen)
- `pack_fulfilled` → maps to `PackFulfilledBridgePayload`
- `unlimited_activated` → maps to `UnlimitedActivatedBridgePayload`
- `pack_reversed` → maps to `RevocationOutcomeBridgePayload`
- `unlimited_period_reversed` → maps to `RevocationOutcomeBridgePayload`
- `subscription_deactivated` → maps to `RevocationOutcomeBridgePayload`

### Control-Plane Change Kinds (TERMINOLOGY.md §6, frozen)
- `songs_unlocked` — requires `accountId` and `songIds`
- `unlimited_activated` — requires `accountId` only
- `candidate_access_revoked` — requires `accountId` only (no detail on why or which songs)

- [ ] Accepts valid HMAC-signed requests from billing service
- [ ] Duplicate `stripe_event_id` deliveries are no-ops (idempotency via PK constraint + conflict check)
- [ ] Returns 2xx success response for duplicates (so billing service doesn't retry)

  - `pack_fulfilled` → creates `songs_unlocked` change with song IDs

- [ ] Not available when `BILLING_ENABLED=false` (404 or guard)
- [ ] Tests cover:
  - Valid signed request → accepted and processed
  - Duplicate delivery → no-op, success response
  - Invalid signature → 401 rejected
  - Expired timestamp (> 5 min old) → 401 rejected
  - Missing headers → 401 rejected
  - Unknown event_kind → appropriate error handling

### 3. Bridge Payload Parsing & Validation
**Issue:** The bridge payloads from S2-05 (`PackFulfilledBridgePayload`, etc.) are TypeScript interfaces. No Zod schema is provided.

**Unclear:** Should the endpoint:
- Manually validate fields?
- Create Zod schemas for each payload type?
- Use runtime assertions?

**Impact:** Type safety at the boundary; affects how invalid/partial payloads are rejected.

### 4. Error Handling for Bridge Dispatch
**Issue:** After HMAC verification and idempotency check, the endpoint dispatches to handlers. If a handler fails, what should happen?

**Unclear:**
- Should the HTTP response be 5xx if a handler throws?
- Should errors be logged but not fail the response (fire-and-forget)?
- Should the endpoint attempt to handle errors from `applyLibraryProcessingChange()`?

**Impact:** Affects retry semantics; billing service may retry on 5xx but not on 2xx. If we return 2xx but internal processing fails, the event is lost.

### 5. Logging & Observability
**Issue:** Story doesn't mention logging, but HMAC rejections and bridge event processing should be observable for debugging.

**Unclear:**
- Should HMAC failures log the request details (sanitized)?
- Should idempotent duplicates be logged?
- Should handler dispatch be logged with event_kind and accountId?

**Impact:** Production debugging; should align with team logging practices.

### 6. Request Body Handling (Streaming vs. Buffering)
**Issue:** HMAC verification requires reading the full request body twice: once to compute the body hash, once to parse the JSON payload.

**Unclear:**
- Should the route buffer the entire body into memory?
- Should it use `.clone()` on the request like the billing service does?
- What's the expected payload size (is buffering safe)?

**Impact:** Memory efficiency; Hono/TanStack patterns for this.

### 7. Account ID Validation
**Issue:** The bridge payloads include `accountId`, but don't specify if the endpoint should verify the account exists before dispatching.

**Unclear:**
- Should the endpoint check that the account exists in the `account` table?
- Should it check that the account has an `account_billing` row?
- What should happen if the account is missing?

**Impact:** Data integrity; prevents orphaned changes for non-existent accounts.

### 8. Response Body & Status Codes
**Issue:** Story doesn't specify exact HTTP status codes or response body structure.

**Unclear:**
- Success response: `200 OK` with `{"success": true}` or `204 No Content`?

**Unclear:**

## Summary

S4-08 bridges the billing service (`v1_hearted_brand`) to the app's control plane. The endpoint must:

1. **Verify HMAC signature** using `BILLING_SHARED_SECRET`, timestamp, and body hash (pattern inherited from S4-01)
2. **Check idempotency** via `billing_bridge_event` PK constraint (from S1-03)
3. **Dispatch to handlers** by `event_kind` (handlers TBD in S4-09)
4. **Inject control-plane changes** via `applyLibraryProcessingChange()` (from S2-05)

All types, database schema, and change helpers are already in place. The endpoint is a thin integration layer — verification logic + idempotency check + routing + change injection.

**Major unknowns** are the exact route path, guard mechanism, error handling semantics, and response contract — these should be clarified before or during implementation to ensure the billing service can call correctly.
