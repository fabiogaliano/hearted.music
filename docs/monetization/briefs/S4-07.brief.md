# S4-07 Context Snapshot: Refund/Dispute + Checkout Expiry Handlers

## Goal

```ts
type EventHandler = (event: Stripe.Event) => Promise<void>;

const EVENT_HANDLERS: Record<string, EventHandler> = {
	"checkout.session.completed": handleCheckoutCompleted,
	"invoice.paid": handleInvoicePaid,
	"invoice.payment_failed": handleInvoicePaymentFailed,
	"customer.subscription.updated": handleSubscriptionUpdated,
	"customer.subscription.deleted": handleSubscriptionDeleted,
};

webhookRoutes.post("/webhooks/stripe", async (c) => {
	const signature = c.req.header("stripe-signature");
	const rawBody = await c.req.raw.clone().text();
	event = stripe.webhooks.constructEvent(rawBody, signature, env.STRIPE_WEBHOOK_SECRET);
	
	const handler = EVENT_HANDLERS[event.type];
	if (!handler) {
		return c.json({ received: true });
	}

	let claimed: boolean;
	claimed = await claimWebhookEvent(event.id);
	if (!claimed) {
		return c.json({ received: true });
	}

	try {
		await handler(event);
	} catch (err) {
		await markWebhookEvent(event.id, "failed", message);
	}
	return c.json({ received: true });
});
```

### Idempotency & Event Marking (S4-05)

**File: `v1_hearted_brand/src/lib/webhook-event.ts`**

```ts
export async function claimWebhookEvent(stripeEventId: string): Promise<boolean> {
	const { count, error } = await supabase
		.from("billing_webhook_event")
		.insert({ stripe_event_id: stripeEventId, status: "processing" as const })
		.select("stripe_event_id")
		.single();

	if (error && error.code === "23505") return false; // already exists
	return true;
}

export async function markWebhookEvent(
	stripeEventId: string,
	status: "processed" | "failed",
	errorMessage?: string,
): Promise<void> {
	const update: Record<string, unknown> = {
		status,
		updated_at: new Date().toISOString(),
	};
	if (status === "processed") {
		update.processed_at = new Date().toISOString();
	}
	if (errorMessage) {
		update.error_message = errorMessage.slice(0, 2000);
	}

	await supabase
		.from("billing_webhook_event")
		.update(update)
		.eq("stripe_event_id", stripeEventId);
}
```

**Usage pattern:**
1. On webhook receipt: `claimWebhookEvent(event.id)` → returns true if this is the winner
2. Second delivery of same event: returns false, endpoint returns 200 immediately
3. After processing: `markWebhookEvent(event.id, "processed")` or `"failed"`

### Bridge Event Sending (S4-05)

**File: `v1_hearted_brand/src/lib/bridge.ts`**

```ts
type BridgeEventKind =
	| "pack_fulfilled"
	| "unlimited_activated"
	| "subscription_deactivated";

interface BridgePayload {
	kind: BridgeEventKind;
	stripe_event_id: string;
	data: Record<string, unknown>;
}

async function signPayload(
	body: string,
	secret: string,
): Promise<{ timestamp: string; signature: string }> {
	const timestamp = Math.floor(Date.now() / 1000).toString();
	const bodyHash = toHex(await crypto.subtle.digest("SHA-256", encoder.encode(body)));
	const message = `${timestamp}.${bodyHash}`;
	const key = await crypto.subtle.importKey("raw", encoder.encode(secret), ...);
	const signature = toHex(await crypto.subtle.sign("HMAC", key, encoder.encode(message)));
	return { timestamp, signature };
}

export async function sendBridgeEvent(payload: BridgePayload): Promise<Response> {
	const body = JSON.stringify(payload);
	const { timestamp, signature } = await signPayload(body, env.BILLING_SHARED_SECRET);

	return fetch(`${env.APP_BRIDGE_URL}/api/billing/bridge`, {
		method: "POST",
		headers: {
			"Content-Type": "application/json",
			"X-Timestamp": timestamp,
			"X-Signature": signature,
		},
		body,
	});
}
```

**Current bridge event kinds in type definition:**
- `pack_fulfilled` (S4-05)
- `unlimited_activated` (S4-06)
- `subscription_deactivated` (S4-06)

**Missing (S4-07):** `pack_reversed`, `unlimited_period_reversed` are in the frozen `BridgeEventKind` enum (from TERMINOLOGY.md §5) but not yet in the TypeScript type definition.

### Existing Reversal RPCs (Dependencies)

#### `reverse_pack_entitlement` (S1-06)

**File: `supabase/migrations/20260405200000_credit_and_pack_rpcs.sql`**

**Signature:**
```sql
CREATE OR REPLACE FUNCTION reverse_pack_entitlement(
  p_account_id           UUID,
  p_pack_stripe_event_id TEXT,
  p_stripe_event_id      TEXT,
  p_reason               TEXT
) RETURNS JSONB
```

**Returns:**
```jsonb
{
  "credits_reversed": INTEGER,
  "revoked_song_ids": UUID[]
}
```

**Behavior:**
- Locks `account_billing` and `pack_credit_lot` (consistent order)
- Zeroes `remaining_credits` on the lot
- Subtracts `remaining_credits` from `credit_balance`
- Writes `credit_transaction` with reason mapping:
  - `p_reason='refund'` → `reason='refund'`
  - `p_reason='chargeback'` → `reason='chargeback_reversal'`
- Revokes bonus unlocks tagged with `granted_stripe_event_id = p_pack_stripe_event_id`
- Revokes newest active `source='pack'` unlocks up to `credits_spent` count
- Never revokes `free_auto`, `unlimited`, `self_hosted`, or `admin` unlocks
- Calls `reprioritize_pending_jobs_for_account` as final step

#### `reverse_unlimited_period_entitlement` (S1-09)

**File: `supabase/migrations/20260405140000_unlimited_reversal_rpc.sql`**

**Signature:**
```sql
CREATE OR REPLACE FUNCTION reverse_unlimited_period_entitlement(
  p_stripe_subscription_id TEXT,
  p_subscription_period_end TIMESTAMPTZ,
  p_stripe_event_id TEXT,
  p_revoked_reason TEXT
) RETURNS TABLE(song_id UUID)
```

**Behavior:**
- Revokes all `source='unlimited'` unlock rows matching the subscription period key
- Sets `revoked_at`, `revoked_reason`, `revoked_stripe_event_id` on matched rows
- Uses WHERE clause `revoked_at IS NULL` for idempotency
- Never revokes other sources
- Returns list of newly revoked song IDs

#### `reverse_subscription_upgrade_conversion` (S1-08)

**File: `supabase/migrations/20260405180000_conversion_lifecycle_rpcs.sql`**

**Signature:**
```sql
CREATE OR REPLACE FUNCTION reverse_subscription_upgrade_conversion(
  p_conversion_id   UUID,
  p_stripe_event_id TEXT
) RETURNS VOID
```

**Behavior:**
- Locks conversion row (must be `status='applied'`)
- Locks `account_billing` before balance mutation
- Restores `remaining_credits` on allocated lots by `reserved_credits` amount
- Increments `credit_balance` by `converted_credits`
- Writes `credit_transaction` with `reason='credit_conversion_reversal'`
- Sets conversion `status='reversed'`

## Constraints (Frozen)

### From DECISIONS.md

| Constraint | Detail |
|---|---|
| **Full pack refund reversal** | 525-song footprint: 500 balance credits + 25 bonus unlocks |
| **Never revoke certain sources** | `free_auto`, `self_hosted`, `admin` unlocks must never be revoked by pack refund |
| **Unlimited reversal key** | `(granted_stripe_subscription_id, granted_subscription_period_end)` — NOT `granted_stripe_event_id` |
| **Idempotency** | All handlers idempotent via `billing_webhook_event` status tracking |
| **Initial invoice refund** | If refunded AND consumed a conversion, restore that conversion first |
| **Pending jobs reprioritization** | All refund flows must leave pending jobs reprioritized to final post-refund band |
| **Ambiguous cases** | Create admin task for manual review when refund path is unclear |
| **Checkout expiry** | Release pending conversion (set status='released'); frees reserved pack value for other use |

| **Unlock revocation reasons** | `refund \| chargeback \| admin` |
| **Conversion statuses** | `pending \| applied \| released \| reversed` |
| **Bridge event kinds** | `pack_fulfilled \| unlimited_activated \| pack_reversed \| unlimited_period_reversed \| subscription_deactivated` |
| **Unlimited access source** | `subscription \| self_hosted` (NULL = none) |
| **Unlock sources** | `free_auto \| pack \| unlimited \| self_hosted \| admin` |

## Open Questions / Risks

### 1. Stripe Refund Event Delivery Model

**Issue:** The story mentions `charge.refunded` and `charge.dispute.created` as the trigger events for refunds and chargebacks. However:
- A refund can happen on a charge created during a pack purchase (mode=payment)
- A refund can happen on an invoice paid for a subscription (mode=subscription + initial invoice with conversion)
- A dispute can involve either or both

**Unclear:** Should the handler:
- Determine invoice vs charge vs dispute by reading the Stripe object structure?
- Look up the charge from the refund object?
- Look up the invoice from a charge reference?
- The story doesn't specify whether `charge.refunded` is the right event or if `charge.dispute.created` is separate. Stripe's actual event types are documented but the mapping to internal billing operations is not specified in the story.

### 2. Ambiguous Case Triggering

**Issue:** The story says "Create admin task for manual review" for ambiguous cases, but:
- What constitutes an ambiguous case? (e.g., refund with no matching pack lot or conversion? refund that's partial? negative amount?)
- Is there a database table for admin tasks or are these created outside the billing service?
- What fields should the admin task contain?

**Unclear:** The story doesn't specify how to detect ambiguity or where/how to record it for human review.

### 3. Bridge Event Kind Type Update

**Issue:** The TypeScript `BridgeEventKind` type in `src/lib/bridge.ts` currently only includes:
```ts
type BridgeEventKind =
	| "pack_fulfilled"
	| "unlimited_activated"
	| "subscription_deactivated";
```

But TERMINOLOGY.md §5 (frozen) specifies:
```
'pack_fulfilled | unlimited_activated | pack_reversed | unlimited_period_reversed | subscription_deactivated'
```

**Risk:** The TypeScript type is incomplete. The implementation must update it to include `pack_reversed` and `unlimited_period_reversed` before sending those bridge events.

### 4. Pack Refund with Partial Spend

**Issue:** `reverse_pack_entitlement` uses `v_credits_spent = original_credits - remaining_credits`. If a lot was partially spent (e.g., 100 of 500 spent):
- `v_credits_to_sub = 150` (remaining, subtracted from balance)
- `v_credits_spent = 400` (spent, revoke up to 400 newest pack unlocks)

This requires revocation of 400 pack unlock rows on a single 500-credit lot. The story doesn't clarify:
- Is it acceptable to revoke across multiple lots if an account has many?
- Should oldest/newest spent credits be revoked preferentially?
- The current logic revokes newest-first (DESC by created_at).

### 5. Initial Invoice Refund with Conversion

**Issue:** The story says "If initial invoice refunded AND consumed a conversion, restore that conversion first". However:
- How does the handler identify that an invoice was the initial invoice for a subscription?
- The `charge.refunded` or `charge.dispute.created` event doesn't directly reference the conversion ID
- The handler must look up: charge → invoice → subscription → account → conversion with status='applied'

**Unclear:** Lookup path and whether it's safe to assume one `applied` conversion per account per subscription.

### 6. Checkout Expiry Handler Scope

**Issue:** The story mentions `checkout.session.expired` should "call `release_subscription_upgrade_conversion`". However:
- A checkout session can be for a pack (mode=payment, no conversion) or subscription (mode=subscription, with conversion)
- How does the handler distinguish?
- The conversion is stored in `subscription_credit_conversion` and linked via `checkout_session_id`

**Unclear:** Should the handler:
- Query `subscription_credit_conversion` where `checkout_session_id = session.id` and `status = 'pending'`?
- If none found, is it a pack-only expiry (no-op)?

### 7. Conversion Lookup from Invoice Refund

**Issue:** On `charge.refunded` for an invoice payment, the handler needs to find the `subscription_credit_conversion` that was applied. The charge object has:
- `invoice_id` reference
- `subscription_id` reference

The `subscription_credit_conversion` table has:
- `stripe_invoice_id` (the initial invoice for the subscription)
- `status = 'applied'` (after conversion was applied)

**Unclear:** Can the handler safely assume there's at most one `applied` conversion per subscription? Or should it query `WHERE stripe_invoice_id = charge.invoice_id AND status = 'applied'`?

### 8. Pack Refund Without Corresponding Fulfillment

**Issue:** If a `charge.refunded` event arrives but there is no `pack_credit_lot` row with the matching `stripe_event_id`, the `reverse_pack_entitlement` RPC will raise an exception: "pack_credit_lot not found".

**Unclear:** Should the handler:
- Catch this exception and create an admin task (ambiguous case)?
- Assume this is a data integrity problem and mark the webhook as failed?
- Skip silently (idempotent no-op)?

### 9. Dispute Timeline

**Issue:** A `charge.dispute.created` event indicates a dispute has been filed, not that the payment was reversed. The actual reversal happens later when Stripe processes the dispute outcome.

**Unclear:** Should the handler:
- Immediately reverse billing state on `charge.dispute.created` (aggressive)?
- Wait for `charge.dispute.closed` (more correct but requires another handler)?
- The story says "charge.dispute.created" but doesn't clarify whether that's the right event or if handling should be on dispute closure.

### 10. Bridge Call Failure Mode

**Issue:** Unlike pack fulfillment (S4-05), refund/dispute handlers have already mutated billing state (called reverse_*_entitlement RPCs) before sending the bridge event. If the bridge call fails:
- Billing state is already reversed (correct)
- Control plane doesn't know (incorrect)
- Songs remain visible in UI until user refreshes or next sync

**Unclear:** Should the handler:
- Retry bridge calls with exponential backoff?
- Log and alert on bridge failure?
- Keep webhook event as `failed` to allow manual bridge retry?
- The story doesn't specify failure recovery for bridge calls in the reversal path.

### 11. Admin Task Creation Mechanism

**Issue:** The story mentions "create admin task for manual review" but:
- There is no `admin_task` or `admin_review` table in the schema
- There's no RPC for creating admin tasks
- The mechanism is unspecified

**Unclear:** Where and how should admin tasks be created?

## Summary

S4-07 implements three webhook handlers that reverse billing state on refunds, chargebacks, and checkout expiry. The infrastructure is in place (webhook routing, idempotency, bridge pattern), and all reversal RPCs (pack, unlimited, conversion) are available. The main implementation work is:

1. Add event handlers to the `EVENT_HANDLERS` map for `checkout.session.expired`, `charge.refunded`, `charge.dispute.created`
2. Implement logic to:
   - Extract account/conversion/lot identifiers from Stripe objects
   - Call appropriate reversal RPCs
   - Send bridge events with updated event kind types
   - Handle ambiguous cases (unclear)
3. Update `BridgeEventKind` TypeScript type to include `pack_reversed` and `unlimited_period_reversed`

**Key unknowns:** Ambiguity detection criteria, admin task creation, dispute event timing, bridge failure recovery, conversion lookup from invoice refunds.
