### Bridge Ingress Endpoint (S4-08 — Existing)

**File:** `src/routes/api/billing-bridge.ts`

Currently receives and dispatches bridge events to handlers via a simple switch statement:

```typescript
export const Route = createFileRoute("/api/billing-bridge")({
	server: {
		handlers: {
			POST: async ({ request }) => {
				// HMAC verification
				const hmacResult = await verifyBridgeHmac(request, secret);
				if (!hmacResult.valid) {
					return Response.json({ error: hmacResult.error }, { status: 401 });
				}

				// Parse and validate payload
				let payload: BridgePayload;
				try {
					const raw = JSON.parse(hmacResult.body);
					payload = BridgePayloadSchema.parse(raw);
				} catch {
					return Response.json(
						{ error: "Invalid request payload" },
						{ status: 400 },
					);
				}

				// Idempotency check via billing_bridge_event PK
				const supabase = createAdminSupabaseClient();
				const { data, error } = await supabase
					.from("billing_bridge_event")
					.insert({
						stripe_event_id: payload.stripe_event_id,
						event_kind: payload.event_kind,
					})
					.select("stripe_event_id")
					.single();

				if (error && error.code !== "23505") {
					console.error("[billing-bridge] Failed to insert bridge event:", error);
					return Response.json(
						{ error: "Internal server error" },
						{ status: 500 },
					);
				}

				// Duplicate delivery — no-op, return success
				if (!data) {
					return Response.json({ ok: true, duplicate: true });
				}

				// Dispatch to handlers
				try {
					await dispatchBridgeEvent(payload);
				} catch (err) {
					console.error("[billing-bridge] Handler dispatch failed:", err);
					return Response.json(
						{ error: "Internal processing error" },
						{ status: 500 },
					);
				}

				return Response.json({ ok: true });
			},
		},
	},
});

// Current dispatch implementation (INSUFFICIENT — story asks for more detailed handlers)
async function dispatchBridgeEvent(payload: BridgePayload): Promise<void> {
	switch (payload.event_kind) {
		case "pack_fulfilled": {
			const change = BillingChanges.songsUnlocked(
				payload.account_id,
				payload.bonus_unlocked_song_ids,
			);
			await applyLibraryProcessingChange(change);
			break;
		}
		case "unlimited_activated": {
			const change = BillingChanges.unlimitedActivated(payload.account_id);
			await applyLibraryProcessingChange(change);
			break;
		}
		case "pack_reversed":
		case "unlimited_period_reversed":
		case "subscription_deactivated": {
			const change = BillingChanges.candidateAccessRevoked(payload.account_id);
			await applyLibraryProcessingChange(change);
			break;
		}
	}
}
```

**Issue:** The current dispatch logic is too simple. It does not:
1. Check if bonus songs exist before emitting for pack fulfillment
2. Insert `billing_activation` marker for unlimited activation
3. Call reversal RPCs and check if access was actually removed for revocation events

## Constraints (Frozen)

From `docs/monetization/DECISIONS.md` and `docs/monetization/TERMINOLOGY.md`:

### Bridge Event Idempotency (DECISIONS.md §2)
- `billing_bridge_event.stripe_event_id` is the PRIMARY KEY
- Duplicate deliveries result in `INSERT ... ON CONFLICT DO NOTHING`, checked before handler dispatch
- Success response (2xx) returned for duplicates so billing service doesn't retry

### Billing Activation Marker (DECISIONS.md §2)
- `billing_activation` row prevents duplicate `unlimited_activated` emissions per period
- Unique constraint on `(account_id, kind, stripe_subscription_id, subscription_period_end)` gates duplicate inserts
- `kind` value frozen: `'unlimited_period_activated'`

### Control-Plane Responsibility (DECISIONS.md §2)
- Control plane never learns *why* access changed, only *that* it changed
- No Stripe-specific identifiers, refund reasons, or chargeback details leak into control-plane code
- `songs_unlocked` carries song IDs; `unlimited_activated` and `candidate_access_revoked` carry only `accountId`

### Revocation Assessment (DECISIONS.md §2, Story S4-09)
- Revocation handler must check whether access was **actually** removed
- If reversal RPCs return no revoked songs, do NOT emit `candidateAccessRevoked`
- Only emit the change if the set of revoked songs is non-empty

### Pack Fulfillment (Story S4-09, DECISIONS.md §1)
- Pack bonus unlocks are up to 25 most-recent liked songs
- Emit `songsUnlocked` only if `bonusUnlockedSongIds.length > 0`

### Bridge Event Kinds (TERMINOLOGY.md §5, frozen)
- `pack_fulfilled` → `PackFulfilledBridgePayload`
- `unlimited_activated` → `UnlimitedActivatedBridgePayload`
- `pack_reversed`, `unlimited_period_reversed`, `subscription_deactivated` → `RevocationOutcomeBridgePayload`

## Open Questions / Risks

### 1. Bridge Payload Schema Mismatch for `unlimited_activated`

**Issue:** The story requires the handler to receive and insert `stripe_subscription_id` and `subscription_period_end`, but:
- The `UnlimitedActivatedBridgePayload` interface omits these fields
- The Zod schema in the bridge endpoint doesn't expect them

**Unclear:** Should the handler be modified to:
- Query `account_billing` separately to get `stripe_subscription_id` and `subscription_period_end` after unlimited activation completes?
- Or should the bridge endpoint schema and payload types be updated first (blocking change)?

**Impact:** Without these fields, the handler cannot insert the `billing_activation` marker as required.

### 2. Revocation Handler Event Kind Mismatch

**Issue:** The story says revocation handler receives "revocation outcome from billing service" but the current bridge endpoint dispatches all three revocation event kinds (`pack_reversed`, `unlimited_period_reversed`, `subscription_deactivated`) to the same handler.

**Unclear:** Should the handler:
- Call `reverse_pack_entitlement` for `pack_reversed` events?
- Call `reverse_unlimited_period_entitlement` for `unlimited_period_reversed` events?
- Call something else for `subscription_deactivated` events (which may not have an associated pack or unlimited period)?

**Impact:** The three event kinds have different reversal logic and may require different RPC calls or reasoning about which access was removed.

### 3. Pack Reversal RPC Requires `pack_stripe_event_id`

**Issue:** The `reverse_pack_entitlement` RPC signature requires `p_pack_stripe_event_id` (the Stripe event ID from the original pack purchase), but the `pack_reversed` bridge payload only carries the current `stripe_event_id` (from the refund/chargeback event).

**Unclear:** How should the handler obtain the original pack purchase's `stripe_event_id`? Should it:
- Query `pack_credit_lot` rows for the account and infer it?
- Expect the billing service to include it in the bridge payload (requiring payload schema changes)?

**Impact:** Without the pack's original event ID, the revocation handler cannot call `reverse_pack_entitlement`.

### 4. Subscription Deactivation and Access Removal

**Issue:** The story says revocation handler "determines whether access was actually removed" and emits `candidateAccessRevoked` only if true. For `subscription_deactivated` events:
- The account's unlimited access is cleared by the billing service's `deactivate_subscription` RPC
- But no active unlock rows are revoked (per DECISIONS.md §1: "Normal cancellation — access through paid period end; previously unlocked songs stay unlocked; no revocation")
- So technically, no access was "removed" — the songs are still unlocked

**Unclear:** Should `subscription_deactivated` always skip emitting `candidateAccessRevoked`? Or should it check whether any unlimited-sourced unlocks exist and treat the loss of future auto-unlocking as "access removal"?

**Impact:** Affects control-plane logic for `subscription_deactivated` events. The story's acceptance criterion says "emit only when access was actually reduced" — does a normal (non-refund) subscription cancellation count as reduction?

### 5. HMAC Auth for Revocation Payloads

**Issue:** The revocation bridge payloads are minimal (only `stripe_event_id`, `account_id`, `event_kind`). The story doesn't specify whether the billing service includes additional context (e.g., `reason: 'refund' | 'chargeback'`) in the payload.

**Unclear:** Should the revocation handler:
- Always assume `reason: 'refund'` (default)?
- Expect the billing service to include it in the payload (schema change)?
- Look it up from `billing_webhook_event` status?

**Impact:** The reason affects which RPC is called and how the revocation is recorded (`revoked_reason` on unlock rows).

### 6. Handler File Location and Organization

**Issue:** The story suggests `src/lib/domains/billing/bridge-handlers.ts` as a "likely touchpoint" but doesn't mandate it.

**Unclear:** Should handlers be:
- Defined as separate functions in `bridge-handlers.ts`?
- Kept inline in the endpoint dispatcher?
- Exported from an existing file (e.g., `unlocks.ts`)?

**Impact:** Code organization and import structure; current dispatcher is inline in the endpoint file.

### 7. Error Handling for Handler Failures

**Issue:** The bridge endpoint currently catches errors from `dispatchBridgeEvent` and returns 500, but the story doesn't specify:
- Should handler failures be retried by the endpoint?
- Should they be logged separately?
- Should the endpoint return success (200) even if handlers fail (fire-and-forget)?

**Impact:** Affects billing service retry semantics and error visibility.

### 8. Duplicate Unlimited Activation Detection

**Issue:** The story says "Duplicate unlimited activation (same period) does not re-emit". The `billing_activation` table has a UNIQUE constraint on `(account_id, kind, stripe_subscription_id, subscription_period_end)`, but:
- The bridge idempotency check uses `stripe_event_id` (deduplicates exact event deliveries)
- The activation marker uniqueness deduplicates by period (prevents re-emission for the same period via different Stripe events)

**Unclear:** What happens if:
- Activation event `evt_1` for period `P1` inserts `billing_activation` and emits change
- Activation event `evt_2` (duplicate delivery of `evt_1`) is received → bridge idempotency skips it ✓
- Different activation event `evt_3` for same period `P1` is received → `billing_activation` UNIQUE constraint fails

Should the handler:
- Catch UNIQUE constraint violations and skip re-emitting?
- Or treat a period-duplicate as an error?

**Impact:** The story says "Duplicate unlimited activation (same period) does not re-emit" but doesn't clarify whether duplicates come from the same event (bridge idempotency) or different events (activation marker uniqueness).

---

## Summary

S4-09 requires implementing three bridge event handlers that are currently only partially implemented (inline in the endpoint). The handlers must:

1. **Pack fulfillment:** Check if bonus songs exist; emit `songsUnlocked` only if non-empty
2. **Unlimited activation:** Insert `billing_activation` marker (idempotent); emit `unlimitedActivated`
3. **Revocation (3 types):** Call appropriate reversal RPC; emit `candidateAccessRevoked` only if songs were actually revoked

**Major blockers:**
- Bridge payload schema missing `stripe_subscription_id` and `subscription_period_end` for unlimited activation
- Unclear which RPC to call for each revocation event kind
- Pack reversal RPC requires original pack event ID not included in bridge payload
- Revocation "access removed" assessment logic undefined for different event types

**These issues should be clarified or resolved before or during implementation to ensure the handlers integrate correctly with the billing service's bridge calls and the app's control-plane changes.**
