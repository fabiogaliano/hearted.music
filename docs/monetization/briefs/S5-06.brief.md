```typescript
export function useOnboardingNavigation() {
	const navigate = useNavigate({ from: "/onboarding" });

	const goToStep = useCallback(
		async (
			step: OnboardingStep,
			options?: {
				phaseJobIds?: PhaseJobIds | null;
				syncStats?: { songs: number; playlists: number };
			},
		) => {
			try {
				await saveOnboardingStep({ data: { step } });
				navigate({
					search: (prev) => ({
						...prev,
						step,
					}),
					state: (prev) => ({
						...prev,
						...(options?.phaseJobIds !== undefined && {
							phaseJobIds: options.phaseJobIds,
						}),
						...(options?.syncStats !== undefined && {
							syncStats: options.syncStats,
						}),
					}),
				});
			} catch (error) {
				// ... error handling
			}
		},
		[navigate],
	);

	return { goToStep };
}
```

```typescript
export const getBillingState = createServerFn({ method: "GET" })
```

## Constraints

### Frozen from DECISIONS.md & TERMINOLOGY.md

1. **Offer ID constants** (TERMINOLOGY.md §2):
   - `song_pack_500` (500 songs + 25 instant unlocks, $5.99)
   - `unlimited_quarterly` ($14.99/quarter, hidden behind `QUARTERLY_PLAN_ENABLED`)
   - `unlimited_yearly` ($39.99/yr, includes priority queue)

2. **Checkout idempotency** (DECISIONS.md §2):
   - `checkoutAttemptId` is UUID, part of signed request body
   - Forwarded as Stripe `idempotency_key`
   - Reused across retries of same intent; new UUID if offer changes

3. **Post-checkout polling** (story scope):
   - Poll `getBillingState` after Stripe success redirect
   - Stop after ~30–60s
   - Stripe webhook retry (up to 3 days) handles late delivery
   - Fallback message: "Your purchase is being processed. Your songs to explore will appear shortly."

4. **User-facing terminology** (TERMINOLOGY.md §2):
   - Unlock (action verb)
   - Unlock more songs (CTA copy)
   - Instant Unlocks (pack bonus label)

5. **Queue bands** (DECISIONS.md §1):
   - Free, no balance → `low`
   - Any positive balance (non-unlimited) → `standard`
   - Quarterly unlimited active → `standard`
   - Yearly unlimited active → `priority`
   - `self_hosted` → `priority`

6. **Unlimited access detection** (DECISIONS.md §2):
   - `unlimitedAccess.kind === "subscription"` indicates active unlimited subscription
   - `past_due` subscription status does NOT grant unlimited access
   - `subscriptionStatus` is normalized from Stripe values

## Open Questions / Risks

### 1. Stripe Redirect URL (Critical)

**Issue:** Story says "Redirect to Stripe Checkout" and "On return: poll `getBillingState`" but does not specify:
- Where does Stripe success_url point? (e.g., `/onboarding?step=plan-selection`, `/onboarding?step=checkout-success`, or different step?)
- Does app receive `session_id` or other query param from Stripe?
- How does app distinguish "returned from successful checkout" from "user manually navigated back to plan-selection"?

**Impact:** Without knowing the redirect target, the post-checkout flow cannot be implemented. The component cannot detect it should enter polling mode.

**Options to resolve:**
- Redirect to same step (`plan-selection`) with a query param flag like `?checkoutSuccess=true`
- Redirect to a new intermediate step like `checkout-success` that handles polling
- Redirect directly to `ready` and handle polling server-side before redirect
- Store `checkoutAttemptId` in browser session/localStorage for post-redirect recovery

### 2. Checkout State Recovery After Redirect

**Issue:** After Stripe redirect, how does the app remember:
- Which offer was being checked out (pack vs quarterly vs yearly)?
- The `checkoutAttemptId` for retry idempotency?

**Current state:** `PlanSelectionStep` generates a new UUID per button click but does not store it.

**Impact:** If component is unmounted/remounted after redirect, the offer and attempt ID are lost. Polling cannot know what billing state change to look for.

**Options:**
- Store in `localStorage` (not ideal for SSR)
- Store in TanStack Router state (passed through navigation)
- Use URL query param (e.g., `?checkoutAttemptId=<uuid>&offer=song_pack_500`)
- Retrieve from browser history / session
- Accept that polling looks for ANY billing state change (less robust but simpler)

### 3. Polling Timeout Behavior

**Issue:** Story says "stop after ~30–60s" and "fallback message shown" but does not specify:
- Should timeout navigate to `ready` anyway?
- Should it show fallback message and wait for user to manually advance?
- Should it retry or give up on polling?

**Impact:** UX and error recovery strategy undefined.

**Likely resolution:** Timeout → show fallback message + navigate to ready auto (billing webhook can still be processed post-onboarding).

### 4. Polling Interval & Backoff

**Issue:** Story does not specify:
- Initial poll interval (1s, 2s, 5s?)
- Exponential backoff or fixed interval?
- Max request count?

**Impact:** Affects server load and user-perceived latency.

**Current pattern:** `SyncingStep` uses fixed 1s interval. Likely similar approach for checkout polling.

### 5. Which Billing State Change to Detect?

**Issue:** For pack, polling should detect `creditBalance > 0`. For unlimited, detect `unlimitedAccess.kind === "subscription"`. But:
- What if webhook delivers both (e.g., plan upgrade + new balance)?
- What if webhook delivers partial state (subscription created but credit reversal not yet processed)?
- Should polling wait for a specific _combination_ of state changes?

**Impact:** May cause premature navigation or delayed timeout.

**Likely resolution:** Offer-specific detection:
- Pack: `creditBalance > previousBalance`
- Unlimited: `unlimitedAccess.kind === "subscription" && subscriptionStatus === "active"`

### 6. Baseline Billing State for Comparison

**Issue:** To detect "billing state updated," polling must compare against a baseline. But:
- What is the baseline? (State before checkout, or state at redirect time?)
- How is baseline captured and passed through the Stripe redirect?

**Impact:** Polling logic cannot determine if state "changed."

**Options:**
- Fetch baseline before calling `createCheckoutSession`, store in state
- Assume baseline is "no balance, no subscription" (reasonable for onboarding users)
- Just wait for any non-zero balance or subscription (simpler, less robust)

### 7. Error Handling During Polling

**Issue:** What if `getBillingState` throws or returns error during polling?
- Retry? Give up? Ignore?
- How many consecutive errors before fallback?

**Impact:** Robustness during service degradation.

### 8. Multiple Checkout Attempts (Retry)

**Issue:** Story says "`checkoutAttemptId` reused on retry" but:
- How does user trigger a retry? (Retry button on fallback message?)
- Does re-clicking the pack/unlimited button reset polling, or is it already in progress?

**Impact:** Idempotency and UX for failed checkouts.

### 9. Free Path Sync Stats

**Issue:** `ReadyStep` expects `syncStats: { songs: number; playlists: number }` (from context or props). Free path calls `goToStep("ready")` without passing syncStats.

**Current code (line 61):**
```typescript
await goToStep("ready");
```

**Problem:** ReadyStep will receive undefined syncStats. Must be:
```typescript
await goToStep("ready", { syncStats: { songs: 0, playlists: 0 } });
```

Or ReadyStep must handle the absence gracefully.

### 10. Post-Checkout Navigation State

**Issue:** After polling confirms purchase, navigation to `ready` must pass the same `syncStats` context as other flows. But:
- What are valid `syncStats` for pack/unlimited users? (May not have completed syncing.)
- Should `syncStats` reflect only free allocation, or something else?

**Impact:** ReadyStep copy and stats display may be misleading post-checkout.

### 11. Component Lifecycle & Cancellation

**Issue:** If user navigates away during polling (e.g., manual back button), polling loop must be cancelled to avoid state updates after unmount.

**Pattern:** SyncingStep uses `let isCancelled = false` in useEffect cleanup.

**Requirement:** Post-checkout polling must follow same pattern.

### 12. Keyboard Shortcut During Polling

**Issue:** If polling is in progress and user presses Enter (from ReadyStep shortcut), what happens?

**Impact:** May cause double-navigation or conflicts.

**Likely resolution:** Polling component disables shortcuts until complete.

### 13. Stripe Session ID Handling

**Issue:** Stripe checkout session may include a `session_id` in the success redirect URL. Should app:
- Capture it for logging/debugging?
- Store it for later retrieval by billing service?
- Ignore it?

**Impact:** Audit trail and error recovery.

### 14. Success Redirect URL Configuration in Billing Service

**Issue:** Story says "Redirect to Stripe" but does NOT specify whether `success_url` and `cancel_url` are:
- Hardcoded in billing service
- Configured via env var
- Passed in request from app

**Impact:** Implementation must assume one of these; coordination with S4-02 may be needed.

## Summary

S5-06 must implement the post-checkout polling flow that bridges the gap between Stripe payment and webhook fulfillment during onboarding plan selection. The key unknowns are:

1. **Where Stripe redirects** (URL target and query params)
2. **How to detect/recover the checkout intent** (offer + attemptId)
3. **Polling mechanics** (interval, timeout, state comparison)
4. **Navigation behavior** (auto-advance vs wait for user)
5. **Error recovery** (retries, fallback messaging)

All can be resolved by implementation agent using patterns established in SyncingStep and standard checkout UX conventions. The infrastructure (createCheckoutSession, getBillingState, polling library patterns) is in place; the main work is composing these into a coherent onboarding flow.
