# Implementation Brief: S3-10 Match/Suggestion Loaders — Entitlement Filtering

## Goal

Add entitlement filtering to `getSongMatches`, `getSongSuggestions`, and `getMatchingSession` in `matching.functions.ts` so revoked/locked songs are excluded immediately at read time.

## Changes Required

### 1. `src/lib/server/matching.functions.ts`

**`getMatchingSession`** — add entitlement filtering to undecided count:

Replace the sequential `getUndecidedSongs` call with a parallel fetch that also calls `select_entitled_data_enriched_liked_song_ids`, then filter before counting:

```typescript
export const getMatchingSession = createServerFn({ method: "GET" })
  .middleware([authMiddleware])
  .handler(async ({ context }): Promise<MatchingSessionResult | null> => {
    const { session } = context;
    const supabase = createAdminSupabaseClient();

    const snapshotResult = await getLatestMatchSnapshot(session.accountId);
    if (Result.isError(snapshotResult) || !snapshotResult.value) return null;

    const matchSnapshot = snapshotResult.value;

    const [undecided, entitledResult] = await Promise.all([
      getUndecidedSongs(matchSnapshot.id, session.accountId),
      supabase.rpc("select_entitled_data_enriched_liked_song_ids", {
        p_account_id: session.accountId,
      }),
    ]);

    const entitledSet = new Set(
      (!entitledResult.error && entitledResult.data
        ? entitledResult.data
        : []
      ).map((r: { song_id: string }) => r.song_id),
    );

    const entitledUndecided = undecided.filter((s) => entitledSet.has(s.songId));

    return { snapshotId: matchSnapshot.id, totalSongs: entitledUndecided.length };
  });
```

## Constraints

From **DECISIONS.md §4 (Read-Model Invariants)**:
> `match_result` is account-scoped via `match_snapshot.account_id` but still requires entitlement filtering (stale results can reference revoked songs)

> Match result and suggestion loaders filter by current entitlement at read time, not only by snapshot contents

> Revoked songs disappear immediately from matching UI without waiting for the next snapshot refresh

> Match snapshots are append-only; revocations do not delete old snapshots — latest supersedes, read-time filtering handles the gap

From **DECISIONS.md §2 (Architecture Invariants)**:
> The canonical entitlement predicate must be used in: … match/session read models

> Locked songs must not expose analysis text or match output

From **DECISIONS.md §4 (Shared artifacts do not imply entitlement)**:
> `song_analysis` rows are global (not account-scoped); existence does not mean the account may see that analysis

**Self-hosted accounts:** `select_entitled_data_enriched_liked_song_ids` uses the same `is_account_song_entitled` logic internally, so `self_hosted` accounts have all songs in the entitled set — no special casing needed.

**Return shape invariant:** Locked songs are excluded (filtered out), not transformed. The `SongMatchesResult`, `SongSuggestionsResult`, and `MatchingSessionResult` shapes are unchanged.

## Risks

1. **`fetchMatchPreviews` scope confusion**: The story lists `fetchMatchPreviews` as a target but it lives in `dashboard.functions.ts` (not `matching.functions.ts`) and is **already implemented** with entitlement filtering (confirmed by existing tests in `dashboard.functions.billing.test.ts`). No changes needed there — skip it.

2. **`select_entitled_data_enriched_liked_song_ids` semantics**: This RPC returns songs that are both entitled AND data-enriched (have `song_analysis`). For match context this is correct — match results only exist for analyzed songs. However, if the RPC is unavailable (not yet migrated in test DB), `supabase.rpc(...)` returns `{ data: null, error: ... }`. The error handling must default to empty entitled set (same as `fetchMatchPreviews` does), which safely excludes all songs rather than bypassing the gate.

3. **`createAdminSupabaseClient` call in `getMatchingSession`**: `getMatchingSession` currently does NOT call `createAdminSupabaseClient()`. Adding the entitlement RPC call requires adding `const supabase = createAdminSupabaseClient();` at the start of the handler. This is a minimal addition consistent with other handlers in the same file.

4. **No existing tests for `matching.functions.ts`**: The test file must be created from scratch. Follow the mock pattern from `dashboard.functions.billing.test.ts` exactly (especially the `createServerFn` mock that directly calls the handler function).
