## Changes Required

Create a new migration file: `supabase/migrations/{timestamp}_entitlement_predicate.sql` containing:

### 1. Main entitlement predicate function

```sql
CREATE OR REPLACE FUNCTION is_account_song_entitled(
  p_account_id UUID,
  p_song_id UUID
) RETURNS BOOLEAN
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT COALESCE(
    -- Case 1: Active unlock row exists
    EXISTS(
      SELECT 1 FROM account_song_unlock
      WHERE account_id = p_account_id
        AND song_id = p_song_id
        AND revoked_at IS NULL
    ),
    -- Case 2: Account has unlimited access
    EXISTS(
      SELECT 1 FROM account_billing
      WHERE account_id = p_account_id
        AND unlimited_access_source IS NOT NULL
        AND (
          unlimited_access_source = 'self_hosted'
          OR (unlimited_access_source = 'subscription' AND subscription_status = 'active')
        )
    ),
    -- Default: false
    false
  );
$$;
```

## Constraints from DECISIONS.md and TERMINOLOGY.md

**Frozen architecture invariants:**
- **Effective entitlement predicate** (DECISIONS.md §2): A song is entitled when **either** unlock row exists with `revoked_at IS NULL` OR account has active unlimited access
- **`past_due` and `unpaid` statuses do NOT grant unlimited access** for `subscription` source (DECISIONS.md §2)
- **`self_hosted` grants access without Stripe state** (DECISIONS.md §2, §3)
- **`unlimited_access_source IS NULL` → no unlimited access** (DECISIONS.md §2)
- **Single canonical predicate** — no per-consumer variants (DECISIONS.md §2)

**Security constraints:**
- `SECURITY DEFINER` with `SET search_path = public` (MONETIZATION_V2_PLAN.md § Security for new billing RPCs)
- Function must be accessible by service_role only via RLS bypass

**Frozen terminology:**
- Function name: `is_account_song_entitled` (frozen in TERMINOLOGY.md §5)
- Subscription status values: exact values from CHECK constraint (frozen)
- Unlimited access source values: `'subscription' | 'self_hosted'` (frozen)

## Acceptance Criteria (from S1-04 story)

- [ ] Returns `true` for a song with an active (non-revoked) unlock row
- [ ] Returns `false` for a song with a revoked unlock row and no unlimited access
- [ ] Returns `true` for any song when `unlimited_access_source = 'self_hosted'`
- [ ] Returns `true` for any song when `unlimited_access_source = 'subscription'` AND `subscription_status = 'active'`
- [ ] Returns `false` when `unlimited_access_source = 'subscription'` AND `subscription_status IN ('past_due', 'unpaid', 'canceled', 'none')`
- [ ] Function is `SECURITY DEFINER` with `search_path = public`

## Risks

**Low risk:** The function logic is straightforward boolean evaluation with clear acceptance criteria. The schema dependencies are satisfied. The function signature and security settings follow established repo patterns.

**Potential edge case:** The function will return `false` for accounts that don't have `account_billing` rows, but according to DECISIONS.md §3, every account gets an `account_billing` row in every deployment, so missing rows are invariant violations, not valid states.
