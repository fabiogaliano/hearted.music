### Bun.serve() Patterns in v1_hearted
**File:** `src/worker/health.ts`
```ts
export function startHealthServer() {
	return Bun.serve({
		port: workerConfig.healthPort,
		fetch(req) {
			const url = new URL(req.url);
			if (url.pathname === "/health") {
				if (!isHealthy || isShuttingDown) {
					return new Response(
						JSON.stringify({
							status: "unhealthy",
							shuttingDown: isShuttingDown,
						}),
						{ status: 503, headers: { "Content-Type": "application/json" } },
					);
				}
				return new Response(JSON.stringify({ status: "ok" }), {
					status: 200,
					headers: { "Content-Type": "application/json" },
				});
			}
			return new Response("Not Found", { status: 404 });
		},
	});
}
```
- Shows basic Bun.serve() pattern with routing on `url.pathname`
- Returns JSON responses with appropriate status codes
- **Note:** v1_hearted does not yet use Hono; this is raw Bun.serve(). S4-01 introduces Hono as a new routing dependency.

**File:** `scripts/dev-embedding-sidecar.ts`
- Another example of Bun.serve() with multiple endpoints and JSON handling
- Shows error handling patterns with `better-result` library for `Result<T, E>` types
- Uses `Response.json()` for JSON responses

### Frozen Constraints from DECISIONS.md

#### Architecture Boundary (Section 2)
- **Repo isolation:** `v1_hearted_brand/` owns Stripe SDK, HMAC auth, anti-abuse, webhook handling, offer catalog
- **v1_hearted/ never imports Stripe or billing service code** — only calls billing service via HTTP bridge with HMAC auth
- **Stripe price IDs stay opaque in billing service** — v1_hearted only knows stable offer IDs (`song_pack_500`, `unlimited_quarterly`, `unlimited_yearly`)

#### HMAC Auth Invariant (Section 2)
- **Bridge pattern:** All billing-service → app calls use HMAC + timestamp + body-hash + replay-window
- **App → billing calls:** Will use same auth (implemented by app in S4-08)
- **Shared secret:** Must match `BILLING_SHARED_SECRET` in both v1_hearted/ and v1_hearted_brand/
- **Signature components:**
  - Timestamp header
  - Raw request body hash (typically SHA256)
  - Shared secret
  - Clock-skew window (e.g., 5 minutes) for replay protection

#### Control Plane Boundary (Section 2)
- **No business logic leakage:** Billing service never imports or runs `applyLibraryProcessingChange()`
- **Control-plane reactions happen exclusively in v1_hearted/**
- **Bridge ingress:** App receives billing events (pack fulfilled, unlimited activated, revocation) and injects them into control plane as `LibraryProcessingChange` variants

#### Data Model Invariant (Section 3)
- **Service-role Supabase client only:** All database operations use service-role key (not anon)
- **No public RLS bypass:** Billing service never exports public endpoints that bypass RLS

#### Env Flags (Section 6)
Frozen variable names:
- `BILLING_ENABLED` (master toggle in app, not directly used by service)
- `BILLING_SERVICE_URL` (in v1_hearted, points to `billing.hearted.music`)
- `BILLING_SHARED_SECRET` (shared across both repos)
- `QUARTERLY_PLAN_ENABLED` (in v1_hearted, not billing-service specific)

#### Offer IDs (Frozen in TERMINOLOGY.md §2)
Canonical internal offer IDs (billing service resolves to Stripe price IDs):
- `song_pack_500` — song pack offer
- `unlimited_quarterly` — quarterly subscription offer
- `unlimited_yearly` — yearly subscription offer (branded as "Backstage Pass" in UI)

#### Bridge Event Kinds (Frozen in TERMINOLOGY.md §6)
Event types that billing service will emit to app bridge (defined in S2-05, consumed here):
- `pack_fulfilled` — pack purchase completed
- `unlimited_activated` — subscription started
- `pack_reversed` — pack purchase refunded
- `unlimited_period_reversed` — subscription period refunded
- `subscription_deactivated` — subscription ended/canceled

## Open Questions / Risks

### 1. Hono Framework Integration with Bun
**Issue:** v1_hearted uses raw Bun.serve() pattern. S4-01 introduces Hono as a new routing dependency. Hono is designed for edge platforms (Cloudflare Workers, Bun, Deno) but no existing proof of pattern in this codebase.

**Risk:** Hono configuration for Bun.serve(), middleware ordering, and error handling patterns need validation. No existing `app` initialization code to reference.

**Unclear:** Exact Hono version and configuration. Story doesn't specify if Hono v3 or v4, or which Bun adapter to use.

### 2. HMAC Implementation Details
**Issue:** Story specifies "timestamp header + raw body hash + shared secret" and "clock-skew window (e.g., 5 minutes)" but does not specify:
- Exact header name for timestamp (`X-Timestamp`, `X-Request-Time`, etc.)
- Exact header name for signature (`X-Signature`, `X-HMAC-SHA256`, etc.)
- Hash algorithm (SHA256 assumed, not stated)
- Timestamp format (Unix epoch seconds? ISO 8601?)
- Signature format (hex, base64?)

**Risk:** App-side signing (S4-08) must match exactly. Any mismatch breaks all requests. Should align with app before implementation.

### 3. No Existing Middleware Pattern
**Issue:** v1_hearted uses TanStack Start (Cloudflare Workers); v1_hearted_brand/ billing service will use Bun HTTP directly. HMAC middleware implementation is novel for this codebase.

**Unclear:** Should middleware throw an error, return 401, or both? How are errors logged and monitored?

### 4. Environment Validation Pattern
**Issue:** v1_hearted uses `@t3-oss/env-core` for server-side env validation. S4-01 could:
- Reuse the same pattern
- Use Zod directly
- Use Hono's built-in env helpers

**Unclear:** Which pattern aligns with team conventions for a separate service?

### 5. Supabase Client Initialization Scope
**Issue:** S4-01 mentions "Supabase JS client initialization (service role)" but does not clarify:
- Global client instance or per-request initialization?
- Error handling for connection failures?
- Connection pooling or sidecar database proxy?

**Unclear:** Whether the service connects directly to Supabase Postgres or through a pooler (PgBouncer, etc.).

### 6. Stripe SDK Scope
**Issue:** Story says "Stripe SDK initialization" as part of the scaffold. S4-02 (checkout sessions) and S4-03 (portal sessions) are the first stories that use Stripe. S4-01 is just initializing the client.
